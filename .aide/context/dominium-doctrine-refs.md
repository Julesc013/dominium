# Dominium Doctrine Refs For AIDE Lite

This file is a compact reference index for Q23 task packets. It is not a
canonical doctrine file and does not restate full doctrine.

## Authority Floor

- `docs/canon/constitution_v1.md`: constitutional execution contract,
  determinism, Process-only mutation, authority, profiles, pack integration,
  compatibility, and validation obligations.
- `docs/canon/glossary_v1.md`: canonical vocabulary for Truth, Perceived,
  Rendered, Process, Law, AuthorityContext, Pack, Domain, Field, Profile,
  Refusal, Replay, and XStack terms.
- `AGENTS.md`: canonical agent governance, authority precedence, current
  program state, work classes, validation/reporting expectations, protected
  areas, ownership cautions, and token-discipline pointer.

## Planning And Ownership

- `docs/planning/AUTHORITY_ORDER.md`: conflict resolution across semantic,
  structural, compatibility, release, and planning authority.
- `docs/planning/SNAPSHOT_INTAKE_PROTOCOL.md`: deterministic intake,
  artifact-family classification, quarantine rules, and silent-inference
  prohibitions.
- `docs/planning/MERGED_PROGRAM_STATE.md`: merged program position, doctrine
  locks, extension zones, unresolved quarantine items, and invalidated
  assumptions.
- `docs/planning/EXTEND_NOT_REPLACE_LEDGER.md`: live roots to preserve, extend,
  consolidate, or treat with review caution.
- `docs/planning/GATES_AND_PROOFS.md`: entry gates, exit proofs, and human
  review checkpoints.
- `docs/planning/POST_PI_EXECUTION_PLAN.md`: fixed macro execution order and
  prompt sequencing constraints.
- `docs/planning/SEMANTIC_OWNERSHIP_REVIEW.md`: root ownership distinctions for
  `fields/` vs `field/`, `schema/` vs `schemas/`, `packs/` vs `data/packs/`,
  `specs/reality/` vs `data/reality/`, and generated evidence.
- `docs/planning/PLAYER_DESIRE_ACCEPTANCE_MAP.md`: desire-family acceptance
  constraints and ordinary-life, engineering, civilization, AI, and future
  interface doctrine.

## Reality Doctrine

- `specs/reality/SPEC_DOMINIUM_UNIVERSAL_REALITY_FRAMEWORK.md`: truth totality,
  sparse materialization, causal closure, representation, substitution,
  continuity, macro/micro consistency, and interface-agnostic reality.
- `specs/reality/SPEC_DOMAIN_CONTRACT_TEMPLATE.md`: required domain-contract
  shape, including primitives, Processes, recognitions, affordances,
  representation, failure, verification, observation, interfaces, scaling, and
  extension paths.
- `specs/reality/SPEC_CAPABILITY_SURFACES.md`: lawful capability surfaces and
  actor classes without UI-only powers or direct truth mutation.
- `specs/reality/SPEC_REPRESENTATION_LADDERS.md`: primitive, recognized,
  formalized, substituted/capsuled, macro, and micro representation rules.
- `specs/reality/SPEC_SEMANTIC_ASCENT_DESCENT.md`: lawful ascent/descent,
  ambiguity, failure, lineage, and anti-invention rules.
- `specs/reality/SPEC_FORMALIZATION_CHAIN.md`: discovery through retirement
  chain, validation, standards, institutional adoption, diffusion, maintenance,
  and provenance.
- `specs/reality/SPEC_CROSS_DOMAIN_BRIDGES.md`: bridge ownership, invariants,
  directionality, capability, representation, formalization, failure, and
  verification rules.
- `data/reality/*.json`: operational registry projections for the above specs;
  use as projection evidence, not as a replacement for `specs/reality/**`.

## Q23 Packet Rule

Task packets should cite these paths and use `.aide/context/repo-map.*` for
metadata. They should not inline whole doctrine files.
