# Dominium Runtime Kernel Model

## 1. Purpose and Scope

This document defines the constitutional runtime kernel model for Dominium.
It exists so early and later Φ work can extract runtime structure from the live repository without letting runtime convenience, product shells, or thin directory naming silently redefine semantic law.

The runtime kernel model answers a narrower question than the completed Λ block.
Λ defines what reality means, how domains lawfully interact, how representations preserve meaning, how formalization propagates designs and institutions, and what actor desires the project must support.
Σ-0 defines who may operate in the repository and what authority rules govern work.
This document defines the lawful runtime host layer that executes those meanings without owning them.

This prompt does not yet define the full component model, service catalog, lifecycle machinery, snapshot implementation, or module-loader implementation.
Those remain later Φ work.

## 2. Core Definition

In Dominium, the runtime kernel is the constitutional host for lawful execution.
It is the runtime-side substrate that schedules, coordinates, and verifies deterministic Process execution over authoritative truth while preserving the boundaries between semantic law, runtime orchestration, products, presentation, and control-plane concerns.

The runtime kernel is not simply "the engine" in the sloppy sense.
It is not a synonym for one directory, one binary, or one implementation language.
It is also not identical to component instances, services, modules, apps, products, renderers, or release systems.

The kernel exists to host and constrain execution.
It does not define semantic ontology, domain ownership, player meaning, governance authority, or publication policy.

## 3. Kernel Responsibilities

At the constitutional level, the runtime kernel is responsible for:

- hosting lawful execution of truth-affecting Processes under deterministic ordering and authority constraints
- enforcing runtime-side discipline for determinism, refusal visibility, provenance, replay integrity, and process-only mutation
- coordinating lawful runtime boundaries between domains without collapsing them into a hidden super-domain
- providing the execution substrate on which later components and services may be layered
- preserving truth, perceived, and render separation at runtime so perception and presentation remain projections rather than mutation authorities
- supporting lawful observation, audit, persistence, replay, and verification hooks as first-class runtime concerns
- providing runtime-side enforcement points for explicit degradation, refusal, and activation policy without inventing new semantic law
- preserving pack-driven integration boundaries so optional content and capability surfaces remain registry- and manifest-governed rather than hardwired into kernel law
- preserving compatibility, identity, and ownership boundaries while consuming canonical contracts and registries

The kernel therefore hosts semantic law in execution form.
It does not become the source of semantic law.

## 4. Kernel Non-Responsibilities

The runtime kernel must not own or silently redefine:

- semantic ontology, including Assemblies, Fields, Processes, Law, or the truth and perceived distinction
- domain ownership truth, domain contracts, or cross-domain bridge law
- canonical governance law, agent authority policy, or review-gated repository behavior
- release, publication, versioning, trust-root, or control-plane doctrine
- product shell semantics for client, server, launcher, setup, tools, or other application surfaces
- UI or UX meaning, including what a capability surface means to a player or operator
- institutional, legal, or social semantics beyond lawful execution hosting
- ownership classifications established by semantic ownership review
- convenience-based rebinding from canonical roots to projected, mirrored, generated, or quarantined roots

If runtime structure requires new semantics, the semantics must be defined in doctrine first and then hosted by runtime.
The kernel may not invent them by adjacency or expedience.

## 5. Kernel Boundary Layers

The runtime kernel sits within a layered constitutional boundary model:

| Layer | Constitutional Role | What It Must Not Do |
| --- | --- | --- |
| Semantic and domain law | Defines reality meaning, domain contracts, bridges, formalization, representations, and lawful action surfaces. | Must not be reduced to runtime convenience. |
| Runtime kernel | Hosts deterministic execution, orchestration boundaries, audit hooks, and runtime integrity. | Must not redefine semantics, ownership, or governance. |
| Components and services | Layer structured runtime capabilities on the kernel. | Must not bypass kernel law or silently create new ontology. |
| Module implementations | Provide concrete code behind component and service abstractions. | Must not become accidental constitutional authority. |
| Products and shells | Client, server, setup, launcher, tools, appshell, and other consumption surfaces. | Must not redefine truth, law, or kernel authority. |
| Presentation and interface adapters | Render, UI, operator views, and interface-specific access paths. | Must not mutate truth outside lawful Process paths. |
| Control-plane and release-plane | Build, packaging, publication, trust, and release governance surfaces. | Must not become runtime truth authority. |

These layers may interact closely in code.
They remain constitutionally distinct even when they share libraries, binaries, or repository neighborhoods.

## 6. Kernel and Truth

The kernel hosts lawful truth execution.
It does not define truth.

The kernel must therefore preserve all of the following:

- total truth remains authoritative even when only a sparse subset is materialized at runtime
- unmaterialized truth is not equivalent to nonexistent truth
- authoritative mutation remains Process-only and law-gated
- perception remains a filtered projection of truth
- rendering remains presentation and must never become a truth authority

Any kernel design that lets runtime shortcuts, observer projections, or presentation loops mutate authoritative state outside lawful Process execution is constitutionally invalid.

## 7. Kernel and Domain Contracts

The kernel must consume domain contracts rather than replacing them.
Domain contracts define primitives, processes, recognitions, affordances, formalizations, substitutions, failures, verification, observation bindings, and scaling expectations.
The kernel provides lawful execution and orchestration substrate for those contracts.

This means:

- the kernel may coordinate multiple domains but may not erase distinct domain ownership
- the kernel may host bridge-aware execution but may not replace explicit cross-domain bridge law
- the kernel may enforce execution legality, ordering, and auditability, but it may not silently reinterpret domain meaning

Kernel-level convenience must never collapse multiple domains into a hidden runtime super-domain.

## 8. Kernel and Capability Surfaces

Capability surfaces remain semantic action surfaces defined over lawful substrate.
The kernel may host execution pathways that realize those surfaces, but it may not reinterpret them as mere UI widgets, button bindings, or hotkey routes.

The kernel must therefore preserve that:

- capability grants permission to attempt, not permission to bypass law
- capability surfaces remain attached to domain-grounded substrate and Process routes
- embodied, abstract, institutional, NPC, and AI access modes remain alternate lawful access paths over the same substrate

Kernel pathways that convert a capability surface into a hidden superpower or a presentation-only shortcut are forbidden.

## 9. Kernel and Representation, Substitution, and Scaling

The kernel must respect representation ladders, semantic ascent and descent, lawful substitution, and macro to micro continuity.
Runtime execution may later support materialization, collapse, expansion, and substitution machinery, but none of those operations are allowed to alter invariants silently.

At minimum, the kernel must preserve:

- declared identity and provenance invariants
- declared topology, connectivity, and conservation-relevant invariants
- continuity across macro and micro transitions
- lawful visibility of substitution, degradation, and refusal where doctrine requires it

Representation change is not permission for the kernel to invent or erase meaning.

## 10. Kernel and Components, Services, and Modules

Later Φ prompts will define component and service doctrine in detail.
This prompt fixes the boundary they must inherit.

Within that boundary:

- the kernel is the constitutional host and execution substrate
- components are structured runtime capability groupings layered within kernel law
- services are orchestrated runtime functions layered within kernel law
- modules are concrete implementation surfaces that may realize kernel, component, or service behavior without becoming constitutional authority by themselves

This prompt deliberately does not freeze the final component taxonomy or service catalog.
It only freezes that components and services must remain downstream of kernel law rather than upstream of it.

## 11. Kernel and Products

Products such as client, server, setup, launcher, tools, and appshell are not the kernel.
They are consuming surfaces, operator shells, or authoritative hosts that rely on kernel-governed execution boundaries.

Product convenience must not redefine:

- what truth exists
- what law permits
- which domains own which semantics
- which roots are canonical

UI- or product-led architecture is therefore forbidden as a source of runtime truth.

## 12. Kernel and Packs and Content

The kernel must remain compatible with pack-driven integration.
It may host activation, loading, refusal, degradation, and runtime compatibility checks later, but it must not turn pack content into hardwired semantic law or treat optional pack absence as permission for hidden fallback.

This means:

- content and capability expansion must remain declarative and registry-governed where doctrine requires it
- the kernel may host lawful pack activation pathways, but it may not treat pack data as executable semantic override
- missing or incompatible packs must produce explicit refusal or explicit degradation rather than silent substitution
- runtime convenience may not collapse `packs/` and `data/packs/` into one interchangeable authority surface

## 13. Kernel and Observability, Persistence, and Replay

Observability, persistence, replay, and verification are first-class kernel concerns.
They are not afterthought tooling.

The kernel must support lawful hooks and integration points for:

- replay equivalence and deterministic audit trails
- persistence and checkpoint semantics
- provenance-preserving externalization of execution evidence
- refusal, degradation, and activation-policy visibility
- verification of runtime legality against semantic and governance inputs

Later prompts may formalize exact structures, but they must inherit that these concerns belong near kernel law rather than as optional decorations.

## 14. Kernel and Anti-Reinvention Doctrine

The runtime kernel model is grounded in existing repository embodiment.
It is not a greenfield fantasy.

Early and later Φ work must therefore treat the following as evidence-rich extension surfaces:

- `engine`
- `game`
- `app`
- `compat`
- `control`
- `core`
- `net`
- `process`
- `server/runtime`
- `server/persistence`
- `libs`
- `client`
- `server`

They must also treat the following as adjacent but non-kernel constitutional surfaces:

- `release`
- `repo`
- `updates`
- `tools`
- `setup`
- `launcher`
- `appshell`

The thin `runtime/` root is not canonical merely because of its name.
It may later host important runtime code, but name prominence is not proof of semantic or architectural authority.

## 15. Runtime Caution Zones

The checkpoint review cautions remain in force for all early Φ work.

Early Φ work must remain explicitly cautious about:

- `field/` versus `fields/`, where ownership review already rejected silent equivalence
- `schema/` versus `schemas/`, where schema law remains canonical and projections remain non-authoritative
- `packs/` versus `data/packs/`, where packaged activation scope and authored pack declaration scope are not interchangeable
- generated, mirrored, transitional, and quarantined roots, which may not be normalized into canonical truth for convenience
- older planning artifacts that still contain numbering or order drift for Φ prompts, because current checkpoint law outranks stale sequencing text

Runtime convenience cannot bypass ownership review.
No early Φ artifact may silently promote projected or generated roots into canonical runtime truth.

## 16. Anti-Patterns and Forbidden Shapes

The following runtime shapes are constitutionally forbidden:

- a kernel treated as a semantic god-object that owns meaning for every domain
- services that invent domain law or bridge law rather than consuming it
- products or shells that redefine runtime truth or authority semantics
- convenience components that swallow ownership, compatibility, or provenance boundaries
- presentation-driven kernel design where UI, render, or operator tooling becomes truth authority
- code adjacency treated as proof of semantic unity
- automatic canonicalization of `runtime/` because the name sounds central
- kernel designs that assume unmaterialized truth does not exist
- runtime models that normalize ownership-sensitive roots for expedience

## 17. Stability and Evolution

This artifact is a provisional but canonical runtime doctrine.
It is stable enough to govern early Φ work, but it is expected to be refined by explicit downstream doctrine rather than by silent code drift.

The allowed immediate refinements are:

- `Φ-1 — COMPONENT_MODEL-0`
- `Φ-2 — RUNTIME_SERVICES-0`

Later prompts may extend this model for lifecycle, persistence, externalization, and operational policy, but they must not contradict these kernel boundaries.
Σ mirrors, task bridges, and later Υ and Ζ work may rely on this model as the runtime boundary floor after early Φ is complete.

## 18. Direct Constitutional Answers

1. The runtime kernel in Dominium is the constitutional host for lawful execution of authoritative Processes and runtime integrity.
2. It is responsible for deterministic execution discipline, execution boundary coordination, audit and replay hooks, and runtime preservation of truth/perceived/render separation.
3. It is not responsible for defining semantic ontology, governance law, product behavior, release doctrine, or UI meaning.
4. It relates to semantic doctrine by consuming and hosting it lawfully rather than redefining it.
5. It relates to services, components, modules, and products as their execution boundary floor, not as their synonym.
6. It must not redefine domain law or ownership because semantic meaning and canonical ownership are established elsewhere in explicit doctrine and review artifacts.
7. The checkpoint cautions that remain in force are ownership-sensitive root distinctions, explicit anti-reinvention bias, rejection of automatic `runtime/` canonicality, and the requirement that stale planning drift must not override the current checkpoint.
