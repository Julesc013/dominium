## 1. Executive Summary


This chat was about designing and planning a Dominium/Domino UI authoring ecosystem. The user’s immediate problem was that the current Windows launcher UI looks mangled and flickers. The larger objective is to build internal tooling that can visually design pixel-perfect, native-compatible UI definitions for setup, launcher, game, and other Dominium tools. The chat evolved from requirements discovery into a detailed phased implementation plan and a large set of copy-paste-ready Codex prompts.

The most important project context established in this chat is that Dominium already has an in-tree UI system called DUI, using a TLV schema/state model with at least three backends described by the user: win32 via Win32 common controls/comctl32, dgfx via Domino DGFX renderer and DSYS event pump, and null for headless operation. The user named existing launcher-related files including `launcher_ui_v1.tlv`, `gen_launcher_ui_schema_v1.py`, `dom_launcher_app.cpp`, `dom_launcher_actions.cpp`, `dom_launcher_catalog.cpp`, `dui_win32.c`, `dui_dgfx.c`, and `dui_null.c`. These were not inspected in this chat, so their contents remain unverified.

The main architectural decision was to build a Phase A Windows-only UI Editor first, then a Phase B full Tool Editor. The Phase A UI Editor is scoped to modern Windows NT / Windows 10+, system DPI only, 100% design scale, and native Win32 preview/runtime controls. Its purpose is to become useful quickly: author canonical UI TLV files, import existing tool UIs, expose a CLI for Codex automation, and generate/action-codegen UIs. The Phase B Tool Editor is the long-term self-hosting editor that should eventually run on all supported systems where compiled, including Windows NT, Linux, Win9x, Mac OS X, Android, classic Mac OS, and others. The Tool Editor should eventually edit itself and all Dominium tools.

The core data decisions are that TLV is the canonical source of truth, edited through a canonical in-memory IR/AST rather than byte-patching. A deterministic JSON mirror is generated by default if small, but it is not a hand-authored source. UI documents should be split from runtime mutable state (`ui_doc.tlv` and optional `ui_state.tlv`). Stable widget IDs are planned as monotonic `u32` values, names are human-readable and renamable, and event bindings use stable string action keys compiled to numeric IDs through a registry and generated code. The assistant recommended a unified C-compatible `domui_event` ABI, and the prompt series adopted it.

The chat produced two prompt sequences. First, an 11-prompt foundational sequence covering repo scan/scaffolding, UI IR, TLV I/O, capability system, layout engine, Splitter/Tabs/Scroll widgets, action codegen, UI Editor MVP, Tool Editor bootstrap, Win32 structural flicker fixes, and hardening. Second, after the user expanded requirements, a 6-prompt UU sequence covering UI discovery/import/export, headless CLI, ops.json scripted edits, Minecraft-style launcher redesign, Minecraft-style setup redesign, and CI/determinism hardening. The Minecraft-style launcher/setup requirement was clarified by the user to mean logical layout only, not graphical styling. The implementation must use native Win32 widgets only, with no owner-draw, custom fonts, custom colors, or skinning.

The user uploaded three files: `/mnt/data/source.zip`, `/mnt/data/SetupC.zip`, and `/mnt/data/LauncherC.zip`. These files were not inspected in the chat. A launcher/setup structural description was generated and accepted as useful by the user, but it should be treated as a conceptual planning artifact rather than verified screenshot analysis until the bundles are inspected.

The most recent future-work thread was IDE-native live editing. The user requested Visual Studio 2026 for Windows tool UI live editing, Xcode for macOS tool UIs, and Linux development tools for Linux UIs. The assistant generated a prompt proposing platform-native preview hosts and IDE build/run integrations while preserving canonical DUI TLV as the source of truth. This preview-host interpretation is an assistant proposal and should be confirmed before implementation if the user expects direct use of Visual Studio/Xcode GUI designers.

A future assistant must not lose the distinction between decisions, assistant recommendations, and unverified assumptions. In particular, do not assume any generated Codex prompt has been executed unless the user says so or repository files confirm it. Do not treat screenshot-derived descriptions as verified until the zip bundles are inspected. Do not convert the Minecraft layout test into a graphical reskin. ## 13. Rationale and Assumptions


The visible rationale for building the UI Editor before the Tool Editor is speed and bootstrapping: a minimal Windows-native authoring tool can repair and generate real DUI documents sooner, then be used to create the more complete Tool Editor. The visible rationale for keeping DUI is that the repo already has a TLV-based UI runtime with multiple backends; replacing it would create a parallel UI system and undermine the stated universal UI definition goal.

The data architecture rationale is safety and determinism. A canonical IR avoids brittle binary patching; TLV remains runtime canonical; JSON mirrors support review without becoming source. Stable IDs, sorted ordering, atomic saves, backups, and deterministic reports support git review, CI, and future Codex automation.

The capability-system rationale is portability. The user wants many platforms and native controls; backend/tier validation is the mechanism that prevents Windows-specific assumptions from becoming permanent schema requirements.

The launcher/setup capability tests are intentionally structural, not aesthetic. The user explicitly clarified that Minecraft-style refers to logical layout. Therefore the accepted implementation target is panel arrangement, navigation flow, step wizard structure, and native widget hierarchy.

Main assumptions needing verification are: actual repo files match user-provided names; existing TLV can be imported; setup tool paths exist as assumed; generated prompts have not necessarily been executed; screenshot bundles may or may not match the assistant’s structural description; preview-host IDE integration may or may not satisfy the user’s intent for live editing.
