## 2. Ultra-Condensed Carry-Forward Capsule


This chat contributes the main UI Editor / Tool Editor architecture and prompt plan for the Dominium/Domino project. The user’s immediate pain point is the existing Windows launcher UI looking mangled and flickering. The project already has a Dominium UI (DUI) TLV schema/state system with win32, dgfx, and null backends. The user provided names of existing launcher and DUI files but the files were not inspected here.

The key strategic decision is phased bootstrapping: build a Windows-only Phase A UI Editor first, then use it to build the fuller Phase B Tool Editor. Phase A is modern Windows NT/Windows 10+, system DPI only, 100% design scale, native Win32 preview, and pragmatic enough to ship quickly. Phase B is the eventual self-hosting, cross-platform Tool Editor that can edit itself and all Dominium tools. The Tool Editor should eventually run wherever compiled across broad targets including Windows NT, Linux, Win9x, Mac OS X, Android, classic Mac OS, and others.

Core data decisions: DUI remains the foundation; TLV is canonical; a canonical IR/AST edits and writes TLV safely; JSON mirror is generated by default if small and is not hand-edited; `ui_doc.tlv` and optional `ui_state.tlv` separation is the preferred model. Stable widget IDs are planned as monotonic `u32`; action bindings are planned as stable string keys compiled to numeric IDs via a registry; generated code is split into generated and user-owned files. A generic backend capability system is required across all backends.

The chat produced two major prompt sets: an 11-prompt foundational implementation sequence and a 6-prompt UU extension sequence. The foundational prompts cover repo mapping, UI IR, TLV I/O, backend capabilities, layout engine, Splitter/Tabs/Scroll widgets, action codegen, UI Editor MVP, Tool Editor bootstrap, Win32 structural batching/flicker fixes, and hardening. UU1–UU6 add UI discovery/import/export, headless CLI, ops.json edit scripts, launcher canonical UI generation, setup canonical UI generation, and CI/determinism hardening.

The user explicitly clarified that Minecraft-style launcher/setup means logical layout only, not graphical styling. The capability tests must use native Win32 widgets only: no owner-draw, no custom fonts, no custom colors, no visual skinning. The launcher structure is header, sidebar navigation, tabbed primary panel with news/instances/settings, and footer. The setup structure is a wizard with header, body pages, and footer navigation. Uploaded screenshot bundles exist but were not inspected in this chat.

The last new workstream is IDE live editing. The user requested Visual Studio 2026, Xcode, and Linux development tools. The assistant generated a prompt proposing platform preview hosts and IDE build/run/watch integration while retaining canonical TLV. That approach is an assistant proposal and needs confirmation before becoming a hard requirement.
