## 2. The Story of the Conversation

* build metadata embedded into binaries;
* changelog automation;
* reproducible packaging;

That was the initial design phase. At that point, the build-number idea was still broad and aggressive: a global build number incrementing frequently.

### The discussion moved from “is this valid?” to “how do we make it modular?”

The user then asked whether the implementation was industry-accepted and what could be improved. The response framed the approach as closer to game engines, operating systems, and long-lived infrastructure than typical game development. The important conclusion was that the design was not exotic, but it was unusually rigorous for games.

Then the user asked what was still missing for modularity and how to support distributions and gameplay without assets. This led to the **content/packs** discussion. The project should support:

* engine/game binaries that boot with zero assets;
* no mandatory textures, audio, fonts, scenarios, or worlds;
* procedural/symbolic/ASCII fallbacks;
* all content via open-format packs;
* packs as capability providers rather than mandatory asset bundles;
* saves and replays that reference pack IDs and schema versions rather than raw file paths.

This topic mattered because the project was trying to avoid the common game failure mode where binaries, content, saves, mods, and platform-specific assets become tangled.

### The user then used this chat to coordinate parallel chats

The user had another chat working on content and systems, and asked for a prompt to inform that chat of everything decided so far. A similar prompt was generated for the applications/platforms/renderers chat. These prompts established authoritative boundaries:

At this stage, the project was not just being designed; it was being coordinated across multiple assistants.

Another important tension was “no silent fallback” versus renderer fallback. The application/platform response resolved this by saying explicit renderer selection must fail if unavailable, while `auto` mode can fallback only with explicit logging. This distinction became important for testability and user transparency.

The user then brought in a scratchpad prompt about DRM, anti-cheat, copy protection, and adjacent control systems. The key claim was that these are not engine features. They are distributed control layers. The prompt placed DRM mainly in launcher/platform/cloud, anti-cheat mainly in server authority plus optional client instrumentation, copy protection in installer/launcher/platform, and explicitly excluded enforcement from engine/game logic.

This mattered because the user wanted future support for such systems without enabling them by default or compromising core principles.

### Demo, tourist mode, services, and piracy became TESTX3

The user then explored the idea of a Steam demo or free download. The user did not want a separate demo product or separate code, but wanted demo restrictions to be hard to bypass. The response reframed the problem: the project is not selling binaries; it is selling authority. The user liked the idea that free/demo players could join servers as **tourists**. This became a major concept: tourists can join real servers, see real players, explore real worlds, and experience the scale of the game, but cannot mutate authoritative state. Server authority enforces the restrictions, making client patching largely irrelevant for durable value.

This led to **TESTX3**, which formalized:

* authority and entitlements;
* base/free, tourist, full-player, service-scoped, and admin authority profiles;
* one distribution for demo/free/paid/service;
* storefront neutrality across Steam, Epic, and website;
* services inside products or products inside services;
* piracy containment by denying durable value extraction, not by preventing copying;
* save/replay/progression rules around authority scopes.

This was an important philosophical and technical decision. It allowed the project to support free downloads, demos, paid upgrades, subscriptions, services, and anti-piracy containment without creating code forks or corrupting the deterministic core.

### Architecture is closed

**FACT:** The latest upstream prompt explicitly says architecture is closed and canon is locked. This decision was not merely suggested by the assistant; it came from the user-pasted upstream canon. It means future chats should not redesign systems, add new primitives, or reinterpret constraints.

The alternative was to keep iterating on architecture. That is now rejected unless the user explicitly reopens design. This affects every future assistant: they should audit, implement, optimize, or maintain, not re-litigate core structure.

### Core ontology is fixed

**FACT:** All systems must reduce to Assemblies, Fields, Processes, Agents, and Law. This matters because it prevents future feature proposals from inventing incompatible primitives. If a new subsystem cannot be expressed through that ontology, it is invalid under the latest canon.

This could be revisited only if the user supplies a newer canon that changes the ontology.

### BUILD-ID-0 replaces earlier build-number models

This is one of the most important decisions. Early in the chat, a global build number was imagined as something that could increment on every build. TESTX modified that toward test-owned increments. The latest upstream canon replaced both with BUILD-ID-0.

**FACT:** GBN is release-only and centrally allocated. BII is used for dev/ci. This decision avoids local/branch/fork collisions and gives release artifacts stronger provenance.

This decision affects CMake, CI, release scripts, changelogs, artifact names, package manifests, and any old build-number scripts in the repo.

### Engine C89 and game C++98

**FACT:** Latest upstream canon states engine C89 and game C++98. This aligns with earlier toolchain support discussions. The visible rationale is legacy compatibility, stable public headers, and no C++ ABI leakage.

The alternative would be to modernize everything uniformly. That was rejected by the toolchain/legacy support goals. Applications and tools may use newer languages, but they must not impose requirements on engine/game.

### Applications are shells and UI is data

**FACT:** Latest upstream canon says applications are orchestration shells; CLI is canonical; GUI/TUI are views over the same command graph; UI is data; tools are read-only by default; setup is only install mutation authority; launcher is reference app.

This decision protects the simulation and install state from hidden mutation paths. It also makes WYSIWYG UI work safer because UI layout and bindings can be treated as data rather than logic.

### Content is optional and capability-based

**FACT:** Earlier sister-chat prompts/responses accepted zero-asset boot, optional packs, capability-based content resolution, and no hardcoded content. This was reinforced by latest upstream “no hardcoded content or magic defaults.”

This decision was made to support headless servers, CI, open packs, mods, old binaries, and flexible distributions. It rejects mandatory base packs and implicit Earth/Sol assumptions in engine.

### Control layers are external and non-interfering

The DRM/anti-cheat scratchpad proposed that these systems belong outside engine/game. TESTX2 formalized that. The rationale was that control systems can break determinism and trust if embedded in core.

This decision affects future DRM, anti-cheat, telemetry, moderation, licensing, and platform integrations. They can exist, but only as optional external capability-gated layers.

### Demo/free/tourist/full are authority profiles

The user accepted the idea that a demo should not be a separate codebase. The chat concluded that demo/free/full/service states should be authority profiles over the same binaries and content.

This was chosen because it minimizes buyer friction, avoids code forks, supports honest demos, and makes piracy less meaningful. It depends on server-side authority for durable value.

### IDEs are projections

This decision came from the user’s desire to use old and new IDEs without managing separate project files manually. The conclusion was that IDE projects must be generated views, not source of truth.

This decision affects repository layout, CMake, CI, `.gitignore`, project files, and GUI design workflows. It might be revisited only if an IDE cannot function without some committed packaging project, in which case that project must be explicitly grandfathered and documented.

### Separate binaries per OS/toolchain/floor

The user supplied a strict matrix. The decision is that compatibility is enforced by build discipline, not hope. One binary cannot honestly target all OSes. Compiler, SDK, CRT, ABI, and runtime floors define reality.

This affects artifact naming, CMake presets, test matrices, VM strategy, and support policy.

### TESTX4 and policy-as-data are recommended, not proven implemented

The language-profile proposal was judged excellent, but this chat did not establish that it has been adopted as final upstream canon. It should be carried forward as a planned or recommended layer unless another chat confirms adoption.

## 6. Ideas Considered but Rejected, Superseded, or Deprioritised

### Global build number incrementing on every build

This was considered at the beginning because the user wanted every build to have a unique global identity. It was superseded because later canon separated release identity from local/dev/CI identity. BUILD-ID-0 is cleaner and avoids branch/fork/local conflicts.

The rejection is final unless BUILD-ID-0 changes.

### Demo as separate product/build

This was considered implicitly through the Steam demo discussion. It was rejected because the user explicitly did not want separate code or a separate product. The authority-profile model is preferred.

It could become relevant only as a storefront packaging workaround, but not as a separate codebase.

### Strong client DRM or engine/game anti-cheat

This was considered when discussing piracy and control systems. It was rejected because it would compromise determinism, testability, modding, trust, and replay. The project instead uses server authority and entitlements to deny durable value extraction.

This rejection is final under current tenets.

### Silent renderer fallback

The application/platform response identified a tension between graceful fallback and no silent fallback. The resolution rejected silent fallback in tests and explicit requests. `--renderer=vulkan` must fail if Vulkan is unavailable. `--renderer=auto` may fallback only with explicit logging.

### One binary for all OS floors

This was rejected by the toolchain matrix. OS compatibility is not just language standard. SDK, CRT, ABI, loader behavior, and toolchain matter. Separate artifacts are required.

### Collapsing Mac support into one Xcode

Rejected because Apple removes SDKs and old OS targeting cannot be assumed from a modern Xcode.

### Using modern VS/v143 for XP

Rejected because VS2022/v143 cannot target XP. Older toolsets are needed. v141_xp is a practical MVP path, but archival support still needs older environments.

### Treating IDE projects as source of truth

Rejected by REPOX. IDE project files are projections or documented packaging inputs.

### Hardcoding Earth/Sol or mandatory base content

Rejected by zero-asset/capability canon. Earth/Sol may exist as content, but the engine must not require them.

### Putting gameplay logic in applications or UI

Rejected by APP canon. Apps are orchestration shells; UI is data.

### Continuing architecture redesign

Rejected by latest upstream canon. The architecture is closed.

## 7. Important Reasoning and Rationale

The visible reasoning throughout the chat was about controlling complexity before it becomes irreversible. Dominium / Domino is large enough that ordinary game-project habits—ad hoc build scripts, hand-managed IDE projects, silent fallbacks, content baked into binaries, UI logic, and client trust—would create long-term failure.

One major tradeoff was between convenience and longevity. For example, a single modern toolchain would be easier, but it would not honestly support old OSes. Separate toolchains and artifacts are more work, but they preserve mechanical truth. Similarly, letting IDEs manage projects would be convenient in the short term, but dangerous across decades. Treating IDEs as projections is more disciplined.

Another tradeoff was between strong DRM and core integrity. The chat explicitly preferred preserving determinism and auditability over attempting impossible piracy prevention. The visible conclusion was that unauthorized users can copy bits, but they should not be able to gain durable value, affect multiplayer, or consume services without authority.

The user appeared to care most about:

* long-term correctness;
* avoiding refactors;
* rigorous testing;
* modularity;
* old and new platform support;
* mechanical enforcement;
* not losing context across chats;
* avoiding hidden assumptions.

A recurring risk being managed was “silent drift”: small local changes that gradually break canon. That is why TestX, RepoX, rules-to-checks, policy-as-data, UI binding validation, build identity rules, and IDE quarantine all became important.

## 16. Final Plain-English Summary

This chat was a long governance and continuity conversation for the Dominium / Domino project. It was not mainly about adding a gameplay feature. It was about making sure the project can survive implementation across many products, teams, chats, operating systems, toolchains, IDEs, distribution models, and future transitions without losing its core principles.

At the start, the user wanted a Git/CMake/versioning system: global build number, manual semantic versions per product, automatic changelogs, protocol/API/ABI versioning, platform builds, packaging, and distributable naming. Early answers proposed a global build number that could increment frequently, but this later changed. The latest upstream canon introduced BUILD-ID-0, which is now the controlling model: product SemVer is manual; build kind is dev/ci/beta/rc/release/hotfix; GBN is centrally allocated and used only for beta/rc/release/hotfix; BII is always present for dev/ci; no distributed artifact may lack GBN; and local builds may not allocate GBN. This supersession is one of the most important facts from the chat.

The conversation then broadened into modular content and asset-free operation. The project should boot with zero assets, use open-format content packs, resolve content through capabilities, and avoid mandatory base packs or hardcoded Earth/Sol assumptions in engine. This enables headless servers, CI, old-binary support, modding, archival use, and flexible distributions.

The user used this chat to coordinate other chats. Prompts were generated for engine/game/content and application/platform/render teams. Those chats responded with acceptance. Later, TESTX arrived as an authoritative self-defending test/governance system. TESTX shifted the conversation from design to enforcement: deterministic tests, no wall clock, no uncontrolled RNG, no floating point in authoritative logic, assertion tiers, blocker tracking, and CI gates. Later upstream canon refined this further and locked architecture.

Control systems became TESTX2. The chat concluded that DRM, anti-cheat, copy protection, telemetry, and similar systems must not be engine/game features. They may exist as optional external control layers, but must be capability-gated, disabled by default, discoverable, and non-interfering with authoritative simulation. Engine/game must not contain secrets or enforcement logic.

Demo/free distribution became TESTX3. The user wanted one package, no separate demo code, and low buyer friction. The chat concluded that the project should sell authority, not bits. Free users can have the same installation and see much of the world, but durable value—persistent progression, economic impact, multiplayer mutation, service access—requires authority. Tourist mode became an important idea: free users can join real servers as non-mutating observers or explorers. Piracy containment is therefore about preventing durable value extraction, not preventing copying.

RepoX emerged when the user wanted to use many old and new IDEs to design GUI apps and build binaries. The conclusion was that IDEs must be projections, not sources of truth. CMake plus TestX/RepoX own the build graph. IDE project files should be generated and disposable, ideally under `/ide`, with only documented packaging exceptions. This protects the repo while allowing VC6, VC7.1, VS2015, modern Visual Studio, CodeWarrior, Xcode, and Linux toolchains to coexist.

The user also supplied a strict OS/toolchain/language matrix. Compatibility is not just language standard. Compiler, SDK, CRT, ABI, and runtime testing define reality. Separate binaries are required for separate OS/toolchain floors. The user later installed Windows 10 with VS2022 and received a practical MVP recommendation: use VS2022 as host with v141/v141_xp and pinned SDKs to build XP through Windows 11 test floors. This is useful for MVP, but not a replacement for true archival VC6/VC7.1 VMs.

A later proposal introduced feature allowlists by component: engine C89, game C++98, public C ABI boundaries, restricted C++ use, no exceptions/RTTI/static init side effects across boundaries, no STL/FP in deterministic paths, and tools allowed to use modern C++17. This was treated as a strong TESTX4 recommendation, but not proven implemented.

Near the end, the user pasted a new upstream canon stating that architecture is closed, canon is locked, and this work is now implementation/audit/maintenance only. It also introduced or confirmed APP-CANON, CLEAN-2, CANON_INDEX, BUILD-ID-0, SRZ-0, VALIDATE-0, engine C89, game C++98, and the ontology Assemblies/Fields/Processes/Agents/Law. That upstream canon is the highest source from this chat.

The final work was about preserving the chat. A context transfer packet was generated, then a downloadable report package, then an in-chat reader version, and now this human-readable report.

What remains to do is verification. The next assistant should inspect the actual repo for `CANON_INDEX.md`, docs status headers, BUILD-ID-0 docs and scripts, `.dominium_build_number`, `update_build_number.cmake`, `CMakePresets.json`, CI workflows, IDE artifacts, UI binding validation, language compliance, zero-asset boot tests, authority profiles, and VS2022 toolsets. The next step is audit, not redesign.
