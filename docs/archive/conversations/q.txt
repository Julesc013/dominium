## 3. Main Topics We Discussed

### Topic 1 — The existing Dominium DUI system

The most important technical discovery was that the project already has a UI system. **FACT:** The user described the current launcher GUI as the Dominium UI, or DUI, schema/state system using TLV. It has at least three backends: Win32 common controls through `comctl32`, DGFX through Domino’s renderer/event pump, and a null backend.

This topic came up because the assistant asked what the current launcher UI stack was. Once the user answered, the architecture shifted from “choose a generic tech stack” to “extend and author DUI.” This matters because it avoids creating a parallel UI system. Instead of adopting Qt, WPF, or another framework, the plan became to build authoring tools around the project’s existing UI runtime.

The discussion also identified likely flicker causes. The Win32 backend is mostly child HWND controls and does not use custom painting. Flicker likely comes from resize relayout, redraw calls, list refresh invalidation, and visibility toggles. This later led to a planned prompt for structural Win32 batching and coalesced relayout.

What remains uncertain is the actual state of the code. The user named files, but the chat did not inspect the repository. A future assistant must verify exact paths, build targets, TLV parser implementation, and current launcher behavior.

### Topic 2 — UI Editor first, Tool Editor second

A major planning topic was the sequence of tools. The original user request was for a “Tool Editor,” but the conversation refined this into two stages.

**FACT:** The user explicitly chose to make the UI Editor first. This first tool is a minimal Windows-only editor that can visually edit layouts and generate TLV directly. **FACT:** The Tool Editor comes later and is intended to become the full first-class DUI authoring environment.

This matters because the final Tool Editor is too large to build as the first move. The UI Editor is a bootstrap tool. It should become capable enough to create the Tool Editor’s own UI. Then the Tool Editor can eventually edit itself, the setup program, the launcher, the game, and other tools.

The tension here is that the user wants the final Tool Editor to avoid hardcoded limitations. The assistant recommended allowing a minimal hardcoded docking substrate in the earliest Tool Editor version if necessary, but this remains tentative. The user clearly prefers the eventual Tool Editor to be free of hardcoded limitations.

### Topic 3 — Native widgets and pixel-perfect behavior

Native widgets were a recurring theme. **FACT:** The user wanted true OS controls, not merely native-looking custom rendering. This means real Win32 controls on Windows, and eventually platform-native controls on other systems where possible.

This requirement shaped almost every design decision. It argued against heavy cross-platform frameworks and pure custom renderers. It also means that pixel-perfect design must be understood carefully: native OS controls vary across OS versions, themes, DPI modes, and platform conventions. The plan therefore included a backend capability system and per-platform handling rather than pretending one static layout can be identical everywhere.

For Phase A, the UI Editor is much simpler: Windows only, system DPI only, design scale 100%. For the Tool Editor, the user wants per-monitor DPI with fallback and multiple design scales. The difference between these scopes is important. Future assistants should not accidentally impose the Tool Editor’s full DPI/scale requirements on the first UI Editor.

### Topic 4 — Canonical data model: IR, TLV, JSON mirror

The conversation spent significant effort on safe data handling. The user selected the safest proposed approach: a canonical in-memory document model that writes deterministic TLV. This rejected direct TLV byte patching.

The agreed direction is that TLV is the canonical source of truth. JSON can be emitted as a deterministic mirror by default if small, but it is not intended for manual editing. The rationale is practical: TLV can remain the project/runtime format, while JSON helps humans review diffs and debug changes.

The assistant also recommended splitting authoring documents from runtime state. The user said the separation seemed logical. This became a planned `ui_doc.tlv` plus optional `ui_state.tlv` model. This matters because an editor should not accidentally save transient runtime state such as scroll positions or selected items into the canonical UI definition.

Unresolved details include the actual TLV wire format, migration strategy from `launcher_ui_v1.tlv`, and how much legacy data can be preserved during import.

### Topic 5 — Layout model and authoring features

The user wanted both absolute layout and dynamic/layout-container behavior. The assistant recommended a deterministic layout model where docking takes precedence over anchors, and anchors take precedence over absolute positioning. The Phase A minimum eventually included absolute positioning, anchors, docking, stack row/column, Splitter, Tabs, and ScrollPanel. GRID was suggested as deferrable to Phase B, though that is not a final user decision.

This topic matters because the UI Editor must be powerful enough to build the future Tool Editor. A simple absolute-position-only editor would not be enough for complex tool layouts. Splitters, tabs, and scroll panels became especially important because the Tool Editor itself needs hierarchy panes, property inspectors, canvas areas, logs, tabs, and scrollable panels.

The unresolved risk is implementation complexity. Real native tab controls, splitter behavior, and scroll panels can be nontrivial in a child-HWND Win32 UI.

### Topic 6 — Action binding and generated stubs

The user wanted both tools to generate C++ stubs and wire them to stable IDs. The assistant recommended storing action bindings in TLV as stable string keys, then compiling them to numeric action IDs through a registry. Generated code would be split into generated files and user-owned files to avoid overwriting user edits.

The planned event system uses a unified C-compatible `domui_event` structure. This lets Win32, DGFX, null, and future backends dispatch events consistently. The stable action key strategy also supports Codex automation: a script can bind `launcher.nav.play` to a button without needing to know a C++ function pointer or numeric ID in advance.

This is a strong planned design, but some details remain assistant recommendations rather than explicit user-confirmed decisions. Future implementation should preserve that uncertainty if writing a formal spec.

### Topic 7 — Importing existing tool UIs

The user later made clear that the UI Editor must import existing tools and modify their UIs. This changed the UI Editor from a new-document authoring tool into a migration and maintenance tool.

The resulting plan added repository UI discovery, an index file, legacy import workflows, import reports, and safe export/replace workflows. The key safety rule is that legacy UI files should not be overwritten implicitly. The UI Editor should import a legacy TLV into a canonical document, report unmapped or approximated fields, and only export/replace a tool UI through an explicit action.

This matters because the launcher already has an existing UI. If the UI Editor cannot work with existing UIs, it will not solve the immediate launcher problem.

### Topic 8 — CLI automation and ops.json

The user wanted Codex to be able to create designs automatically. This led to the CLI and `ops.json` plan.

The CLI modes include validation, formatting/canonicalization, code generation, UI building, and later applying scripted edits. The `ops.json` format is a pure-data edit script. It can create widgets, delete widgets, set rects, set layout, set properties, bind events, validate, and save. Selectors can refer to widgets by ID or path.

This matters because it turns the UI Editor into an automation substrate, not just a manual GUI tool. Codex can generate a UI script, apply it headlessly, validate the result, generate stubs, and build the target tool.

The risk is that deterministic scripting requires careful parser behavior, stable selectors, and idempotent conventions. Ambiguous widget paths or non-idempotent creation operations could make automated designs unstable.

### Topic 9 — Launcher and setup capability tests

The user wanted a concrete test: have Codex use the UI Editor to create launcher and setup designs similar to Minecraft’s launcher/setup layouts. The user later clarified that this refers only to the logical layout.

The planned launcher has a native-control structure: header/banner, left sidebar navigation, main tabbed area, news scroll page, instances list/detail page, settings page, and footer/status bar. The planned setup UI is a wizard: welcome page, install location page, components/options page, progress page, finish page, and footer navigation buttons.

These tests matter because they prove the UI Editor is capable before building the Tool Editor. If the UI Editor cannot import, script, generate, codegen, and build real launcher/setup UIs, it is not ready to bootstrap the Tool Editor.

The unresolved issue is that the uploaded screenshot bundles were not inspected. The logical specs are accepted planning artifacts, not verified screenshot extractions.

### Topic 10 — IDE-native live editing

At the end, the user asked about using Visual Studio 2026 to live edit Windows tool UIs, Xcode to live edit macOS tool UIs, and Linux tools for Linux UIs. The generated prompt proposed a preview-host strategy. This approach preserves the canonical DUI/TLV model. It does not make Visual Studio or Xcode designer files the source of truth. The assistant explicitly framed import from `.rc`, `.xib`, or GtkBuilder XML as optional future one-way import, not canonical editing.

## 7. Important Reasoning and Rationale

The visible reasoning in this chat centered on managing scope without sacrificing the long-term architecture.

The first tradeoff was speed versus final completeness. Building the full Tool Editor directly would better match the end-state vision, but it would require too many systems at once. The UI Editor-first strategy reduces initial risk while still contributing to the final system. The UI Editor becomes a bootstrap tool, not a throwaway if it shares the canonical IR, TLV writer, validator, layout engine, and codegen infrastructure.

The second tradeoff was native controls versus custom rendering. Native controls are harder to make perfectly uniform across OS versions and DPI settings, but they satisfy the user’s requirement for native/OEM behavior. Custom rendering would give more visual control but would fail the true-native-control requirement. The compromise is to use native widgets for standard controls and allow custom rendering only for custom widgets or backends that cannot support a given feature.

The third tradeoff was binary TLV versus human-readable authoring. TLV is already part of the project and likely efficient for runtime. But humans and review tools need visibility. The chosen compromise is TLV as canonical plus deterministic JSON mirror. This lets source control and debugging work without changing the authoritative format.

The fourth tradeoff was Windows-first implementation versus cross-platform future. Since the immediate broken UI is the Windows launcher, Windows-first is practical. But the generic capability system and backend-neutral layout engine are intended to prevent Windows assumptions from infecting the schema.

The fifth tradeoff was GUI authoring versus automation. A visual editor helps humans, but Codex needs a deterministic non-interactive surface. The CLI and `ops.json` engine solve this by making UI edits reproducible and scriptable.

The sixth tradeoff was IDE-native editing. Visual Studio and Xcode have their own native designer ecosystems, but those do not naturally edit DUI TLV. The preview-host proposal preserves DUI as canonical while using IDE build/run/watch loops. This is practical but may not fully satisfy a desire for direct IDE designer manipulation.
