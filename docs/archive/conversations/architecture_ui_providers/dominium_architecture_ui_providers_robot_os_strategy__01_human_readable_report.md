## 1. One-Page Orientation

This chat was a strategic architecture and design convergence session for Dominium/Domino. It began from a transferred knowledge base about rebuilding Dominium’s GUI and binary strategy, then evolved into a broad platform-specification discussion covering repository structure, product shells, renderer/platform priorities, native applications, language standards, provider architecture, Workbench, UI authoring, and finally the game’s core identity as a robotic seed-civilisation simulator with a diegetic robot operating-system interface.

The user was trying to prevent Dominium from becoming a brittle one-off project. The repeated concern was that the codebase, files, directories, APIs, schemas, renderers, GUIs, and data formats should remain portable, modular, extensible, replaceable, and reusable for other Domino-engine games or even unrelated engine/game projects. The conversation therefore treated the project less like a simple indie game and more like a long-lived platform or OS-like codebase. That framing shaped almost every technical recommendation: stable contracts, replaceable implementations, service boundaries, provider manifests, compatibility matrices, command/action/view/document systems, and validators.

The earliest phase dealt with GUI and binary architecture. The retained direction was that every product should have CLI, every product should have at least TUI or deterministic TUI stubs, and GUI support should be modular. Setup and launcher should use OS-native/OEM+ apps where appropriate; the client should use a rendered UI; server remains headless/CLI/TUI first; tools are case-by-case. The user then pushed on directory structure, wanting fewer top-level folders and a more future-proof organization. The resulting doctrine was ownership-based roots: apps compose, runtime implements, contracts define, content supplies, tools validate, and archive preserves historical material.

The conversation then reworked the platform and rendering plan. A key correction was that `vector2d` should not be treated as a renderer backend; it is drawing/canvas functionality. A software renderer remains useful as a CPU framebuffer/reference/test/fallback path. After the user announced the project had moved to C17/C++17 with floors of Windows 7 SP1, Mac OS X 10.9.5, and Linux, older fixed-function targets such as OpenGL 1.1/2.1 and Direct3D 5/7/8/9 were deprioritised to research/back-port lanes. OpenGL 3.3 became the preferred first direct hardware target, followed by Direct3D 11, then Metal/Vulkan/Direct3D 12 later. The user also brought in raylib, SDL2, Lua, rlgl, rlsw, raygui, raudio, rtextures, and rmodels as potential accelerators. The best synthesis became: use them aggressively, but only as replaceable providers behind Dominium-owned service contracts.

A major part of the chat was UI/UX architecture. The user wanted CLI, TUI, rendered GUI, native GUI, and later VR/immersive UI to be projections of one shared presentation model rather than four independent UI systems. Workbench became the planned authoring environment for views, widgets, layouts, themes, HUDs, TUI panels, rendered screens, workspaces, and UI packs. The final UX turn connected this to gameplay: because players are robot avatars whose consciousness boots into nodes and downloads into manufactured bodies, the UI can be diegetic. Menus, HUD, Workbench, TUI, CLI, and immersive shells can all feel like a robot OS.

The most important design shift came near the end. The user stated that Dominium should be a robotic seed-civilisation game, not a labour-management game. Players are machine consciousnesses/robot avatars. The mothership contains human knowledge, starter resources, fabrication capacity, and AI planning. The planet is generated from science-bound parameters. Players explore, mine, refine, fabricate, build cities, spawn labs, industrial networks, and eventually terraform or build megaprojects. The sim focuses on energy, materials, machines, logistics, sensors, fields, permissions, heat, maintenance, construction, and machine graphs, not human workers.

Future relevance: this chat should feed several chapters of a master Project Spec Book, including architecture doctrine, repo layout, provider strategy, renderer plan, language/toolchain baseline, UI/Workbench platform, Robot OS UX, and game design pillars. A future assistant must not flatten everything into “use raylib” or “robot UI theme.” The essence is: first-party contracts and simulation law; replaceable providers; profile-selected products; shared UI projections; robotic seed-civilisation as the game identity.

### 2.3 Platform/render/native-app evolution

The chat then moved into platform and renderer planning. Initially the discussion included older lanes such as OpenGL 1.x/2.x, Direct3D 9, DX12, Vulkan, Metal, Win32, Cocoa, X11/Wayland, SDL2, AppKit, WinForms, SwiftUI, WinUI, GTK, and others. Over time, the plan sharpened. The user challenged `vector2d`, leading to the conclusion that it should be a drawing/canvas layer, not a renderer backend. Software renderer was defined as a CPU framebuffer renderer for tests, fallback, and reference behavior, not necessarily a full 3D engine.

When the user announced the transition to C17/C++17 and minimum systems of Windows 7 SP1, Mac OS X 10.9.5, and Linux, many old lanes became irrelevant for first-wave work. OpenGL 1.1/2.1 and old DirectX were deferred. OpenGL 3.3 became the first direct hardware target, Direct3D 11 the Windows target, and Metal/Vulkan/DX12 later advanced lanes.

### 2.4 Broad platform engineering doctrine

The user asked what practices would make the code portable, modular, extensible, reusable, replaceable, and future-proof. The response expanded into general engineering doctrine: stable contracts, replaceable implementations, schema/version discipline, C-compatible ABI boundaries, command/view/result/refusal models, deterministic replay, provider manifests, forbidden include checks, conformance tests, semantic IDs separate from file paths, and validators. The large principle became: contracts are stable; implementations are replaceable.

### 2.5 Workbench and shared UI platform

The user then focused on CLI/TUI/rendered/native GUI sharing and Workbench. The conversation converged on a command/action/result/document/view/patch/projection spine. Workbench should not be a one-off editor or raygui object editor. It should author Dominium UI artifacts: views, layouts, widgets, themes, TUI panels, HUDs, workspaces, and UI packs. Those artifacts are then projected into command, text, rendered, native, headless, and later immersive shells.

### 2.6 Raylib/SDL/Lua provider doctrine

The final synthesis was not “make Dominium a raylib game.” It was: use raylib aggressively as a seed provider suite, SDL2 as a platform/input/audio provider, Lua as a pinned script provider, and Dominium’s own contracts as the authority. Directory structure should remain service-first: `runtime/render/providers/raylib/`, not `runtime/raylib/render/`.

### 2.7 Game design shift: robotic seed civilisation

The user then supplied the strongest game-design direction. Dominium should become a robotic seed-civilisation game, not a labour-management game. Players are robot avatars/machine consciousnesses. The mothership provides knowledge, starter fabrication, spawn capacity, and AI planning but is limited by feedstock, throughput, heat, energy, safety protocols, reserves, rare materials, and scale. Gameplay becomes exploration, surveying, mining, refining, fabrication, construction, logistics, sensor networks, spawn labs, terraforming, and megaprojects. Human worker NPCs are deliberately absent from the core.

## 3. Main Topics Discussed

### Topic 1 — GUI, binary, and product shell architecture

The chat preserved the earlier direction that every Dominium product should always work through CLI, should have TUI or deterministic TUI stubs, and may have modular GUIs. Setup and launcher should be OS-native/OEM+ where useful; the client should be rendered-first; server is headless/CLI/TUI first; tools are case-by-case. The conclusion remains that GUIs are thin shells over shared contracts, not independent product architectures. Uncertainty remains around exact native-shell implementation order and how much native GUI is first-wave versus deferred.

### Topic 2 — Repository and directory structure

The discussion repeatedly emphasized ownership-based layout. Top-level folders should answer “what architectural layer owns this?” rather than “what topic is this?” The target roots include apps, engine, game, runtime, contracts, content, docs, tests, tools, release, archive, scripts, cmake, external, and possibly sdk/examples. The user later pasted a report saying top-level structure is largely clean, but residual issues remain: schema taxonomy, pack internal layout, runtime/engine `compatx`, project graph, serialization/session placement, AIDE state classification, and report-bundle integrity. This should feed the repo-layout and governance chapters of a spec book.

### Topic 3 — Renderer architecture and software renderer

The chat clarified that renderer backends and drawing primitives are separate. `software` means a CPU framebuffer renderer useful for tests, fallback, deterministic screenshots, and a proof that UI/game code does not require GPU APIs. `vector2d` should not be a renderer; it is drawing/canvas functionality. After the C17/C++17 and system-floor change, OpenGL 3.3 became the preferred first direct hardware renderer, D3D11 second, Metal/Vulkan later, and old fixed-function/old DirectX deferred.

### Topic 4 — Platform/provider architecture

The strongest synthesis was service-first/provider-backed/profile-selected/contract-governed/third-party-fenced. Services are Dominium-owned: platform, input, render, draw, audio, asset, script, UI, storage, diagnostics. Providers implement them: null, raylib, rlgl, rlsw, SDL2, raudio, Lua, later OpenGL33/D3D11/Metal/Vulkan/native host providers. Profiles select provider combinations. Third-party types must not cross into engine, game, contracts, content, saves, replays, packs, or public SDK.

### Topic 5 — Language and compatibility baseline

The user stated that the project had transitioned to C17 and C++17 and raised minimum systems to Windows 7 SP1, Mac OS X 10.9.5, and Linux. The visible conclusion was to use C17 for ABI-facing law-like structures, packets, fixed-width types, deterministic math, save/replay records, and low-level facades; C++17 for runtime machinery, provider implementations, Workbench, apps, resource ownership, and orchestration. Public boundaries remain C-compatible and POD/versioned.

### Topic 6 — Workbench as UI authoring environment

Workbench should be the tool that designs and edits rendered GUI, TUI, HUD, widget, theme, and workspace artifacts. It should edit Dominium documents and patches, not provider-specific widgets. It should support View Studio, Layout Studio, Control Catalog, Theme Lab, TUI Studio, Rendered GUI Studio, Preview Matrix, and Validation/Evidence panels. The first proof should not be a full interface editor; it should be a practical dashboard such as validation summary or node/body status shown across projections.

### Topic 7 — Shared UI/UX system

CLI, TUI, rendered GUI, native GUI, headless, and immersive/VR should be projections of the same command/action/view/document system. The UI is not just buttons and panels; the reusable truth is commands, results, refusals, diagnostics, evidence, actions, views, documents, patches, and capabilities. This lets menus, setup, launcher, Workbench, server admin, client HUD, and robot OS shells share one architecture.

### Topic 8 — Robotic seed-civilisation game design

The user explicitly defined the core game direction. Players are robots/machine consciousnesses seeded by a mothership. The game simulates energy, materials, machines, logistics, sensors, permissions, fields, construction, heat, maintenance, and machine graphs rather than human workers. The mothership gives knowledge and precision but not scale. Players build industrial chains, remote spawn labs, cities, and eventually terraforming/megaproject infrastructure. Nanotech is an actuator, not magic. Machine assemblies compile into functional process graphs.

### Topic 9 — Planet generation, fields, and terraforming

The game should use star-system/planet generation, not initially a “galaxy generator.” It should support forward generation from physical parameters and reverse generation from desired gameplay/planet characteristics. A common field-layer stack should support procedural generation, real data, overrides, mod packs, terraforming, construction, damage, and temporary previews. Gameplay layers must conserve matter/energy where appropriate; editor/worldgen layers need not.

### Topic 10 — Robot OS, diegetics, and epistemics

The final UX direction is that the player boots into an OS, connects to nodes/universes, downloads consciousness into fabricated bodies, and operates through sensor/communication networks. The UI should reflect what the player’s body, faction, node, or sensor network can know. Facts should have metadata such as source, confidence, observation tick, network path, latency, and permission. Fog-of-war becomes diegetic and anti-cheat relevant.

## 7. Important Reasoning, Rationale, and Tradeoffs

The central tradeoff was speed versus purity. Raylib, SDL2, Lua, raygui, rlgl, rlsw, and related libraries can accelerate visible progress, but if they leak into game law, UI documents, saves, replays, or public contracts, they become architectural capture. The chosen compromise is to use them as providers behind Dominium-owned services and conformance tests.

A second tradeoff was breadth of compatibility versus first-wave progress. Earlier ambitions included very old Windows, old OpenGL, old DirectX, old macOS, 32-bit, and even exotic/retro systems. After the C17/C++17 and system-floor shift, these became back-port/research lanes. The project still preserves modularity for them, but it does not let them dictate the first renderer or UI architecture.

A third tradeoff was deep simulation versus playability. The seed-civilisation design can explode into full physics, ecology, climate, fluid, heat, economics, logistics, and AI. The visible rationale was to start with systems that produce gameplay leverage: field layers, cut/fill, material/energy-constrained construction, machine graphs, logistics, heat, maintenance, permissions, and fog-of-war. Full groundwater, ecology, dust storms, and atmospheric chemistry can come later.

A fourth tradeoff was customization versus trust. The user wants fully customizable UI/UX like an OS. That is compatible with MMO integrity only if the client can style, arrange, and summarize known information but cannot reveal hidden data, bypass permissions, hide critical warnings, or automate unauthorized actions.

## 16. Compact Human Summary

This chat was one of the major convergence points for Dominium. It began from a transfer brief about GUI and binary strategy, but it grew into a broad architecture and design foundation for the whole project. The user’s recurring concern was that Dominium must not become a one-off indie codebase or a pile of platform-specific UI and renderer hacks. The project should be portable, modular, extensible, replaceable, and reusable: not just for Dominium, but for other games on the Domino engine and possibly other future engine/game projects.

The early discussion preserved the GUI doctrine: every product should have CLI, TUI should exist or be stubbed deterministically, GUIs are optional/modular, and the product logic must remain behind stable backend contracts. Setup and launcher can use OS-native/OEM+ GUI shells; client is rendered-first; server is headless/CLI/TUI first; tools are case-by-case. This was tied to the repo structure problem: the source tree should be organized by architectural ownership, not by topics or historical accidents. The desired top-level roots are apps, engine, game, runtime, contracts, content, docs, tests, tools, release, archive, scripts, cmake, external, and later sdk/examples if needed.

The renderer discussion evolved significantly. Initially there were old lanes like OpenGL 1.x/2.x and Direct3D 9, but after the user stated that the project had moved to C17/C++17 and now targets Windows 7 SP1, Mac OS X 10.9.5, and Linux, the first-wave plan changed. Old fixed-function renderers and old DirectX should be deferred to research/back-port lanes. OpenGL 3.3 should be first because it supports early portable shader work across Windows/Mac/Linux, then Direct3D 11 for Windows, then Metal, Vulkan, and Direct3D 12 later. A key correction was that `vector2d` should not be a renderer backend; it is drawing/canvas functionality. Software rendering remains useful as a CPU framebuffer/reference/test/fallback path.

The strongest architecture synthesis was the service/provider/profile model. Dominium owns services and contracts: platform, input, render, draw, audio, asset, script, UI, storage, diagnostics, commands, actions, views, documents, patches, saves, replays, packs, profiles, and provider law. Third-party libraries implement providers. Raylib, rlgl, rlsw, raygui, raudio, rtextures, rmodels, SDL2, and Lua can be used aggressively, but only as replaceable provider implementations. Their types must not leak into engine, game, contracts, content, saves, replays, packs, game law, or public SDK. Apps remain generic; profiles select provider combinations.

The Workbench and UI discussion established that CLI, TUI, rendered GUI, native GUI, headless, and future immersive/VR interfaces should not be separate UI systems. They should be projections of one command/action/view/document model. Workbench becomes the authoring environment for UI artifacts: widgets, layouts, themes, TUI panels, rendered screens, HUDs, workspaces, preview fixtures, and UI packs. It edits Dominium documents and patches, then validates and previews them through providers.

The biggest design decision came near the end: Dominium should become a robotic seed-civilisation game, not a labour-management game. Players are robot avatars or machine consciousnesses. A mothership contains human knowledge, starter resources, spawn fabrication, and AI planning, but is limited by feedstock, energy, heat, rare materials, safety protocols, and throughput. Gameplay is exploration, surveying, mining, refining, fabrication, power, logistics, sensors, permissions, spawn labs, industrial cities, terraforming, and megaprojects. Nanotech is a construction actuator, not free matter or magic. Machines compile into process graphs. Fog-of-war is diegetic because robots only know what their sensors, satellites, relays, and faction networks know.

The final UI/UX synthesis connects directly to that premise. The interface should feel like a robot operating system. The player boots into an OS, connects to a universe, downloads consciousness into a fabricated robot body, and disconnects into backup/on-ice state. Menus, Workbench, HUD, CLI, TUI, GUI, and future VR shell are all projections of that OS. Users can customize themes, layouts, shells, HUDs, terminal styles, workspaces, and sensor overlays, but customization must not reveal hidden data, hide critical warnings, bypass permissions, or automate illegal actions.

The best next action is to implement the first provider and Robot OS wedges: provider manifest schema, forbidden include validator, null providers, raylib/rlgl/rlsw/raygui/raudio providers, client/workbench/server profiles, and one shared `view.node.status` projected into command, text, rendered OS screen, and HUD.
