### Launcher executes binaries

**FACT:** The user explicitly stated that launcher-selected settings will execute client or server binaries. This is final in this chat.

This decision made sense because Dominium targets many platforms and renderers. A single runtime binary capable of every role/backend combination would be more complex. Separate binaries let the build system produce curated client/server variants, while the launcher and scripts select the appropriate one.

This affects:

* Launcher core.
* Spawn API.
* Scripts.
* CMake targets.
* Capability matrix.

It would be revisited only if the project later wants runtime-loadable platform/render plugins or a universal binary model.

### Launcher backend C89, frontends C++98

**FACT:** The user explicitly required C89 backend code and C++98 frontend code.

This decision fits the project’s retro/portable constraints. The backend should contain the durable logic: accounts, instances, launch profiles, capabilities, and launch plans. The frontend can render UI and handle platform events.

This affects file layout. It implies future code like `game/launcher/core/*` should be `.c/.h`, while Win32/Cocoa/SDL frontends may be `.cpp` or `.mm` within C++98 constraints.

### Single-system client/server binaries

**FACT:** The user said client/server binaries can have single-system support. This was a practical concession. The launcher can be cross-platform in concept, but each game binary can target a specific platform/render combination.

This affects binary naming and script dispatch. It also makes capability filtering more important, because the launcher must not offer binaries that do not exist or cannot run.

### Unified scripts

**FACT:** The final user-facing commands are `dominium`, `setup`, `dom-client`, and `dom-server`.

This decision made sense because users should not need to remember binary names such as `dom_client-posix-sdl2-gl2`. Scripts provide stable entrypoints.

`dominium` and `setup` supersede earlier `dom-launcher` and `dom-setup` naming. Whether those older names remain as aliases is unresolved.

This avoids maintaining parallel `win32_legacy` versus `win32_full`, or `macosx_minimal` versus `macosx_full`, unless future technical evidence proves such a split unavoidable.

The assumption is that a single backend can be conservative enough for launcher use while still serving richer clients where appropriate. That assumption may break for extreme legacy targets, but the decision is final for now.

### Universal software renderer

**FACT:** The user explicitly requested a software renderer capable of both vector and graphics using texture files, working on all systems as backup.

This replaces `vector_soft`. It matters because the launcher can use software rendering, the game can fall back to software rendering, and missing texture files can still produce useful vector/placeholders.

The launcher should use the universal software renderer, not require GPU backends. A GPU-based launcher skin could exist later, but it is not the base design.

## 7. Important Reasoning and Rationale

The conversation was shaped by a few repeated tradeoffs.

First, portability versus maintainability. The user wants wide platform support, including modern, legacy, and retro systems. One response would be to create many special-case launchers and renderers. The user explicitly moved away from that. The rationale was to avoid duplicate minimal/full implementations and instead use full single implementations where possible, with a universal software renderer as fallback.

Second, frontend flexibility versus backend purity. The user wants launcher code to work across systems, but also accepts that platform-specific frontends may need C++98. The chosen model separates pure C89 logic from system-specific UI skins. That gives portability without forcing every UI implementation to be identical.

Third, visual flexibility versus sim determinism. The user wants cameras, 3D views, 2D views, vector modes, graphics modes, overlays, and CCTV. The visible reasoning was that these are client/render concerns. They should not affect the deterministic simulation. The solution is to keep camera/view/render mode state outside the sim.

Fourth, user convenience versus binary complexity. Multiple binaries are inevitable if client/server builds are single-system and renderers differ. Scripts solve that. The user-facing command remains simple even if the internal binary matrix is large.

Fifth, ambition versus verification. The support tiers are ambitious. The chat preserved them as product goals, but repeatedly marked external facts as needing verification. That distinction matters: “we want to support Linux 2.4” is a user decision; “this will build and run with our dependencies” is not established.

## 16. Final Plain-English Summary

This chat, **Dominium Architecture III**, was primarily about designing the launcher and the systems around it for a deterministic, portable, retro-to-modern simulation game called Dominium. The user began by establishing strict project constraints: Dominium is a deterministic fixed-step simulation with C89/C++98-era boundaries. `/engine/core` and `/engine/sim` are C89, core sim cannot use floats, deterministic tick order and serialization are non-negotiable, and renderer/audio/UI/platform code must not mutate sim state.

The first practical target was the launcher. The user wanted launcher tabs for News, Changes, Mods, Instances, Accounts, and Settings, plus bottom controls for account, instance, platform, renderer, graphical/headless, and client/server selection. The conversation quickly clarified that the launcher should not switch into client or server mode inside the same process. Instead, launcher selections should execute separate client or server binaries. That became a major architectural decision.

The user then set the language split: launcher backend/core code should be C89, while frontends for each binary can use C++98. This led to the core launcher architecture: a system-agnostic C89 core that knows about launch types, accounts, instances, profiles, platform/render capabilities, state, viewmodel, and launch plans; platform-specific process adapters; and frontend skins such as an engine-rendered launcher, native Win32 launcher, CLI, or TUI. Client/server binaries can be single-system builds, and the launcher/scripts dispatch to the right binary.

The user-facing scripts became another core outcome. The final commands are `dominium`, `setup`, `dom-client`, and `dom-server`. `dominium` launches the launcher. `setup` launches installer/repair/uninstaller behavior. `dom-client` and `dom-server` directly dispatch to client/server binaries. Earlier names like `dom-launcher` and `dom-setup` were superseded or left as possible aliases.

A major render design change occurred. The earlier idea of a separate `vector_soft` renderer was superseded. The user wants one universal **software renderer** that works on all platforms and can draw both vector graphics and texture-based graphics. This renderer is the fallback for launcher and game. Every renderer should support vector-only and full graphics modes so the game can run in CAD-style view, full textured view, missing-file fallback, or performance mode. The latest renderer list is DX9.0c, DX11, GL1.1, GL2, VK1, and software. DX12 was discussed earlier but omitted later, so it remains unresolved.

The view/camera design expanded considerably. The user wants instant seamless zoom at any scale, instant switching between top-down 2D and first-person 3D, and arbitrary cameras for free cam, map views, content creation, HUD cameras, CCTV, overlays, windows, and offscreen targets. These are client/render-side systems and must not affect deterministic sim state.

Tier 2 adds Windows 98 SE to Me, Mac OS 9 to 9.2, Mac OS X 10.6 to 15.0, and Linux 2.6.18 to current.
