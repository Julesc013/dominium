### Topic 3 — Platform API and platform backends

The platform layer, called `dsys`, is supposed to abstract OS-specific services: lifecycle, time, sleep, window creation, fullscreen/windowed mode, input events, filesystem, paths, directory iteration, and process spawning.

This topic came up because the user wants setup, launcher, tools, and game to run on many systems: modern Windows, old Windows, DOS, macOS, Mac OS Classic, Linux, BSD, Android-like systems, SDL, CP/M, and a custom Carbon/Z80 environment. It was therefore necessary to avoid direct platform calls in game/product code.

The conclusion was to use a `dsys` ABI with backend vtables and capability descriptors. Backends should advertise what they support. For example, CP/M or DOS may only support CLI/TUI, not windows or process spawning. Unsupported functions should fail clearly or be stubbed, not pretend to work.

A platform backend prompt template was generated, along with parameter sets for Win32, Win16, DOS32, DOS16, CP/M-80, CP/M-86, Carbon, Cocoa, X11, Wayland, POSIX, SDL1, SDL2, and null.

The unresolved issue is practical feasibility, especially for retro targets and Carbon OS. Their actual toolchains and system calls remain unverified.

### Topic 4 — Renderer API and renderer backends

The renderer layer, `dgfx`, became one of the major technical workstreams. The user wants support for many renderers and display systems: DirectX 7/9/11, Vulkan 1, OpenGL 1/2, QuickDraw, Quartz, Metal, GDI, VESA, VGA, CGA, EGA, XGA, Hercules, MDA, and software. Each backend should support 2D, 3D, vector, and raster graphics where possible.

The conclusion was to use a renderer command-buffer IR. Game/tools build a backend-agnostic command buffer with commands such as clear, viewport, camera, pipeline, texture, draw sprites, draw meshes, draw lines, and draw text. Renderers consume that IR according to their capabilities.

This matters because it allows the same game or tool rendering logic to run under modern GPU APIs, software renderers, and retro graphics modes. The software reference renderer, `softref`, is especially important because it becomes the canonical CPU implementation for testing semantics.

The unresolved issue is the exact IR payload layout and how far limited renderers should go. A CGA or MDA backend cannot realistically support modern 3D in the same way as Vulkan. The agreed approach is explicit capability flags and graceful fallback or no-op behaviour.

### Topic 5 — Unified UI and presentation independence

A major topic was how launcher, setup, tools, and game should present themselves. The user wanted a platform to be required but the renderer to be optional beyond software. The launcher and tools should be able to use native UI where appropriate, TUI/CLI where needed, or custom GFX UI when desired.

Initially, native GUI and custom GUI were discussed separately. The user then asked if they could be unified. The answer became a single `DOM_UI_GUI` mode with backend flags such as `DOM_UI_BACKEND_NATIVE` and `DOM_UI_BACKEND_GFX`. CLI and TUI remain separate modes: `DOM_UI_CLI` and `DOM_UI_TUI`.

This matters because it lets products be described once. A launcher can expose the same views and actions whether presented as a native Windows UI, a text UI, a CLI, or a custom rendered interface. It also makes renderer dependency optional for non-game products.

The conclusion was to define UI modes separately from UI backends. This is one of the central product architecture decisions.

### Topic 6 — Product contracts and graceful degradation

The user asked whether all this was enough to support headless, CLI, TUI, and GUI for setup, launcher, game, and tools under arbitrary platform/renderer combinations. The answer was yes in principle, but only if each product has an explicit **feature contract**.

A product contract says what the product requires and what is optional. For example, a launcher might require GUI native or allow fallback to TUI/CLI, while the game requires GFX. A setup CLI does not need a renderer. A tool with a preview canvas may need a renderer only for that canvas.

This topic mattered because “graceful degradation” can become dangerous if implicit. A program should not silently half-work because a renderer is missing. It should either disable optional widgets, fall back to a declared mode, or fail with a clear reason.

The conclusion was to use capability descriptors and runtime selection. Products should be described once, then bound to available drivers through a selector.

### Topic 7 — Package, mod, pack, and instance management

The package/instance system is central to the whole ecosystem. The user wants installed versions, official base/DLC/mod content, downloaded third-party mods and packs, shared data, instance-specific overrides, and arbitrary combinations without duplicating every possible mix.

The architecture treats installable units as packages. Package kinds include game code, game data, tools, launcher UI extensions, and shared packs. Instances are configured combinations of packages in load order, plus settings and paths.

This system supports setup, launcher, game, tools, and modding. Setup installs packages and creates default instances. The launcher lists packages and attaches/detaches them from instances. The game loads active packages for an instance. Tools inspect and build packages.

The unresolved issue is the exact manifest/schema format. JSON, TOML, and INI-like examples appeared in prompts, but no final schema choice was made.

### Topic 8 — Setup architecture

Setup was eventually planned as a cross-platform setup engine plus native wrappers. The user specifically wanted platform-appropriate installers. On Windows, that means MSI and EXE bootstrapper in an authentic Windows style. On macOS, app bundles, `.pkg`, and `.dmg`. On Linux, `.deb`, `.rpm`, `.run`, or AppImage-like formats. Retro systems get smaller TUI or console installers.

The key design is that native wrappers should not reimplement install logic. They should call or delegate to a common setup core that understands install scopes, packages, instances, verification, repair, uninstall, and paths.

This matters because without a setup core, Windows MSI, Linux packages, and macOS packages could all behave differently and corrupt install consistency.

The unresolved issue is that external packaging details require verification. WiX, macOS notarization, Linux packaging, and AppImage tooling are all external-world facts and may be stale.

### Topic 9 — Launcher architecture

The launcher was designed as a common core with views and actions, not as a hardcoded GUI. It should manage instances, packages, mods, tools, saves, settings, and previews. CLI, TUI, native GUI, and GFX GUI shells should all call the same launcher core.

The launcher exposes views such as instances, packages, mods, tools, saves, world preview, orbit map, etc. Actions include launch instance, create/delete/edit instance, list packages, enable/disable mods, open settings, and view world. Launcher extensions can add views, actions, and canvases.

### Domino and Dominium separation

**FACT:** The user accepted the idea of calling the reusable engine/core/platform layer **Domino**. Dominium remains the game/product layer.

This decision made sense because the user wants the engine to be reused for other games and projects. It affects everything: directory names, API prefixes, docs, product boundaries, and future SDKs. It should be considered final unless the user explicitly renames the project structure.

### ABIs/APIs first

**FACT:** The user set the major implementation plan as: first fully implement every ABI/API, second every platform, third every renderer, then setup, launcher, tools, and game.

This decision supersedes earlier plans. The rationale is that platforms, renderers, setup, launcher, tools, and game all need a stable interface surface. If the project starts with backends or products before defining ABIs, later refactors will be expensive.

This decision affects the Codex prompt roadmap. Future assistants should not jump directly into game systems unless the user explicitly requests it.

### Platform required, renderer optional for non-game products

**FACT:** The user wanted platform support required, but renderer support optional beyond software. Launcher/setup/tools should be able to use native UI, CLI, or TUI. Renderer should only be needed for game or custom canvas/preview features.

This decision makes products more portable. It also reduces dependencies for setup and launcher. It depends on `dom_ui`, product contracts, and capability descriptors.

### Unified GUI mode

**FACT:** The user asked whether GUI modes could be unified. The resulting design uses `DOM_UI_GUI` and separate backend flags such as `DOM_UI_BACKEND_NATIVE` and `DOM_UI_BACKEND_GFX`.

This avoids treating native and custom GUI as separate product types. It means a GUI launcher can use native widgets, custom renderer UI, or both. A future assistant should preserve this distinction.

### Product contracts and explicit fallback

**FACT:** The conversation concluded that graceful degradation is good only when declared. Products need contracts that say what is required and what is optional.

This decision prevents hidden failures. For example, a launcher can hide a preview canvas if no renderer is available, but a game cannot pretend to be playable without gfx. This affects setup, launcher, tools, and game bootstrapping.

### Renderer command-buffer IR

**FACT:** The renderer architecture uses a command-buffer IR. Backends consume commands such as clear, viewport, camera, pipeline, texture, draw sprites, draw meshes, draw lines, and draw text.

The rationale is backend independence. It allows software, GPU, and retro renderers to share semantics. It also supports logging, replay, remote rendering, and testing. The exact IR payload layout remains to be finalized.

### Numeric policy

**FACT:** The numeric palette was reduced to `Q4.12`, `Q16.16`, and `Q48.16`, plus ordinary integers. Inventory counts remain whole integers only. `Q4.12` is primarily for compact storage of bounded fields.

This decision came from comparing fixed-point alternatives. The rationale was memory efficiency, determinism, and avoiding type proliferation. It affects almost every simulation system.

### Signed world origin at center/sea level

**FACT:** The user wanted `0,0,0` to be the world center at sea level. This led to signed world coordinates and toroidal wrapping.

This decision affects coordinate math, chunk mapping, and world/space transitions.

### Cartesian space plus Keplerian orbits

**FACT:** Space should store authoritative positions in Cartesian coordinates, while using Keplerian on-rails orbit components for orbital motion. Polar/spherical coordinates are derived for UI or indexing.

This decision avoids polar-coordinate issues while supporting realistic-looking orbital mechanics. It rejects full N-body by default.

### Unified construction graph

**FACT:** Vehicles, buildings, machines, portable structures, and static/moving assemblies should be unified as constructions composed of materials and parts in graphs/frames.

This decision was driven by the user’s desire for arbitrary construction. It avoids separate building/vehicle/machine systems. It is a core future game architecture decision.

### Blueprints and jobs

**FACT:** Blueprints are plans or operations, not raw world copies. They compile to jobs. Manual player actions and queued worker/robot/AI actions should use the same job path.

This supports blueprinting, automation, replay, and AI.

### OS-native setup wrappers

**FACT:** The user wants setup binaries to match official OS setup styles. That means MSI/EXE on Windows, pkg/dmg/app bundle on macOS, deb/rpm/run/AppImage-like on Linux, and simpler TUI/console installers on retro systems.

The common install logic should still live in setup core.

### Tools and SDKs as first-class products

**FACT:** The user wants all useful dev tools and SDKs, not just a launcher/game. This includes mod and pack SDKs, game editor/creator, game data editor, save editor, world editor, launcher editor/creator, mod editor/creator, and packaging tools.

This affects Phase 6 and future ecosystem planning.

## 6. Ideas Considered but Rejected, Superseded, or Deprioritised

### Starting with platforms/renderers before ABIs

This was the earlier plan, but it was superseded. The current roadmap starts with ABIs/APIs because all backends and products need stable interfaces. Reconsider only if the repo already has a solid ABI layer.

### Renderer-mandatory launcher/setup/tools

This was rejected because launcher/setup/tools should be able to run natively, in CLI, or in TUI without needing the graphics stack. Renderer use remains optional for canvases/previews.

### Separate native-GUI and gfx-GUI modes

This was superseded by a unified `DOM_UI_GUI` mode with backend flags. Reconsider only if the UI model proves too limiting.

### Full fixed-point type matrix

The idea of making many signed/unsigned variants first-class was rejected as redundant. Keep ordinary integers plus `Q4.12`, `Q16.16`, and `Q48.16`.

### q48.16 everywhere

Rejected for dense fields because of memory overhead. It remains appropriate for sparse aggregate values.

### Universal milli-unit integers

Rejected as a general representation because it loses flexibility and still needs metadata. It may be useful for very specific counters.

### Polar/spherical space storage

Rejected for authoritative state due to angle wrap, trig, singularity, and non-linear resolution problems. Still useful for display and orbit bands.

### Full N-body global orbital simulation

Rejected as default because it is too expensive and unnecessary for the intended gameplay. Approximate/scripted effects may appear later.

### Prefab-only building system

Rejected because the user wants arbitrary construction. Prefabs remain useful templates, not the construction primitive.

### Giant Codex prompts

Rejected because Codex is more likely to tangle broad refactors. The current workflow is one bounded prompt per subsystem/backend/product stage.

## 7. Important Reasoning and Rationale

The visible reasoning in the chat consistently favoured **long-term modularity without uncontrolled abstraction**. The project needs to support many platforms, renderers, products, and gameplay systems. That requires clear seams: platform APIs, renderer APIs, product contracts, plugin ABIs, schemas, and events. However, the chat also explicitly warned against over-abstracting everything. The chosen approach is to modularize the seams that matter — backend vtables, command/query APIs, packages, events, views, and schemas — while keeping core simulation and UI primitives concrete enough to implement.

The renderer design was chosen because direct renderer calls would bind the game to backend-specific semantics. A command-buffer IR lets the same canvas or scene be rendered by software, OpenGL, DirectX, Vulkan, or retro framebuffer modes. The software reference renderer gives a baseline semantics.

The UI design was chosen because setup, launcher, tools, and game are not the same kind of product. The game needs graphics. The launcher and setup often do not. A native OS installer or native launcher should not carry unnecessary renderer dependencies. The `DOM_UI_GUI` + backend flags design keeps native and custom GUI unified while preserving flexibility.

The numeric design was shaped by determinism, memory efficiency, and gameplay clarity. Whole-integer inventory avoids fractional item problems. Fixed-point physical quantities allow deterministic simulation across platforms. The tiered storage approach prevents dense fields from consuming huge memory.

The construction design was shaped by the user’s ambition. If buildings, vehicles, machines, and spacecraft were separate primitives, arbitrary construction and destruction would be difficult. Treating all of them as construction graphs over parts and materials gives one model for static and moving structures.

The space design balances realism and computation. Keplerian on-rails orbits, Lagrange sites, belts, and graph routes provide realistic-feeling progression without full orbital physics everywhere. Local 6-DOF bubbles preserve piloting/docking feel where needed.

The implementation roadmap was reshaped by risk management. The user saw that implementing later prompts too early could bake in narrow assumptions. The new plan puts ABIs and APIs first.

## 16. Final Plain-English Summary

This chat was a major architecture-planning session for **Dominium** and its reusable engine layer, **Domino**. Its most important value is that it did not merely list features; it built a coherent implementation and abstraction strategy for a very large game ecosystem.

The conversation started from a practical build-order concern: develop platforms and renderers, then use them to implement the launcher, setup, and game. But over time the project grew in scope and clarity. The user wanted portable, per-user, and system-wide installs; versioned game code; base/DLC/mod content; packs for graphics, sound, and music; instance management; launcher customisation; and support across many platforms and renderers. This required a much deeper foundation than a simple game executable.

The reusable engine layer was named **Domino**. Domino is responsible for platform abstraction, rendering, audio, UI, input, core services, packages, instances, simulation control, canvas rendering, table/tree/view models, events, and plugin APIs. **Dominium** is the game/product layer built on Domino, containing setup, launcher, tools, game rules, content, and products.

The final implementation order changed during the chat. The earlier plan of doing platforms/renderers first was superseded. The current plan is: first implement every ABI/API; second implement every platform; third implement every renderer; fourth build setup; fifth build launcher; sixth build tools and SDKs; seventh build the game. This is one of the most important things to preserve.

A central product decision is that platform support is mandatory but renderer support is optional for launcher/setup/tools. A launcher should be able to run as CLI, TUI, native GUI, or GFX GUI. Renderer use is optional unless a custom canvas or preview is needed. The game client, however, requires graphics. UI is unified into `DOM_UI_NONE`, `DOM_UI_CLI`, `DOM_UI_TUI`, and `DOM_UI_GUI`, with backend flags for native and GFX implementations.

Rendering is handled through a command-buffer IR consumed by multiple renderer backends. Planned renderers include software reference, modern APIs such as DirectX, Vulkan, and OpenGL, Apple APIs such as QuickDraw/Quartz/Metal, Windows GDI, and retro modes such as VGA, CGA, EGA, VESA, Hercules, and MDA. Each backend should expose capabilities and degrade or fail explicitly. The software reference renderer is intended to define canonical behaviour.

The numeric and simulation architecture is deterministic. Gameplay economy and inventory use whole integer counts. Fixed-point types are used for physical and spatial values: `Q4.12`, `Q16.16`, and `Q48.16`. `Q4.12` is mainly a compact storage codec for bounded fields. `Q16.16` is the general runtime fixed-point type. `Q48.16` is used for large aggregate values like mass, volume, charge, and energy. The sim runs at arbitrary UPS independently of client FPS or vsync.

The world model is a sparse 3D grid over a `2^24 m` toroidal surface, with origin `0,0,0` at the world center and sea level. The user wants realistic planetary surfaces with climate, weather, hydrology, atmosphere, geology, biomes, and other systems, but the architecture should use shared primitives like fields, substances, mixtures, reservoirs, and networks rather than one-off arrays.

The construction model is also foundational. The user does not want buildings as prefabs. They want arbitrary construction: cut/fill terrain, mineshafts, open-cut mining, earthworks, sprawling factories, underground labs, skyscrapers, vehicles, ships, aircraft, spacecraft, stations, and portable structures. The design therefore unifies buildings, vehicles, machines, and structures as construction graphs made of parts and materials. Prefabs and blueprints are templates over these primitives. Blueprints are planned operations/diffs that compile into jobs. Manual player work and queued worker/robot/AI work should share the same job system.

Space was designed to be realistic-feeling but computationally cheap. Authoritative space positions are Cartesian. Orbits use Keplerian on-rails elements with optional drifts and precession. Lagrange sites, rings, belts, radiation, gas clouds, asteroids, comets, moons, planets, black holes, and wormholes can be represented through sites, fields, graphs, and special rules. Full N-body physics is not the default. Seamless surface-to-space transitions should be achieved with reference frames, sim tiers, transition bands, streaming, and dual representations.

The chat also planned setup, launcher, and tools. Setup has a shared core and OS-native wrappers: Windows MSI/EXE, macOS pkg/dmg/app, Linux deb/rpm/run/AppImage-like, and retro text installers. The launcher is a common core exposed through views, actions, tables, trees, and canvases; CLI/TUI/GUI shells all present the same model. Tools are first-class and include asset compiler, pack builder, replay inspector, test runner, world editor, save editor, game data editor, launcher editor, mod editor, pack tools, SDKs, and a GUI editor host.

Many Codex prompts were generated: Phase 1 ABIs/APIs, platform backend templates, renderer backend templates, setup prompts, launcher prompts, tools prompts, and a Phase 7+ roadmap. Actual repo implementation remains unverified. Before doing anything, the repo should be inspected. The most important next action is to verify the actual repository state and then proceed with Phase 1, beginning with the Domino ABI/API prompt.
