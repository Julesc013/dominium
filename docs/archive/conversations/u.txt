## 3. Main Topics We Discussed

### Topic 1 — Cross-platform launcher with native OS elements

The first topic was whether Dominium could have a launcher like older Minecraft launchers: fast, native-feeling, common across platforms, and not dependent on a heavy renderer.

**FACT:** The user wanted one common launcher codebase, native OS elements, and minimal rendering.

The key conclusion was that there are two separate goals hidden inside “one common launcher.” The **launcher logic** can be common across platforms. That includes discovery, validation, profiles, packs, manifests, version checks, launch plans, diagnostics, and audit logs. But native UI controls are not themselves cross-platform. A Win32 button, Cocoa button, GTK button, and classic Mac control are different things. Therefore, a native-looking launcher requires thin platform-specific UI adapters.

The future connection is straightforward: keep the launcher core common, but allow platform-specific native shells under `launcher/platform/...`. Do not put business logic in those shells.

### Topic 2 — Launcher as a control plane

The launcher was repeatedly framed as a **control plane**, not a store, package manager, DRM system, background service, or simulation component.

This means launcher code should mostly answer questions like:

* What is installed?
* What versions are compatible?
* Which profile is selected?
* Which packs are enabled?
* What command should be run?
* What environment should be passed?
* Why did a launch fail?
* What does the audit log say?

It should not install files, mutate installed payloads, decide game rules, repair silently, or hide errors.

This matters because a launcher can easily become an unsafe dumping ground. This chat repeatedly pushed it back into a narrow, inspectable orchestration role.

### Topic 3 — Setup and launcher separation

A central boundary was the separation between setup and launcher.

**FACT:** Setup is the only install mutation authority.

Setup installs, updates, verifies, repairs, rolls back, writes installed-state manifests, and owns file ownership. Launcher reads setup-produced manifests and may invoke setup through explicit contracts, but it does not perform setup’s work.

This boundary was discussed early in terms of launcher responsibilities, then hardened later by the canonical repo and applied prompts. It remains one of the most important constraints to preserve.

### Topic 4 — Common CLI/TUI/GUI semantics

The user clarified that CLI, TUI, and GUI core functionality should be implemented in one common core. The important nuance is that “core functionality” does not mean rendering. It means shared commands, intents, state transitions, output models, and workflows.

The final application canon later phrased this as: **CLI is canonical; GUI/TUI are views over the same command graph.**

This matters because it prevents platform-specific drift. If CLI has `verify`, GUI must not implement a different hidden version of `verify`. If GUI has a pack enable flow, it should map to the same command/intent as CLI and TUI.

### Topic 5 — UI as data, not logic

The later application canon established that UI must be defined via UI IR or a declarative layer. UI should not contain business logic.

This means UI files and widgets should describe controls, layouts, bindings, accessibility tags, localisation keys, and command connections. They should not decide profiles, mutate state, repair installs, choose packs, or infer simulation rules.

The future work implied by this is a command graph, UI IR schema, binding validator, and tests ensuring CLI/TUI/GUI parity.

The conversation moved from a standalone launcher tree to a product-first monorepo. This was a major evolution.

The final structure treats `engine`, `game`, `client`, `server`, `launcher`, `setup`, `tools`, `libs`, and `schema` as top-level products or shared foundations. Launcher is not under engine. Setup is not under launcher. Shared contracts are not hidden in game or engine internals.

This matters because filesystem layout becomes an enforcement tool. If code lives in the wrong product directory, it is a sign of boundary confusion.

The engine must not include launcher, setup, tools, or `dom_contracts`. It must export only engine public API under `engine/include/domino`. The chat established that cross-product contracts belong in a neutral layer, not in engine or game internals.

That layer is:

The launcher should use these channels for installed-state, product manifests, launch contracts, setup invocation contracts, build identity, and other cross-product information.

## 16. Final Plain-English Summary

This chat was about getting the Dominium Launcher into the right architectural place inside the larger Dominium / Domino project. It began with a practical question: could the post-Plan 8 system support a cross-platform launcher with one common codebase, native OS elements, and little or no renderer, similar to older Minecraft launchers? The answer evolved into a broader launcher/application-layer plan.

The first key insight was that “one common launcher” has to be split into two ideas. One common **functional core** is realistic and desirable. That core can handle installed-state discovery, manifest parsing, profiles, packs, version checks, launch planning, diagnostics, and audit logs. But native UI elements are platform-specific. Therefore, the launcher can have shared behavior while using thin platform-specific presentation shells for Win32, macOS, Linux, classic systems, and other targets.

The conversation then expanded into modularity, extensibility, mods, packs, CLI/TUI/GUI parity, setup separation, repository layout, build systems, and future documentation. Early ideas included a standalone launcher directory tree, manual Visual Studio/Xcode projects, and a possible DUI-style UI facade. Those ideas were useful stepping stones, but later user-provided canon superseded several of them. The final direction is product-first: launcher is a top-level product under `launcher/`, not part of `engine/`.

The canonical launcher structure became `launcher/core/discover`, `launcher/core/profile`, `launcher/core/invoke`, `launcher/ui`, `launcher/platform`, and `launcher/tests`. The core handles business/control-plane logic. The UI layer handles shared interaction semantics, command graph, view models, and UI IR bindings. The platform layer handles OS glue and native presentation only. It must not contain profile resolution, pack logic, install mutation, or hidden command semantics.

The most important boundary is engine purity. The engine must remain reusable and independent. It must not contain launcher, setup, tools, or application contracts. Its public headers belong under `engine/include/domino`. Cross-product contracts belong in `libs/contracts/include/dom_contracts`, and canonical data formats belong in `schema/`. This prevents shared APIs from polluting engine or game internals.

The second most important boundary is setup versus launcher. Setup is the only authority for installation, updating, verification, repair, rollback, and installed-state manifests. Launcher may display errors, validate state, and invoke setup through explicit contracts, but it must not install or repair files directly. This preserves auditability and prevents hidden mutation.

The build story also changed. Earlier advice about hand-authored Visual Studio/Xcode projects was superseded by applied Codex prompts that made CMake authoritative. Visual Studio 2026 should use CMakePresets/Open Folder, and generated VS projects derive from the CMake target graph. This matters because CMake target include/link rules enforce product boundaries.

The broader application-layer canon later locked the context: architecture is closed, canon is locked, and applications are orchestration shells. Applications must be content-agnostic, rule-agnostic, authority-agnostic, and must not mutate authoritative state or contain gameplay logic. CLI is canonical. GUI/TUI are views over the same command graph. UI is data via UI IR, not business logic. Accessibility and localisation are mandatory. RepoX/TestX/BUILD-ID-0 govern changelogs, validation, build identity, compatibility, and mismatch refusal.

Two major Codex prompts were pasted as already applied: one for CMake/VS2026/build repair and one for engine purity/contract ownership. This chat did not independently verify that those prompts succeeded. Therefore, the most important unresolved issue is current repo truth. That prompt should make the launcher code/docs explicit: architecture, boundaries, core module READMEs, interaction model, platform adapter docs, contract/schema usage, determinism, build workflow, and final status. It should not redesign APIs or add features.

The things to preserve from this chat are the final launcher boundaries, the product-first repo structure, setup-only install mutation, engine purity, neutral contracts, CMake authority, CLI command graph, UI IR requirement, accessibility/localisation rules, RepoX/TestX/BUILD-ID governance, and the warning that applied prompts remain unverified until the repo is checked.
