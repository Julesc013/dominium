## 7. Important Reasoning and Rationale

The visible reasoning throughout the chat was shaped by a few core concerns.

First, the project needs **authority integrity**. If a client, renderer, launcher, setup tool, or inspector can mutate simulation state outside engine/game authority, the project’s law, replay, multiplayer, and anti-cheat models become unreliable. This is why so much emphasis was placed on applications as shells, tools as auditable, and rendering as non-authoritative.

Second, the project needs **platform and renderer independence**. If renderers create windows, then each renderer needs OS-specific windowing code, and every backend becomes harder to port. If platform code owns windows and surfaces, renderers can stay focused on rendering. This also allows headless servers and software/CI paths to remain clean.

Third, the project needs **staged bring-up**. A real windowed executable is a concrete milestone, but trying to build every GPU backend at once would be fragile. The proposed safe order is: platform runtime, null path, software renderer, client loop, renderer selection, then GPU backends one at a time.

Fourth, the project needs **bounded compatibility**. The user wants broad support, including modern systems, old APIs, consoles, vendors, and vintage systems. But supporting everything equally is impossible. Capability tiers and support classes were proposed to manage the support surface honestly.

Fifth, the project needs **modularity without premature overengineering**. The user wants plug-and-play and broad binary distribution. Modules, manifests, dynamic plugins, framegraph, and render devices are plausible answers. But they depend on the actual codebase, build system, and platform constraints. They should not become final requirements until verified.

The main assumptions were:

* engine/game can be hosted through public APIs;
* platform and render can be separated cleanly;
* the existing repository can accommodate a runtime/module architecture;
* broad renderer/platform support is desirable but must be tiered;
* the old snapshot is available for verification.

Several of these assumptions remain unverified.

## 12. Risks, Failure Modes, and Things Future Chats Might Get Wrong

The biggest risk is **repo hallucination**. A future assistant might assume the prior claimed repository tree is real. To avoid this, inspect the actual old snapshot before any implementation prompt.

Another major risk is **treating assistant proposals as user decisions**. Platform runtime, capability tiers, module architecture, framegraph, and support classes are strong proposals, not final decisions. Future reports must preserve that status.

A third risk is **authority leakage**. Client, tools, runtime, or renderer code could accidentally affect authoritative simulation. Avoid this through dependency boundaries and clear APIs.

A fourth risk is **renderer/window coupling**. If renderers create windows, portability and headless support suffer. Avoid this by separating platform from render.

A fifth risk is **GPU-first complexity**. Starting with Vulkan/D3D/Metal before a software window path may slow progress. Avoid this by bringing up null/software first.

A sixth risk is **support matrix explosion**. Trying to support every OS, renderer, vendor, console, and vintage machine equally is not sustainable. Avoid this with support classes and capability tiers.

A seventh risk is **plugin ABI instability**. Dynamic modules may be attractive but fragile. Avoid this by designing versioned manifests and stable boundaries if dynamic modules are chosen.

An eighth risk is **overpromising console or vintage support**. Console SDKs and vintage systems have real constraints. Treat them as experimental or private-adapter topics until verified.

A ninth risk is **setup trust model weakness**. Without a clear integrity model, installation and content validation will be unreliable.

A tenth risk is **tool god-mode**. Tools must remain governed and auditable, or they undermine the project’s law model.

An eleventh risk is **engine/game API insufficiency**. If engine/game do not expose public host APIs, APP0 may be blocked or require expanded permissions.

A twelfth risk is **aggregation loss**. Future spec-book aggregation could flatten uncertainty or delete rejected options. Avoid this by preserving labels and caveats.
