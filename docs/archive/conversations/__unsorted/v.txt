## 5. Decisions Made and Why

### Quick decision overview

### Profiles instead of modes

FACT: The decision to avoid hardcoded modes was accepted repeatedly. The user and assistant discussed survival, hardcore, creative, observer, lab, spectator, editor, godmode, and sandbox. The conclusion was that code should not contain `survival_mode`, `creative_mode`, or similar toggles. Instead, the system should use profiles and law/parameter bundles.

This decision made sense because the user wanted modular, extensible behavior. Hardcoded modes would make future variations brittle. Profiles allow survival, hardcore, creative, and lab experiences to be composed rather than forked.

This decision affects future gameplay, UI, multiplayer, and tooling. It should only be revisited if the entire profile/law architecture fails, which is not currently established.

### Process-only mutation

FACT: Process-only mutation was treated as a core invariant. This means UI, session tools, camera movement, time control, survival actions, and future systems should not directly mutate state. They should emit intents that become validated Processes.

The rationale was deterministic replay. If mutation is scattered through arbitrary setters or UI callbacks, replay and audit become difficult or impossible. This decision affects every future feature and must be verified in code.

### Truth / Perceived / Render separation

FACT: The project adopted strict separation between the authoritative TruthModel, the PerceivedModel produced through observation/lens/law/authority filtering, and the RenderModel consumed by presentation code. Renderer must never access TruthModel directly.

This was decided to avoid epistemic leakage. It also supports multiplayer fairness, survival immersion, debugging/lab modes, and future spectator/observer differences.

### XStack removability

FACT: The user explicitly said they wanted XStack to be a portable agentic development suite that could be removed with no impact on code/data/runtime. This decision is foundational. XStack can validate the repo, but runtime must not depend on it.

The alternative — embedding XStack into game/engine runtime — was rejected because it would destroy portability and contaminate production code.

### FAST/STRICT/FULL verification

FACT: The user rejected verification that takes hours during local/agentic development. The solution was a stratified verification model: FAST for normal iteration, STRICT for structural checks, FULL/FULL_ALL for deeper or release-like checks. The visible rationale was that governance should behave like an incremental compiler, not a CI system.

This decision depends on cache correctness, sharding, and run-meta isolation.

### Run-meta isolation

FACT: The chat established that normal verification should not dirty tracked files. Snapshot mode is the only appropriate writer to canonical tracked reports. However, the later transcript reports committed `docs/audit` artifacts, creating a possible contradiction. This is not resolved and must be audited.

### Diegetic survival

FACT: The user explicitly said survival-style gameplay should avoid non-diegetic epistemic UI or HUD. The decision was to handle this through lens/law gating: survival defaults allow diegetic lenses only, while lab/debug/observer can use non-diegetic tools.

This affects future survival implementation. It is not yet a completed feature.

## 6. Ideas Considered but Rejected, Superseded, or Deprioritised

### Hardcoded modes

This idea was rejected because it would create brittle code branches. It was considered only because normal games often use “Survival Mode,” “Creative Mode,” and similar labels. The final architecture allows those labels for users but not as code logic.

This rejection is final.

### Full verification on every prompt

This was rejected after the user experienced huge delays. Full checks were too expensive and made autonomous development impractical. The replacement was FAST/STRICT/FULL with caching and sharding.

This rejection is final for local agentic iteration, though full checks remain appropriate for release or explicit full-all validation.

### Runtime dependence on XStack

This was rejected because the user wants XStack to be portable and removable. XStack should validate and orchestrate development, but engine/game/client/server should not need it to run.

This rejection is final unless the user fundamentally changes the product/tool boundary.

### Dumping the Constitution into README

This was rejected because README should be accessible. Deep doctrine belongs in canon/architecture docs. This is final for documentation UX.

### Live-switch law profiles mid-session early

This was deprioritised, not permanently rejected. The concern was deterministic complexity. The preferred early approach is a session restart boundary for changing experiences/laws.

It could become relevant later once transition contracts mature.

### Global full micro-simulation

This was rejected because it is computationally impossible at galaxy/trillion-agent scale. The alternative is macro capsules, interest regions, solver tiers, and deterministic activation.

This rejection is final as a global default.

### Rewriting gate in a faster language

This was considered when Python seemed slow. It was rejected/deprioritised because the real bottleneck was algorithmic: full scans, excessive work, poor caching, and RepoX scope explosion. Rewriting in Rust or C++ would not fix that without architecture changes.

It could be reconsidered only if a true CPU-bound hotspot remains after correct incremental design.

### Manually rewriting all old prompts

This was rejected because it would be too much work and would not scale. The chosen strategy was to use prefix/control/reconciliation approaches to absorb bad prompts.

## 16. Final Plain-English Summary

This chat was a large continuity and handoff conversation for the Dominium / Domino project. Dominium is being developed as a deterministic universe simulation/game platform, with Domino as the deterministic engine layer and Dominium as the game/application layer. The architecture repeatedly emphasized process-only mutation, deterministic replay, named RNG streams, fixed-point authoritative state, no omniscience, and strict separation between TruthModel, PerceivedModel, and RenderModel. These principles matter because the user wants the project to support extremely ambitious future gameplay and simulation domains without collapsing into hardcoded spaghetti.

A major reason this chat exists is that prior autonomous development attempts were failing operationally. The user described repeatedly queueing prompts for Codex or other agents, only to have them stop at gate failures, missing tools, stale artifacts, or small mechanical blockers. This was unacceptable because the user wanted to wake up to real progress, not excuses. The visible conversation therefore developed a strong doctrine: prompts are untrusted input, mechanical blockers should be remediated automatically, and only true semantic ambiguity should require escalation. The user does not want future assistants to ask whether to fix obvious issues.

This doctrine led to XStack, the portable agentic development/governance suite. XStack includes systems like RepoX, TestX, AuditX, ControlX, PerformX, CompatX, and SecureX. The chat spent a lot of time hardening XStack: making verification fast, incremental, cached, sharded, and deterministic; separating FAST, STRICT, and FULL profiles; preventing run-meta from dirtying tracked files; adding snapshot mode; ensuring XStack can be removed without breaking runtime; and avoiding governance creep. The key conclusion was that XStack should serve development, not dominate it. It should behave like a compiler, not like a CI system that reruns everything all the time.

The chat also established important architectural decisions for the game itself. There should be no hardcoded mode flags. Survival, Hardcore, Creative, Observer, Lab, Spectator, Editor, and similar experiences should be represented by profiles, law profiles, parameter bundles, lenses, entitlements, and packs — not branches like `survival_mode`. All state changes should go through Processes. UI and rendering should never read TruthModel directly; they should consume PerceivedModel and RenderModel. Packs should be data-only, compiled into registries and lockfiles before runtime. Runtime should not dynamically merge raw packs.

Another major thread was documentation and glossary work. The user wanted a normative glossary, not just a descriptive list of terms. Terms like Assembly, Field, Process, Agent, Law, Determinism, Lens, Observation Kernel, Refusal Contract, Macro Capsule, Fidelity, Domain, Solver, Artifact Class, FAST, STRICT, FULL, RepoX, TestX, and AuditX were treated as architectural terms. Deprecated terms such as `survival_mode`, `creative_mode`, `debug_mode`, `godmode`, and `sandbox` were identified as dangerous because they encourage hardcoded behavior. The assistant also argued that README should be an accessible front door, while deep doctrine should live in canon and architecture docs.
