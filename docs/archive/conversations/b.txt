## 1. One-Page Orientation

This chat was mainly about turning the Dominium + Domino project from a collection of partially overlapping architecture ideas into a coherent, refactorable, future-proof structure that could be implemented by Codex and then preserved for future ChatGPT conversations.

At the highest level, the discussion separated two names and responsibilities. **Domino** is the engine. It lives under `source/domino`, provides deterministic simulation, platform abstraction, graphics rendering, input, audio, UI foundations, mod/package systems, and cross-platform support. **Dominium** is the product suite built on top of Domino. It lives under `source/dominium` and contains the user-facing products: Game, Launcher, Setup, and Tools.

The chat began as a presentation-layer architecture thread: graphics, UI, UX, asset packs, sound packs, music packs, vector/raster rendering, theming, and how everything should sit on top of existing `dsys` and `dgfx` layers. That early discussion established a strong rule: no product UI or rendering path should bypass Domino’s platform and graphics abstraction layers. UI should eventually become a shared toolkit usable by the game, launcher, setup, and tools, with a vector-only fallback so the system remains functional even with no raster packs installed.

The conversation then moved into products and binaries. At first, there was discussion of whether headless, CLI, TUI, and GUI modes could be merged into one application. That led to a more general product architecture. The user then clarified that they wanted separate products: Game, Launcher, Setup, and Tools. From there, the architecture settled into four product binaries, each using shared Domino systems, rather than one monolithic executable.

A major decision was that **Game is one product**, not separate client/server/demo products. It should launch in different modes through flags: client, listen server, dedicated headless server, and demo/full. The assistant recommended one game binary with flags because it avoids client/server version skew, duplicate product metadata, and divergent runtime paths. The user accepted this direction by continuing the architecture around it. Demo was treated similarly: not a separate `/demo` product, but content and instance policy controlled by a demo base pack and `instance.json` flags.

Another major thread was installation, distribution, and long-term version management. The user did not want every instance to store a full suite copy. The resulting model was `DOMINIUM_HOME`: a shared repository of product builds, mods, and packs, plus lightweight instances that reference exact versions. Versioning was another central theme. The user wanted separate version numbers for core, launcher, game, setup, tools, and protocols. ## 3. Main Topics We Discussed

### Topic 1 — Domino vs Dominium

The first core distinction is between Domino and Dominium.

**FACT:** Domino is the engine. It belongs under `source/domino` and `include/domino`. It includes the simulation, platform abstraction, graphics IR/backends, audio, input, mod/package systems, and UI foundations.

**FACT:** Dominium is the product suite built on Domino. It belongs under `source/dominium` and contains products such as Game, Launcher, Setup, and Tools.

This distinction mattered because many later decisions depended on it. If Domino were treated as a separate installed runtime, the project would risk version conflicts. If Domino were renamed to `engine`, the source tree would lose clarity and mismatch existing public include paths. If Dominium product logic stayed mixed into engine code, the architecture would become harder to maintain.

The conclusion was that Domino remains a named engine in source and may later have a developer SDK, but it should not be a global runtime product installed beside Dominium for end users.

What remains uncertain is the exact degree to which existing code currently respects that boundary. The user provided a repo tree, but this chat did not inspect or modify code directly.

**FACT:** The user wanted a separate set of binaries per product.

Each product should use shared Domino systems. They should not contain their own platform/rendering implementations. The Game product handles runtime play/server modes. Launcher is the suite hub. Setup installs/imports/repairs/uninstalls. Tools aggregates modcheck and future tools.

This product model prevents both extremes: it avoids a monolithic everything-binary, but also avoids uncontrolled proliferation of tiny products for client, server, demo, individual editors, etc.

### Topic 7 — Versioning and Compatibility

Versioning became one of the most important topics.

The final model has:

* Domino core version.
* Game version.
* Launcher version.
* Setup version.
* Tools version.
* Suite version equal to Game version.
* Separate format versions.
* Separate protocol versions.

This matters for long-term compatibility. A world created with Game 2.5 and certain mods can keep referencing those versions even after Game 2.6 is installed. Garbage collection can later remove unreferenced entries safely.

### Decision: keep Domino as `source/domino`

The user asked whether the repo should have `source/Domino` and `source/dominium` or `source/engine` for Domino. The recommendation was to use `source/domino` and `source/dominium`.

This was decided because Domino is a named engine with public headers under `include/domino`. Calling it `engine` would create ambiguity and require documentation to repeatedly explain that “engine” means Domino. It also preserves a clean mapping between include and source trees.

This affects all future source layout and documentation. It should only be revisited if the engine itself is renamed.

### Decision: do not install Domino as a separate runtime

The user asked whether Domino should be bundled with Dominium or installed separately. The recommendation was to bundle/static-link Domino into Dominium products, not install a separate global runtime.

The rationale was version safety. If multiple suites install different Domino core versions, a shared runtime can create DLL/ABI/version conflicts. Since product builds already declare which core version they were built against, the safest model is for each product build to contain or internally link its exact Domino core.

A separate Domino SDK for developers remains possible. That is different from a player-facing runtime.

### Decision: flatten `source/dominium`

The user explicitly chose:

```text
source/dominium/
  common/
  game/
  launcher/
  tools/
  setup/
```

This superseded earlier structures involving `source/dominium/products`. It is one of the clearest user decisions in the chat.

The implication is large: CMake, includes, docs, tests, and build artifacts all need updating.

### Decision: four Dominium products

The user wanted separate product binaries rather than one combined app. The settled products are Game, Launcher, Setup, and Tools.

This balances separation and control. It avoids one monolithic binary while also avoiding excessive product proliferation.

### Decision: Game client/server/demo are modes, not products

The user asked whether Game should have a client and server in separate binaries or startup flags. The recommendation was startup flags in one Game product.

This decision made sense because the simulation path should remain unified. Separate client/server products would increase version compatibility burden. A dedicated server can still run headless. A wrapper can exist later, but not as a separate architecture product.

Demo follows the same principle: it is content/instance policy, not a separate binary.

### Decision: Launcher is the main access point

The user explicitly stated that the Launcher should be the main point of access to the whole suite. This turned Launcher into the suite hub.

The Launcher should not be a thin “start game” button. It should manage instances, product versions, tools, mods, packs, diagnostics, and actions.

### Decision: actions-based launching

The actions system was introduced to avoid hardcoded launcher behavior. Products expose `actions.json`; Launcher discovers and executes actions.

This is extensible. A new tool or game mode can appear as an action without editing Launcher logic.

### Decision: DOMINIUM_HOME repo/instances

The user wanted many product versions and many instances without storing the whole suite repeatedly. The repo/instance model solves that.

## 6. Ideas Considered but Rejected, Superseded, or Deprioritised

### One monolithic binary

Early discussion considered unifying headless, CLI, TUI, and GUI modes into one application. That idea was not wrong, but it was superseded when the user asked for separate product binaries. The final model keeps mode selection within each product, especially Game, but not a single binary for all products.

### Separate client/server products

Separate client and server binaries were considered because the Game has client and server roles. This was rejected as the default because it increases build and compatibility complexity. It may be reconsidered for a minimal dedicated-server distribution, but only as a wrapper or specialized build, not as the default architecture.

### Separate demo product

A separate demo product was rejected. It would duplicate Game behavior and complicate saves, mods, and compatibility. Demo should be content/profile gating through a demo base pack and instance flag.

### Global Domino runtime install

Installing Domino separately from Dominium was rejected. The visible reason was avoiding shared-runtime version conflicts. A future developer SDK is acceptable, but not a global player-facing runtime.

### `source/engine`

Using `source/engine` instead of `source/domino` was rejected because it hides the named engine identity and conflicts with `include/domino`.

### Strict base version equality

Hardwiring base mod version equal to Game version inside engine logic was rejected. It should be a release convention, not a loader invariant.

### Ambiguous `x64`

Using `x64` was rejected because the user disliked its ambiguity. Explicit tags should be used.

### Full UI/packs rewrite in the first refactor

## 7. Important Reasoning and Rationale

The strongest visible reasoning pattern in the chat was: avoid architecture that creates hidden coupling or version skew.

This explains the rejection of a global Domino runtime. It also explains the rejection of separate client/server/demo products. If the project is intended to support old saves, old mods, old instances, and many platforms, then runtime behavior must be traceable to exact product builds and content versions. Static/internal linking and build metadata make that possible.

Another recurring rationale was extensibility through data/metadata instead of hardcoding. The actions system is a clear example. Rather than teaching Launcher how to call every tool or game mode, product builds describe their actions. The repo/instance model is another example: instances are data that reference versions, not copied directories.

The user also cared about future-proofing. Many decisions anticipate long-term project growth: multiple OS families, many architectures, many versions, many tools, many mods/packs, multiple UI styles, future aggregation of old chat reports, and future Codex implementation. That is why the assistant repeatedly recommended explicit schemas, version metadata, compatibility ranges, and source boundaries.

A major tradeoff was between simplicity and flexibility. Matching base mod version to Game version is simple for users, but hardwiring equality is inflexible. The final compromise preserves the public mental model while allowing compatibility-based exceptions.

Another tradeoff was between immediate implementation and future architecture. The UI/packs architecture was detailed, but later refactor prompts were deliberately more focused on source/product/versioning. The visible rationale was that implementing the entire UI/packs system during the same source refactor could make the Codex change too broad and risky.

The user seemed to care most about not losing context, not repeating old decisions, making Codex prompts concrete, and keeping the architecture modular enough to evolve across many platforms and years.

## 16. Final Plain-English Summary

This chat developed the architecture and implementation plan for reorganizing the Dominium + Domino project into a more modular, extensible, and robust long-term structure.

The central distinction is that **Domino is the engine** and **Dominium is the product suite**. Domino lives under `source/domino` and provides the deterministic simulation, platform abstraction, graphics IR/backends, audio, input, UI foundations, and mod/package systems. Dominium lives under `source/dominium` and contains the user-facing products built on Domino. The user explicitly rejected a generic `source/engine` directory and explicitly chose a final Dominium layout without a `products` subdirectory.

The final source layout decision is one of the most important outcomes. Dominium should contain `common`, `game`, `launcher`, `tools`, and `setup` directly under `source/dominium`. Existing `source/dominium/products/common`, `game`, `launcher`, and `setup` should be moved into that flattened structure. Existing `source/dominium/rules` should be moved under game as `source/dominium/game/rules` according to the generated refactor plan. Existing `tools/dominium_modcheck` should become part of the DominiumTools product, likely under `source/dominium/tools/modcheck`.

The product model settled on four products: Game, Launcher, Setup, and Tools. The user wanted separate product binaries, not one monolithic binary. However, Game itself should be one product with several modes. Client, listen server, dedicated server, and demo/full should be handled through startup flags and instance/content configuration, not separate products. The key flags discussed were `--mode=gui|tui|headless`, `--server=off|listen|dedicated`, `--instance=<id>`, and `--demo`. A thin server wrapper may exist later, but the architecture should not treat server as a separate product by default. Demo is likewise not a separate product; it should be represented by a demo base pack and `instance.json` flags.

Launcher became the main suite entry point. It should not merely start the game. It should manage product builds, instances, mods, packs, tools, diagnostics, and actions. The actions system was introduced so Launcher can discover what products and tools can do without hardcoding executable paths. Product builds should expose `actions.json`, and Launcher should enumerate actions such as “run client,” “run dedicated server,” “modcheck,” “world editor,” or “pack editor.”

Installation and storage were another major focus. The user did not want every instance to store a full suite. The answer was the `DOMINIUM_HOME` model: a shared repository of product builds, mods, and packs, plus lightweight instances that reference exact versions. The repo stores product builds under paths keyed by product version, core version, OS family, and architecture. Mods and packs are stored by ID and version. Instances have `instance.json`, config, saves, and logs. Setup imports suite contents into the repo non-destructively and should not overwrite existing installed versions. Garbage collection can later remove unused repo entries after scanning all instances.

Versioning was carefully separated. Domino has a core version. Game, Launcher, Setup, and Tools each have independent product versions. Suite version equals Game version, as the user explicitly requested. Formats and protocols have separate version numbers and compatibility ranges. Products expose `DomProductInfo` through `--introspect-json`, using `DomCompatProfile` to describe supported core, save, pack, replay, network, launcher-game, and tools-game compatibility. Compatibility should degrade gracefully: full, limited, read-only, or incompatible/refuse. The goal is not “everything always works,” but rather “never crash or corrupt; refuse safely when needed.”

The official base modpack should usually match the Game/Suite version for user clarity. But this should be a release convention, not a hard engine invariant. The user asked whether making that link arbitrary was wise, and the conclusion was that strict equality would create problems for engine-only or data-only patches. DLCs, user mods, official packs, and user packs should have independent versions and compatibility ranges.

The chat also covered platform and architecture naming. The user disliked `x64`, so the architecture uses explicit tags such as `x86-64`, `x86-32`, `arm-64`, `wasm-32`, and `z80-8`. Package names should follow `Dominium-Suite-<GAME_VERSION>-<OSFam>-<Arch>.<ext>`.

Early in the chat, a large UI/graphics/packs architecture was defined. It remains important but is mostly later-phase. The future UI stack should be shared by Game, Launcher, Setup, and Tools; should render through Domino’s dgfx/canvas; should avoid OS-native drawing; should support deterministic layout; and should always have a vector-only fallback even when no raster packs are installed.

A concrete existing repo tree was pasted by the user. Based on that tree, several Codex prompts were generated: a master refactor prompt, a post-refactor consistency prompt, a prompt for Codex to generate a future starter prompt, and an extended starter prompt generator. Later, a maximum-fidelity context transfer packet and downloadable report package were produced. These are artifacts for continuity, but they do not prove the repo was changed.

The most important unresolved issue is whether Codex has applied any of the prompts. Future work should begin by verifying current repo state. If the refactor has not been applied, the next best step is to run or phase the master refactor prompt. If it has been applied, the next step is a consistency pass: search stale paths, verify CMake, run tests, check product introspection, and update docs.

The next assistant should preserve the architecture exactly: `source/domino`, flattened `source/dominium`, four products, one Game with modes, Launcher as hub, DOMINIUM_HOME repo/instances, action-driven execution, Suite version equals Game version, no global Domino runtime, no separate demo product, no ambiguous architecture names, and no assumption that generated prompts have already been executed.
