## 7. Important Reasoning and Rationale

The central visible reasoning was that broad support requires **classification**. Without tiers, “support as many devices as possible” becomes impossible to implement or test. A real support claim must say what product runs, on what hardware, through what runtime, with what feature set, under what QA standard, and with what distribution or legal constraints.

Another major rationale was that **top-tier targets must shape architecture early**. Since Android, iOS, and Web are primary, the product cannot be designed only around desktop PC assumptions. Mobile requires touch, lifecycle handling, low-memory resilience, variable device performance, and store constraints. Web requires browser sandboxing, WASM limitations, graphics fallback choices, storage decisions, and asset-size control.

The console rationale was about gatekeeping and legal/process risk. Current consoles are not just different hardware; they are closed ecosystems with approval processes, SDKs, certification, and NDAs. Legacy consoles add extra risk because of obsolete tools and firmware/BIOS issues.

The Web rationale had two sides. Web is a broad reach platform because modern browsers exist on many devices. But after the user’s decision, Web is not merely broad fallback; it is a primary target. That tension matters: Web must be designed as serious product support while still acknowledging its constraints.

The legacy rationale was scope control. Very old systems can be intellectually interesting and may support reduced demos, but they should not dictate the main product unless intentionally elevated. This preserves the portability ambition without letting it derail the core platform plan.

The reporting rationale was continuity. The user wanted future chats to avoid re-explanation, avoid lost context, and eventually feed a master Project Spec Book. That is why the chat moved into handoff packets, normalized reports, and aggregation-ready artifacts.

## 8. Plans, Future Work, and Next Steps

### First next step: create a Platform Support Matrix

The most important future action is to turn the current strategy into a matrix. This matrix should start with the Tier-0 targets: PC, Android, iOS/iPadOS, and Web.

It should define, for each platform family, the support tier, product scope, minimum OS/API, CPU architecture, graphics backend, input model, audio model, storage/save model, networking, distribution channel, QA obligations, legal or certification constraints, dependencies, risks, verification status, and next action.

This matters because it converts broad ambition into testable commitments. Without this matrix, future conversations can keep repeating the same uncertainty.

### Second next step: define support categories

Before filling out the matrix, Dominium needs a clear meaning of “support.” Suggested categories are:

* full product / full parity;
* full product with constrained features;
* reduced product;
* engine/core-only;
* research/demo;
* emulator-hosted compatibility;
* remote-play only;
* community port;
* unsupported.

These categories are inferred from the conversation, not finalized by the user. They are necessary because the same word “support” cannot mean the same thing for Windows 11, iPhone, WebAssembly, PS5, PS1, CP/M, and Apple Watch.

### Third next step: define Tier-0 platform contracts

Each Tier-0 platform needs a contract.

For PC, this means deciding Windows/macOS/Linux baselines.

For Android, this means API level, ABI, Vulkan/GLES, phones/tablets/TV/Automotive/Go/Chromebook/vendor ROM policy.

For iOS/iPadOS, this means minimum OS, device generations, iPad support, App Store implications, Apple Silicon/Intel Mac split, and Apple-family classification.

For Web, this means browser versions, WASM features, WebGPU/WebGL policy, PWA/offline rules, storage, asset loading, and performance expectations.

### Fourth next step: decide engine/toolchain

The chat did not decide whether Dominium uses a custom engine, Unity, Unreal, SDL/GLFW, or another stack. That decision is critical because platform feasibility depends on it. Until the engine/toolchain is known, renderer planning, mobile support, Web export, console viability, and XR support remain speculative.

### Fifth next step: verify external facts

Many facts in this chat are time-sensitive. The project must verify current official information about Android, Apple, browsers, WebGPU, WebAssembly, Steam Deck, console programs, XR runtimes, and graphics APIs.

This should happen before implementation planning, not after.

### Sixth next step: classify secondary targets

After Tier-0 is clear, the project should classify PC handhelds, current consoles, legacy consoles, XR devices, Apple TV/watchOS/visionOS, Android TV/Automotive, and emulator/legacy targets.

This should be done carefully so that broad ambition is preserved without false commitments.

## 12. Risks, Failure Modes, and Things Future Chats Might Get Wrong

The most serious risk is that a future assistant treats assistant suggestions as user decisions. The user explicitly decided PC/Android/iOS/Web priority. The assistant inferred many other details. Those must remain separate.

Another risk is overclaiming platform support. The user listed many devices, but the project has not promised full support for all of them. A future assistant must not say Dominium supports every PlayStation, Xbox, Nintendo, Android variant, Apple device, browser, headset, and retro OS without tier definitions.

A third risk is stale external facts. Platform rules change. Hardware status changes. Browser support changes. SDK and store requirements change. Anything current-world-facing must be verified from current official sources.

Another major risk is under-designing mobile and Web. If Android, iOS, and Web are top-tier, then UI, input, renderer, storage, assets, memory, and lifecycle must be designed for them early.

There is also a risk of scope explosion. The platform list is enormous. Without prioritisation, it can consume the entire project. The matrix/contract approach is the mitigation.

Legacy and console risks are also significant. Firmware/BIOS redistribution, unofficial tooling, and closed-console certification can create legal or practical blockers. Future work must separate official commercial support from research or compatibility paths.

Finally, there is a risk of bad aggregation. When this chat is combined with other chats, similar topics might be merged incorrectly. For example, another chat may define engine architecture, release strategy, or platform priority differently. The aggregator should preserve provenance and contradictions.

## 16. Final Plain-English Summary

This chat was about deciding how broadly Dominium should run across operating systems, devices, and runtime environments. It began with a broad question about “what systems” the game would support. The first answer went very wide, including retro and bare-metal systems such as Z80-class computers, CP/M, MS-DOS, Windows 3.x, Classic Mac OS, AmigaOS, OS/2, and similar targets. That early direction was useful as a portability brainstorm, but it was not the final plan.

The user then narrowed the question to what operating systems Dominium could run on. The assistant introduced a distinction between the full Dominium product and a possible reduced engine/core layer, using the name “Domino” for the latter. The distinction is useful because it separates full product support from constrained technical demos, but the name “Domino” was never confirmed by the user and should not be treated as official.

The conversation then expanded into a much larger platform inventory. The user wanted to support as many devices as possible across PlayStation, Xbox, Nintendo, PC handhelds, Android, iOS, Web, AR, VR, and related runtimes and APIs. The user provided a detailed list covering current and legacy consoles, handhelds, Android device classes, Apple devices and operating systems, WebAssembly and browser technologies, AR/VR/XR devices, OpenXR/WebXR, and graphics/runtime technologies such as Vulkan, OpenGL, DirectX, Metal, SDL, GLFW, Unity, Unreal, Wine, and Proton.

The most important decision came when the user explicitly said: **Android, iOS, and Web will be the primary focus after PC and should be primary top-tier support likewise.** That is the controlling outcome of the chat. It means PC is the first primary platform family, and Android, iOS, and Web are also primary first-class targets. They are not secondary ports. Future planning must treat mobile and Web constraints as architectural requirements from the start.

The assistant then synthesized a working support hierarchy: Tier-0 is PC, Android, iOS/iPadOS, and Web; Tier-1 is current commercial consoles such as PlayStation, Xbox, and Nintendo systems, gated by official platform access and certification; Tier-2 is legacy consoles, obsolete handhelds, retro operating systems, bare-metal targets, emulator-hosted paths, research builds, or engine-only experiments. This tier model is useful, but only the PC/Android/iOS/Web priority was explicitly decided by the user. The rest remains assistant-inferred and should be confirmed before becoming formal policy.

The chat also established that “support” must be defined carefully. Supporting Windows, iPhone, WebAssembly, Steam Deck, PS5, PS1, Android Automotive, Apple Watch, and CP/M cannot mean the same thing. Future work needs explicit support categories such as full product support, constrained full product, reduced build, engine/core-only, emulator-hosted, remote-play only, research/community port, and unsupported. This prevents the project from making vague or impossible claims.

The main next action is to create a **Platform Support Matrix**. It should list every relevant platform family and define its tier, support scope, minimum OS/API, CPU architecture, graphics backend, input model, audio, storage/save model, networking, distribution path, QA obligation, legal or certification constraints, dependencies, risks, verification status, and next action. After that, the project should define formal Tier-0 platform contracts for PC, Android, iOS/iPadOS, and Web.

Many things remain unresolved. The chat did not decide the engine or toolchain. It did not choose Unity, Unreal, custom engine, SDL, GLFW, or any other stack. It did not decide Android API levels, iOS versions, supported browser versions, WebGPU/WebGL policy, PWA requirements, distribution channels, PC OS baselines, Steam Deck tier, console generation priorities, XR priority, or legacy support commitments. It also did not verify current platform rules, SDKs, store policies, browser capabilities, hardware status, or console developer requirements.

The biggest risks are overclaiming support, treating assistant suggestions as user decisions, relying on stale platform facts, under-designing mobile/Web, letting legacy targets dominate the project, and merging this chat incorrectly with future reports. The best protection is to preserve the explicit user decision, label uncertainty, verify current facts, and convert the discussion into a formal platform matrix before implementation planning.

The later part of the chat created a Context Transfer Packet and then a downloadable report package for future aggregation. Those artifacts preserve the state of this conversation, but the substantive value of the chat is the platform-support direction: **Dominium should be planned as a PC + Android + iOS + Web first-class product, with consoles, handheld variants, XR, and legacy systems classified through explicit support tiers rather than vague promises.**
