# Reader Brief — Dominium Timekeeping and 2038 Resilience

## What this chat was about

This chat developed a durable timekeeping framework, starting from Y2038 and ending with Dominium-specific architecture guidance. The key result is that Dominium should preserve a strict split: ACT for authoritative logical simulation, DSYS monotonic time for runtime/pacing, observer clocks for derived perception/replay, and civil/astronomical time as projection only.

## Top 20 things to know

1. Y2038 is about signed 32-bit time representations, not all 32-bit machines.
2. Some 32-bit apps will continue; vulnerable time functions may fail.
3. Clock rollback is not a general solution.
4. Unsigned 32-bit is not a future-proof absolute timestamp fix.
5. Signed 64-bit is the right baseline for long-lived project time.
6. One signed 64-bit nanosecond scalar is too narrow for ±10,000 years.
7. 100 ns and 1 µs scalar ticks are practical for that range.
8. Split `int64 seconds + int32 nanos` is cleaner when possible.
9. Durable software needs separate Instant, Duration, Monotonic, and Civil types.
10. Monotonic time should be used for intervals and pacing, not authority.
11. Civil/local time must not be treated as a raw offset only.
12. Dominium already documents simulation time as logical, not wall-clock.
13. `dom_act_time_t` is signed 64-bit ACT seconds.
14. DSYS owns platform monotonic microsecond timing.
15. Cross-shard ordering uses logical ticks and deterministic keys.
16. Observer clocks are derived and non-authoritative.
17. ACT→BST/GCT/CPT conversions are currently placeholders/projections.
18. Main work is hardening boundaries, not rewriting the model.
19. External platform facts must be verified before formal spec use.
20. This package should feed the future Dominium master spec book.

## Decisions

The most important repo-supported decision is that Dominium authority must remain logical ACT, not wall-clock. The strongest recommendations are to freeze ACT units/serialization, audit DSYS timers, and keep civil time as projection-only.
