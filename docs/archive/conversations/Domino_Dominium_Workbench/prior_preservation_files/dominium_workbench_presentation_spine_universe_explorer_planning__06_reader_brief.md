# Reader Brief — Dominium Workbench, Presentation Spine, and Universe Explorer Planning

This chat is a major planning pivot. It began with a Windows UI Editor/Tool Editor design and ended with a broader Dominium Workbench Platform architecture. The old editor plan is superseded as a final product. Its useful pieces should be recycled into Workbench modules.

Top things to know:
1. Workbench is not authority; it is a projection over contracts/services/commands/documents/evidence.
2. CLI, TUI, rendered GUI, native GUI, and headless should share one semantic spine.
3. TUI is first-class.
4. Rendered GUI is built on software renderer first; hardware later.
5. Native SDK GUIs remain optional projections.
6. UI is data-described through views, layouts, controls, widgets, themes, style tokens, and projections.
7. Visual edits should produce document patches and evidence.
8. Components, services, providers, packs, modules, workspaces, apps, and artifacts are distinct.
9. Workbench modules are built from smaller reusable services/views/documents.
10. Progressive self-hosting is required.
11. The immediate likely slice is COMMAND-RESULT-VIEW-SLICE-01, subject to repo verification.
12. Validation Dashboard is the best first Workbench module.
13. Project Graph Explorer and Agent Work Board are early high-value modules.
14. Interface Studio and Theme Lab come after projection/runtime proof.
15. Universe Explorer should be lawful inspection/materialization, not renderer free camera.
16. Robot seed-civilization systems can be built as runtime artifacts and Workbench labs.
17. Code-only primitive themes can support useful UI; richer packs/assets are optional.
18. OEM+ themes are mimic profiles, not copied OS assets.
19. Current repo facts in pasted contexts need verification.
20. The future spec book should treat this chat as a core Workbench/presentation architecture source.

Best next step: verify current repo state, then draft COMMAND-RESULT-VIEW-SLICE-01 if gates allow.
