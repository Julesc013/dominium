# CONTEXT TRANSFER PACKET — Dominium Workbench, Presentation Spine, and Universe Explorer Planning

## 29.1 Ultra-Condensed Bootstrap Brief
This chat evolved from an old Windows UI Editor / Tool Editor planning session into a much larger architectural direction: Dominium Workbench Platform. The old editor plan was superseded as a final product. Its useful parts — visual UI authoring, TLV/UI docs, CLI scripts, import/export, validation, preview, property inspectors, hierarchy trees, codegen, and launch/setup layout tests — should be recycled into Workbench modules, especially Interface Studio, Theme Laboratory, and UI/HUD Sandbox.

The central decision is that Workbench is not authority. The true center is contracts, commands, services, documents, patches, providers, packs, modules, apps, artifacts, diagnostics, evidence, and tests. Workbench is the richest human/agent projection over that system. CLI, TUI, rendered GUI, OS-native GUI, and headless modes should all project the same command/result/refusal/document/view spine. Apps such as client, server, launcher, setup, and Workbench should compose shared runtime services and presentation contracts rather than owning product behavior directly.

The user emphasized portability, modularity, extensibility, replaceability, and reuse for other Domino-based games and even other engine projects. Important vocabulary: component = source/build ownership unit; service = callable runtime capability; provider = replaceable implementation; pack = distributable authored payload; module = functional extension unit; workspace = Workbench composition; app = shipped product composition; artifact = persisted versioned output/input.

Workbench should be built by progressive self-hosting: hand-coded seed substrate, command/result/document/view contracts, CLI/TUI/rendered proof, minimal Workbench shell, safe artifact editing, self-editing of layouts/themes/modules, then product/packs/modules/releases/agent WorkUnits. The next useful slice discussed was COMMAND-RESULT-VIEW-SLICE-01, unless current live repo gates require dependency-direction repair first. That slice should prove one command, one typed result, one view contract, projection descriptors, CLI output, TUI/text output, rendered-placeholder output, diagnostics, and evidence.

The chat also defined many future Workbench modules/workspaces: Validation Dashboard, Project Graph Explorer, Agent Work Board, Pack Browser, Interface Studio, Module/Pack Foundry, App Composer, Renderer/Theme Laboratory, Build/Test Console, Release Forge, Replay/Trace Viewer, Universe Explorer, Planet Generator Lab, Field Layer Stack Editor, Cut/Fill Terrain Delta Lab, Construction Blueprint Studio, Machine Graph Lab, Factory/Process Graph Editor, Robot Body/Spawn Fabricator Lab, Sensor/Fog-of-War Lab, Logistics/Infrastructure Planner, Governance/Claims Lab, and Performance/Complexity Budget Lab.
