### Decision: Domino Engine must remain ISO C89

[FACT] This was explicitly stated by the user in the opening constraints. It is final unless the user later supersedes it.

The rationale is portability and deterministic low-level control. It affects every future implementation prompt. Code generated for the engine must not use C99 features, C++ features, compiler extensions, platform APIs, or undefined behavior.

This decision might only be revisited if the user deliberately changes the engine language constraint.

### Decision: Dominium UI/tools must remain portable C++98

[FACT] This was also explicitly stated by the user. It is a hard constraint.

The rationale is similar: portability and compatibility. Dominium can be more expressive than the C engine, but it still must not depend on modern C++ features unless the constraint changes.

### Decision: No floats where determinism matters

[FACT] The user explicitly required no floats where determinism matters and instructed use of fixed-point Q formats.

The rationale is deterministic replay and cross-platform behavior. Floating-point can vary across platforms, compilers, instruction sets, and optimization settings. The proposed architecture therefore uses fixed-point math for coordinates, pose, temperatures, pressures, energy, flow, voltage, and other simulation-relevant quantities.

[UNCERTAIN / UNVERIFIED] Exact Q formats were not finalized.

### Decision: No hardcoded game semantics in the engine

[FACT] This was an explicit user constraint.

The engine should not know what “iron,” “dual gauge,” “MUTCD,” “substation,” or “ore smelter” means in game terms. It may know generic structures such as prototypes, ports, slots, materials, physics properties, graphs, and commands. The meanings come from content and rules.

This affects every subsystem. It is why roads, rails, signs, facilities, machines, and materials must be defined through RES/content rather than engine hardcode.

[INFERENCE] This was assistant-proposed and not contradicted; the user continued building on it.

The decision is that UI actions should compile into commands, and gameplay rules should evaluate deterministically. This makes replay and lockstep multiplayer possible. It also makes UI/tools a presentation and command-emission layer rather than a direct state mutator.

Alternatives considered implicitly: direct UI mutation or game-specific engine code. These were rejected because they violate determinism and layering.

[INFERENCE] This was assistant-proposed in response to the user’s overlap examples and became the working plan.

It lets one alignment host rails, lanes, sidewalks, ducts, cables, pipes, lights, catenary, barriers, drainage, and more.

This affects TRANS, rendering, simulation extraction, editing, collision/clearance, structural loads, and user tools.

[INFERENCE] The proposed solution was a small deterministic packing VM or equivalent content-driven rule system.

The rationale is mod extensibility. If cross-section compatibility is hardcoded, every new infrastructure type requires engine edits. If rules are content-defined, mods can add new profiles and slot arrangements.

## 6. Ideas Considered but Rejected, Superseded, or Deprioritised

### Independent overlapping splines

This idea was not developed as a serious final option, but it was the implied naive model behind the user’s question. It was superseded by corridors with slots.

It was rejected because overlapping independent splines would not reliably represent shared rails, shared structures, clearance envelopes, co-located utilities, or deterministic editing. It might remain useful as an import or authoring input, but it should canonicalize into corridors.

### Global grid-based construction

The user explicitly questioned whether the system was stuck to grids. The design rejected global grid lock. Grids remain useful as snapping aids, but not as authoritative engine constraints.

This rejection is strong for engine logic. It could be reconsidered only for optional UI modes, local building grids, or content-defined snapping.

### Direct UI mutation of engine state

This was rejected because it conflicts with deterministic replay and lockstep. UI tools should emit commands. The engine applies commands deterministically.

### Hardcoded gameplay semantics

This was rejected by the user’s initial constraints. The engine should not hardcode roads, rails, dual gauge, signs, recipes, or standards as game semantics. It can provide generic systems.

### Tolerance-based solvers and unbounded convergence

These were rejected for deterministic paths because they can vary by platform, iteration order, and runtime conditions. Fixed iteration counts, bounded budgets, and quantized thresholds are preferred.

### Raw world-space placement as the primary model

Raw world pose is not fully rejected. It is allowed as a fallback. But it is deprioritized because it does not survive host edits as well as parametric anchors. The preferred model is host-relative placement.

### Render mesh as simulation source-of-truth

This was rejected because meshes are poor authoritative data for deterministic simulation and editing. The source of truth should be parametric authoring data; meshes and tiles are compiled caches.

### Hand-placing every micro-object

This was rejected as the primary UX for real infrastructure. It may remain available for detail editing, but not for building highways, rail yards, substations, or plants at scale.

### AC power and multi-phase fluids as immediate baselines

The advanced simulation plan suggested simpler baselines first: deterministic DC-like load flow for power and simpler fluid network models before advanced AC or multi-phase complexity. These are deprioritized rather than permanently rejected.

## 7. Important Reasoning and Rationale

The visible rationale throughout the chat was shaped by a few pressures.

First, the engine must be deterministic. That requirement rules out casual use of floating point, unordered iteration, OS state, unbounded solvers, tolerance-driven convergence, and direct UI mutation. Second, the engine must be data-driven and mod-extensible. That rules out hardcoded gameplay semantics. Fifth, the user clearly wanted future continuity. The later part of the chat moved into Codex prompts, refactor integration, context transfer, downloadable packaging, and in-chat reading. ## 16. Final Plain-English Summary

This chat was a major architecture planning session for the Dominium/Domino deterministic engine. It began with advanced simulation and expanded into the broader question of how the engine should support complex, realistic, user-built infrastructure while remaining deterministic, data-driven, portable, and mod-extensible.

The starting point was a strict technical foundation. Domino Engine must be ISO C89, deterministic, fixed-point where determinism matters, free of OS APIs, free of hardcoded game semantics, and safe for replay and multiplayer lockstep. Dominium, the UI/tools/game layer, must stay in a portable C++98 subset. These constraints shaped every later design choice. The engine should provide generic systems; content and Dominium should provide meaning.

The first design area was advanced simulation. The chat sketched domains for heat, power, fluids, hydrology, atmospheres, vehicles, structural loads, destruction, and mod-extensible physics. These systems would use fixed-point deterministic math, stable tick ordering, content-defined parameters, TLV formats, serialization, replay, and checksum hooks. This was a high-level plan, not code.

The next step was gameplay layering. The chat established that gameplay mechanics should be implemented as deterministic commands plus data-driven rules. The UI should not directly mutate engine state. Instead, player actions become quantized commands that the deterministic engine applies in stable order. Dominium and content define recipes, research, signs, standards, machines, facilities, and other game semantics. This protects replay, multiplayer lockstep, and modding.

The central architecture work focused on infrastructure. The user wanted overlapping lines and splines to combine realistically: dual-gauge rails, roads with footpaths, utilities, electrification, street lights, pipelines, and undersea cables. The key design was to replace independent overlapping splines with **corridor bundles**. A corridor is a canonical alignment. A content-defined cross-section provides slots. Attachments occupy slots over station ranges. This lets many systems share one alignment without engine-specific hardcode.

To make this scalable, the chat introduced an authoring/runtime split. Users and tools edit high-level splines, corridors, junctions, attachments, modules, and rules. The engine compiles those into microsegments, frames, occupancy bitsets, connectivity graphs, spatial indices, structural references, and decor/render tiles. Performance depends on dirty flags, hot/cold data separation, deterministic rebuild budgets, and chunk-aligned indices. Extensibility depends on content-defined rules, possibly through a small deterministic packing VM.

The user then broadened the goal: the system should let players build basically anything they see in real life, including highways, interchanges, overpasses, subways, rail yards, substations, power plants, and industrial facilities. The answer was that players should edit high-level infrastructure primitives: corridors, junctions, vertical layers, modules, facilities, structures, anchors, and generated carriers. Complex infrastructure should be built through parametric templates and constraints, not by manually placing every micro-object.

Signage and markings became the DECOR system. Road markings are parametric decals. Signs are props. Automatic rules generate default signs and markings from topology, standards, and context. Manual overrides let users pin, suppress, move, or edit generated items. The same concept later generalized beyond roads to buildings, interiors, terrain, props, and rooms. This became the Host Surfaces + Anchor Frames model, where every host exposes stable local frames for placement.

A major late conclusion was that the engine must not be grid-bound. The user asked directly whether arbitrary location, inclination, rotation, and placement were possible. The answer established that the engine should be grid-agnostic. Grids are optional UI snapping aids only. Authoritative placement should use continuous fixed-point world space and local parametric spaces such as corridor stationing, surface `u,v`, structure-local frames, and room-local coordinates. Determinism comes from fixed-point math, quantized commands, stable IDs, ordered iteration, fixed iterations, and parametric anchors—not from grids.

The conversation then moved into implementation planning. A nine-step Codex prompt plan was created: define coordinate/placement specs, implement deterministic pose/frame math, upgrade TRANS for 3D corridors, upgrade STRUCT for arbitrary orientation and anchors, implement DECOR host placement, define deterministic placement commands, refactor UI snapping, add performance hardening, and update docs/invariants/migration notes. Another prompt was created for an existing GPT-5.2 refactor chat so these requirements could be merged into that plan rather than forked.

The main decisions to preserve are: strict language/determinism constraints; engine/game semantic separation; command-based UI actions; corridor bundles for co-located infrastructure; authoring data separate from compiled runtime caches; high-level infrastructure authoring through corridors/junctions/modules; DECOR as generated/manual visual detail; host-relative anchors for placement; no grid lock; and careful staged implementation.

The main unresolved issues are practical and technical: actual repo structure, existing refactor plan contents, current grid assumptions, exact Q formats, orientation representation, command schemas, packing VM design, microsegment policy, carrier ownership, junction/facility schemas, DECOR semantic keys, promotion boundary, localization, and checksum/source-vs-cache rules.
