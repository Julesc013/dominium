## 5. Decisions Made and Why

### Decision overview

The most important decisions were:

1. Domino remains C89 and deterministic.
2. Dominium remains C++98 and thin.
3. Engine/core do not contain actual gameplay data.
4. Content lives in packs/mods/TLV.
5. Every subsystem should use stable registries, models, prototypes, instances, and versioned serialization.
6. Logistics should use splines, packets, containers, and robotic manipulators rather than tile-tick per-item systems.
7. Buildings should support flexible player design, interior zones, and realistic terrain/foundation handling.
8. Vehicles/weapons should compile from designs into runtime-cheap objects.
9. GUI-first validation is required.
10. Path D advanced simulation is the next new-chat focus.

### Domino is C89; Dominium is C++98

FACT: This came directly from the user’s project prompt. It is final unless the user changes the project philosophy. The rationale is portability: Domino must target old and modern platforms, including retro systems where modern C/C++ assumptions are unsafe. C89 also forces a stable, simple ABI. Dominium can use C++98 because it is the product layer, but it still must remain portable and thin.

This affects every implementation prompt. Codex must not use C99 mixed declarations, VLAs, C++11, exceptions, RTTI, or platform-specific convenience APIs unless explicitly allowed.

### Determinism and fixed-point math are non-negotiable

FACT: The starter prompt emphasized deterministic fixed-point simulation across platforms and architectures. This is final. It exists because the project wants replay, deterministic networking, and stable behavior across old and modern hardware.

The alternative—using platform-dependent floats, wall-clock randomness, or nondeterministic solvers—would break replay/network guarantees. Advanced simulation must therefore use fixed-point solvers and controlled tick order.

### Engine/core must not know actual data definitions

FACT: The user explicitly reinforced this after the first seven prompts. This is one of the clearest decisions in the chat. The engine should not have hardcoded “iron,” “drill,” “belt,” etc. Actual definitions belong in packs, mods, and TLV content.

This affects the resource system, process runner, transport system, vehicles, research, policies, and advanced simulation. The engine may know generic concepts like “material,” “fluid,” “container,” “spline,” “process,” “field,” “volume,” or “structure,” but not game-specific instances.

This decision could be revisited only if the user deliberately chooses to build a non-generic game-specific engine, which is contrary to the entire conversation.

### The repeated subsystem pattern was adopted as the working architecture

INFERENCE: The exact phrase “Core + Model + Proto + Instance + Registry + TLV” came from assistant synthesis, not directly from the user. However, the user repeatedly asked to extend the architecture to the whole engine and then into Codex prompts, so this became the working design.

The rationale is that it gives each subsystem a stable core while letting behavior and content vary. It also supports mods, backward compatibility, and future physical models.

This decision should be treated as strong but still user-confirmable before being canonized word-for-word in a formal spec.

### Logistics should be spline-based and aggregate-based

FACT: The user explicitly wanted belts as splines and wanted arbitrary endpoints/midpoints. The assistant’s conclusion was that all transport paths should share a spline primitive and use transport profiles. Containers and packets reduce item simulation load.

This decision rejects tile-grid belt systems as the main architecture. It also supports arbitrary roads, rails, conveyors, pipes, tubes, chutes, escalators, etc.

It might be revisited only for a specialized content pack or local visualization, not as the core transport simulation.

### Inserters should be generic robotic manipulators

FACT: The user suggested inserters could be generic robot arms programmed for many tasks. The assistant developed that into job-driven deterministic manipulators.

The rationale is realism and extensibility. A robot arm can move items between belts, machines, crates, pallets, shelves, assembly stations, and more. It avoids tile-locked behavior.

This remains a design direction, though exact kinematics and programming UI are unresolved.

### Containers and packaging are fundamental

FACT: The user explicitly asked for crates, bags, pallets, containers, jars, cans, and boxes to reduce physical item simulation while obeying mass/volume/density. The assistant integrated this into logistics and process systems.

The rationale is both realism and performance. The project should not simulate thousands of loose items when real logistics would package them.

The unresolved part is how deeply nested containers may be and how detailed packing constraints should become.

### Buildings should be flexible but simulation-friendly

FACT: The user wanted Sims-style building design and realistic sloped terrain. The assistant proposed a hybrid: a building-local grid/shell, arbitrary world placement/yaw, interior zones, auto-generated foundations, and simplified collision.

This decision is partly an assistant proposal accepted by continuation. It is practical because fully freeform CAD-like buildings from day one would be hard for pathfinding, environment zones, and structural checks. The local grid can later be replaced or supplemented by geometry metamodels.

It could be revisited if the user wants freeform construction earlier.

### Vehicles/weapons compile into single runtime objects

FACT: The user explicitly asked how to construct vehicles/weapons and place/spawn them as single objects. The assistant proposed design-time blueprints compiled into aggregate runtime prototypes.

The rationale is performance and determinism. The player can design detailed module graphs, but the runtime object is a single body with aggregate state and module damage.

This remains important for Path D vehicle physics.

### GUI-first development was required

FACT: The user explicitly said they did not want CLI-only testing and wanted a minimal GUI. This changed the prompt sequence. Prompt 8 was generated to prove the rendering path, and later prompts built GUI demo flow and debug panels.

This matters because future assistants should not suggest only headless or CLI tests as the first validation path.

## 6. Ideas Considered but Rejected, Superseded, or Deprioritised

### Factorio-style tile/tick belts

This was rejected because the user wanted more realistic conveyors and lower processing cost. A per-tile, per-item simulation would produce excessive entity/tick load and push gameplay toward puzzle-like belt mechanics rather than realistic logistics. The replacement is spline-based transport with packets and containers.

This rejection is effectively final for the core engine. A content pack could still emulate tile belts visually or locally, but the main transport substrate should be spline-based.

### Simulating every individual item

The user’s packaging request made clear that most items should be aggregated into containers. Per-item simulation is too expensive and unrealistic at large industrial scale. The chosen model is to simulate containers, piles, packets, and inventories, with loose world items being exceptional.

This rejection is final for bulk logistics. Individual item entities may still exist for small visible cases or debugging.

### Hardcoded resource families in the engine

The pasted dres proposal explicitly rejected core knowledge of ore, oil, trees, mining drills, or wells. The chat carried that forward. Resource semantics belong in resource models and content prototypes, not the resource core.

This is final because it supports modding and future resources.

### Machines checking specific resource names

Similarly, machines should not ask “is this iron ore?” or “is this oil?” They should use profiles, tags, ports, process definitions, and model APIs. This prevents machine code from becoming a hardcoded web of resource-specific branches.

This is final.

### OS-native drawing

The starter prompt rejected OS-native drawing. All UI and rendering must go through `dgfx`, DVIEW, and DUI. This is final.

### CLI-only testing

The user explicitly rejected relying only on CLI. Headless and CLI remain necessary for servers, automation, and tools, but the user wants a minimal GUI smoke test early.

This is final as a workflow preference.

### General rigid-body physics for everything

The assistant deprioritized a full general rigid-body simulation for every object because it would be expensive and difficult to make deterministic across all targets. Instead, dynamic physics is constrained to vehicles, characters, movable props, debris, and other selected entities. Structures are static unless damaged.

This is tentative. Path D may refine it.

### Full per-voxel CFD for atmosphere/hydrology

The assistant deprioritized full 3D CFD because it would be too expensive. The selected direction is coarse hydrology cells, environment fields, zones, volumes, and network solvers. This is a practical compromise.

This could be reconsidered only for local high-fidelity simulations or tools.

### Forced flat foundations or ugly foundation blocks

The user explicitly did not want this. The proposed alternative is realistic automatic foundations, piles, stepped floors, cut/fill, and retaining walls.

This rejection is strong.

### Runtime simulation of every vehicle part

The user wanted vehicles/weapons to spawn as single objects. The detailed module graph should exist at design/build time, but runtime should be aggregated. This is final for performance.

### One giant Codex prompt

The user asked to split into fewer than ten. The one-prompt approach was superseded by seven prompts, then extended with prompts 8–18.

### Blindly preserving stale existing code

The user warned existing major components may be out of date. Future Codex prompts must inspect and reconcile code rather than duplicate or blindly preserve stale implementations.

## 7. Important Reasoning and Rationale

The strongest visible reasoning in the chat is that every architectural choice had to balance four pressures: determinism, performance, realism, and extensibility.

For determinism, fixed-point math and stable tick ordering are essential. This is why the chat avoided platform-dependent physics, wall-clock randomness, uncontrolled scripting, and raw binary network structs. It also motivated world hashes, replay validation, and deterministic content/tool outputs.

For performance, the chat repeatedly avoided per-object/per-item/per-tile simulation when aggregate models would work. Belts became splines with packets. Items became containers. Vehicles became compiled runtime objects. Environment became zone/field graphs instead of CFD. Dropped items became piles. Buildings use simplified collision and derived caches.

For realism, the user pushed against arcade-style simplifications. Conveyors should behave like real infrastructure, not tile puzzles. Buildings should sit realistically on slopes. Packaging should obey mass and volume. Machines should have ports, robotic manipulators, and assembly-line throughputs. Environmental systems should account for rain, wind, pressure, gases, temperature, pollution, and flooding.

For extensibility, the chat avoided special-case engine code. Content packs define actual things. Subsystems register models. TLV schemas support versioning. Blueprints compile to protos or instances. Worldgen providers are modular. Tools generate content. This is why the architecture became broad: the user is building not just a game, but an engine and product ecosystem.

The biggest tradeoff is complexity. The assistant repeatedly tried to keep advanced systems coarse and modular rather than over-specified. The user explicitly asked for extensibility without crossing diminishing returns. That means future work should prefer minimal stable interfaces and testable vertical slices over huge theoretical frameworks.

## 16. Final Plain-English Summary

This chat was a major architecture and planning conversation for the Dominium/Domino project. It began with the user supplying a comprehensive master prompt defining Domino as a deterministic fixed-point C89 engine and Dominium as a C++98 product suite built on top. That initial context established the project’s strict constraints: deterministic simulation, stable ABI, portability across retro and modern targets, no platform-specific behavior in simulation, platform abstraction through `dsys`, rendering through `dgfx`, vector-first UI fallback, versioned TLV content/save formats, and preservation of old saves, packs, mods, and tools.

The first concrete topic was resource mechanics. The user pasted a modular resource-system proposal from another chat, and that became the baseline for `dres`: a resource core that only stores generic channels and values; resource models that interpret those values; content prototypes for deposits/materials/items/processes; and interaction systems such as machines, tools, jobs, and logistics. From there, the discussion expanded into materials, items, recipes, deposits, extraction profiles, jobs, and structures.

The conversation then shifted from abstract resources to gameplay and simulation systems. The user asked how a player would build a complete machine workshop. The answer framed the workshop as a chain: extraction, processing, components, power, logistics, control, and assembly. The user then pushed for realistic, low-CPU conveyors and inserters. This produced one of the most important design directions: transport should be spline-based and packet/container-based, not tile-by-tile per-item simulation. Inserters became generalized robotic manipulators rather than fixed tile devices. The user then expanded transport to arbitrary endpoints, midpoints, roads, rails, pipes, tubes, chutes, slides, ladders, elevators, escalators, and other infrastructure, all unified under a generic spline/profile system.

Packaging became a core performance design. The user wanted crates, bags, pallets, containers, jars, cans, boxes, and similar packaging so the engine would not need to simulate every item individually. The proposed system uses container prototypes and runtime container states, with mass, volume, density, and packing constraints derived from content data. Packing/unpacking is represented as generic processes.

The conversation then broadened into buildings, construction, vehicles, gravity, destruction, and dropped items. The user wanted Sims-style arbitrary buildings, machines inside buildings, off-grid placement, arbitrary rotations/elevations, and realistic sloped terrain construction without ugly flat foundations. The proposed design used local building shells, arbitrary world placement/yaw, simplified simulation geometry, interior zones, and automatic foundations such as piles, stepped floors, cut/fill, retaining walls, bridges, and tunnels. Vehicles and weapons were designed as player-authored module graphs that compile into single runtime objects with aggregate physical stats and module state. Gravity and destruction were supported in constrained ways, and dropped items became aggregate piles or containers.

The user then asked how hydrology, atmosphere, lithology, rain, wind, temperature, pressure, pollution, gases, radiation, magnetic fields, and light interact with buildings and enclosed spaces. The answer proposed a zone/field model: buildings, vehicles, caves, and terrain generate volumes; portals and materials connect them; and environmental models update pressure, temperature, gases, humidity, light, pollution, water, radiation, and related fields. This later became the foundation for Path D advanced simulation.

The chat then expanded into the whole engine and product suite. The repeated architectural pattern became **Core + Model + Proto + Instance + Registry + TLV**. Every subsystem should have a core API, pluggable model family, data-only prototypes, runtime instances, registries, and versioned TLV save/load.
