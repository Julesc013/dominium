### Topic 4 — Stable APIs, ABI boundaries, and extensibility

One of the most important topics was how to make platform and renderer APIs extensible without constantly breaking old code. The answer evolved into a strict ABI policy:

* Public boundaries use C ABI.
* Structs are POD.
* No C++ types, templates, exceptions, RTTI, STL, or inline semantics across ABI boundaries.
* Every interface starts with `u32 abi_version` and `u32 struct_size`.
* Core APIs remain small.
* Features grow through versioned vtables or optional extension interfaces.
* Optional features are discovered through `query_interface`.

This topic matters because the user wants Dominium to last for years and support everything from retro systems to future AAA features. Without a stable ABI and extension pattern, advanced backends would force complexity onto simple platforms, and old data/plugins could break.

The final plan embeds these ideas in Prompt 1.

### Topic 5 — Capability registry and deterministic backend selection

Another major topic was how to choose backends. Instead of hardcoding platform/render decisions or scattering `#ifdef`s throughout logic, the plan uses a centralized capability registry.

Backends declare required OS/ISA features, determinism grade, performance class, feature bits, and function pointers. The launcher and engine selection logic then choose implementations based on profiles such as `compat`, `baseline`, or `perf`, plus explicit flags like `--gfx=soft` or `--lockstep-strict=1`.

This matters because it allows the same engine to run on minimal, compatibility, or performance-oriented configurations without changing correctness. It also makes backend selection inspectable, which helps debugging and reproducibility.

Prompt 2 implements this.

### Topic 6 — Determinism grades and lockstep safety

The user explicitly required determinism grades:

* D0: bit-exact.
* D1: tick-exact.
* D2: best-effort.

The plan formalized that lockstep strict mode requires D0. No backend may silently downgrade determinism. Optimized paths need deterministic reference fallbacks.

This matters because Dominium’s deterministic simulation and replay/networking goals could be destroyed by a platform-specific optimized path or renderer/audio behavior feeding back into simulation. The plan therefore separates deterministic control from non-deterministic presentation.

The software renderer and null backends are intended as D0 references. DX9 is likely D2 unless proven otherwise, and should not be used as an authoritative lockstep dependency.

## 5. Decisions Made and Why

A compact decision map:

| Decision area              | What was decided                                          | Status                             |
| -------------------------- | --------------------------------------------------------- | ---------------------------------- |
| Language baseline          | Domino C89, Dominium C++98 forever                        | FACT / final                       |
| Modern standards           | Optional acceleration only                                | FACT / final                       |
| Runtime language switching | Rejected                                                  | FACT / final                       |
| ABI boundaries             | C ABI, POD, function tables                               | FACT / final                       |
| Interface layout           | `u32 abi_version`, `u32 struct_size` first                | FACT / final                       |
| Architecture               | Facade + backend per evolving subsystem                   | FACT / final                       |
| Backend selection          | Central capability registry                               | FACT / final                       |
| Determinism                | D0/D1/D2, lockstep strict D0 only                         | FACT / final                       |
| Serialization              | TLV-as-ABI with versioning/migrations                     | FACT / final                       |
| Networking                 | Versioned handshake with build/schema/content/determinism | FACT / final                       |
| Active plan                | Final 9-prompt post-master pack                           | FACT / final                       |
| Tier‑2 scope               | Gated/compile-correct in current pack                     | FACT with caveat                   |
| Future AAA features        | Roadmap/spec slots, not current implementation            | INFERENCE / current scope decision |

### C89/C++98 as permanent baselines

The user explicitly stated that Domino core must remain ISO C89 and Dominium layer must remain C++98. This was final and non-negotiable. The rationale is compatibility and longevity: the engine wants to support old toolchains, retro platforms, and stable source/data compatibility. This affects every public header, every ABI boundary, and CMake presets.

Alternatives such as adopting C11 or C++11 as the baseline were not accepted. They may exist only as optional acceleration layers.

This decision would only be revisited if the user explicitly reverses the core language policy.

### Runtime language-standard switching was rejected

The user clarified that compatibility is source-tree and data compatibility, not a runtime choice between C89 and C11. This prevents a confused architecture where launcher profiles might select language standards. The plan explicitly says profiles can select backends and features, not the language standard used to compile the engine.

The rationale is that language standards are compile-time properties. Runtime selection makes sense for backend implementations, not for C89 versus C11.

### Stable ABI boundaries use C ABI and POD structs

The user required that all cross-module, cross-DLL, and plugin boundaries use C ABI only, POD structs only, and explicit function tables. No C++ types, templates, exceptions, RTTI, STL, or inline semantics can cross those boundaries.

The reasoning is straightforward: cross-compiler and cross-language ABI stability is fragile. A C ABI with versioned POD structs is much safer and more portable.

This affects DSYS, DGFX, plugin APIs, jobs, SIMD, IO streaming, and future extensions.

### Every interface starts with `abi_version` and `struct_size`

This was user-mandated. It gives every interface a way to negotiate version and size. That matters when old and new modules interact, because callers can know which fields/functions exist.

This also makes API evolution mechanical: new versions or extensions can be added without silently corrupting ABI layout.

### Facade + backend architecture was chosen

For any subsystem that may evolve or optimize later, the plan requires a stable facade and one selected backend at runtime. The minimum subsystems include platform facets, renderers, jobs, SIMD/math kernels, and IO streaming.

The rationale is that advanced features should not force simple or legacy platforms to implement everything. A CP/M or DOS backend can have a minimal stub/baseline implementation; Win32 or modern platforms can have optimized/threaded/SIMD backends. Callers talk to the facade.

This decision supersedes a monolithic platform API as the long-term architecture.

### Central capability registry was chosen

Backends must declare their requirements and capabilities. A centralized registry then chooses backends based on launcher profiles, runtime detection, and deterministic selection rules.

This decision was made because hardcoded selection does not scale. The user wants compatibility profiles, performance profiles, lockstep strictness, and future advanced features. A registry makes selection inspectable and reproducible.

### Determinism grades were formalized

The user defined D0, D1, and D2:

* D0 means bit-exact.
* D1 means tick-exact.
* D2 means best-effort.

Lockstep strict mode requires D0. This decision protects deterministic replay and multiplayer. It also lets presentation backends such as DX9 or future ray-tracing renderers exist without pretending they are authoritative deterministic systems.

If a backend’s determinism cannot be proven, the safe default is D2.

### Serialization became public ABI

The user required saves, assets, replays, and config to be treated like ABI. The plan uses one chunked/TLV container format with versioned chunks, skip-unknown behavior, and explicit migrations.

The rationale is long-term compatibility. Dominium is expected to evolve over years; unversioned formats would become technical debt and break old saves/mods/replays.

### Networking handshake was versioned

The user required handshake fields: engine build ID, simulation schema version, determinism profile, and content pack hashes. This prevents mismatched multiplayer sessions from proceeding silently.

The default policy is to reject mixed versions unless explicitly supported.

### The final active plan became the 9-prompt pack

Several earlier prompt plans existed, but the user’s later preference for fewer larger prompts and git commit discipline produced the final 9-prompt pack. This is the active implementation plan. Earlier prompt packs are historical or superseded.

### Tier‑2 backend completion was scoped as gating and compile correctness

This is a decision with a caveat. Because Codex runs on Windows, the final prompt pack treats non-Windows Tier‑2 platforms as host-gated and compile-correct when built on their native hosts, not fully runtime-validated on Windows. This makes practical sense, but it may not satisfy a stronger interpretation of “fully complete” if the user later demands runtime Tier‑2 validation.

This may need to be revisited.

### Future AAA features were preserved as roadmap material

The user discussed many future systems. The plan did not make them current implementation tasks. Instead, it preserved them as architecture slots and future spec-book candidates.

This avoids scope explosion while keeping the engine future-proof.

## 6. Ideas Considered but Rejected, Superseded, or Deprioritised

### Runtime C89/C11 switching

This was explicitly rejected. It was considered only because the user wanted baseline and modern layers, but the user clarified that language standards are compile-time, not runtime-selectable. Runtime selection is allowed for feature profiles and backends only.

This rejection is final unless the user reverses the language policy.

### CMake dependency downloads

The user prohibited adding dependencies that require downloads unless explicitly approved. This affects Vulkan, Wayland protocols, shader compilers, GLEW/GLAD, DXC, and any future audio/voice codecs. System-provided SDKs may be used when present, but the build system must not fetch dependencies.

This rejection is final for current prompts.

### C++ ABI boundaries

The idea of exposing C++ types or templates across module boundaries was rejected by the stable ABI policy. It might be acceptable inside private implementation code, but not across public/plugin/module boundaries.

This rejection is final for ABI surfaces.

### Monolithic platform API

A monolithic platform layer was superseded by platform facets. The old API may remain as a compatibility shim, but the long-term architecture is facet-based.

This matters because filesystem, time, threading, networking, audio I/O, clipboard/text, windowing, and input can evolve independently.

### Single fast path only

The user explicitly prohibited “single fast path only” implementations. Every optimized path must have a deterministic reference fallback. This prevents the engine from depending on SIMD/threaded/platform-specific behavior for correctness.

### Unversioned data formats

Unversioned serialization was rejected. Every save/asset/replay/config format should use a versioned TLV container with skip-unknown behavior and migration paths.

### Full Vulkan or D3D12 implementation in the current pack

Vulkan and D3D12 were discussed as future/advanced renderers. Vulkan appears in the prompt pack only as a gated backend that compiles when system SDKs exist. D3D12 was explicitly treated as planned/future because it was not part of the current `dgfx` contract described in the chat.

These are temporary deprioritisations. They can become future prompt phases.

## 7. Important Reasoning and Rationale

The visible reasoning throughout the chat was shaped by a few recurring tradeoffs.

First, the user wanted both **long-term compatibility** and **future advanced features**. Those goals conflict if advanced features are bolted directly into core APIs. The solution was to keep core APIs small and stable, and add advanced capabilities as extensions. This lets early/retro/simple backends exist alongside future high-end backends.

Second, the user wanted **determinism** but also wanted high-end graphics and audio. Rendering and audio can be nondeterministic across hardware, drivers, and platforms. The plan therefore separates authoritative deterministic simulation from presentation. Renderer/audio output cannot feed back into simulation. The software renderer and null backends can provide deterministic references, while modern GPU/audio paths can be D2 presentation systems.

Third, the user wanted **C89/C++98 compatibility** but also future performance. The plan therefore uses baseline implementations as correctness paths and modern/SIMD/threaded layers as optional accelerations. Correctness cannot depend on modern features.

Fourth, the user wanted **Codex productivity** but not broken intermediate states. This led to fewer larger prompts, but each prompt still contains discovery commands, verification, stop-and-ask conditions, and commit discipline. The plan tries to balance Codex’s ability to generate large code changes with the need to preserve buildability.

Fifth, the user wanted **future spec-book construction**. That means uncertainty and provenance matter. The chat repeatedly preserved labels such as FACT, INFERENCE, and UNCERTAIN / UNVERIFIED. This prevents future aggregation from treating assistant suggestions or brainstorms as binding decisions.

The biggest visible risk being managed was architectural entropy: if platform/render/audio/input/networking/modding all grow in ad hoc ways, the engine will become unmaintainable. The chosen answer was mechanical discipline: vtables, extensions, capabilities, determinism grades, TLV formats, build matrices, smoke tests, and changelog-ready commits.
