## 2. Ultra-Condensed Carry-Forward Capsule

This old chat is the platform-support planning source for Dominium. The core outcome is not a final implementation spec but a structured priority decision and target inventory. The user first asked what systems would be supported, then narrowed to what operating systems Dominium would run on, then expanded to console families, PC handhelds, Android, iOS, Web, AR/VR/XR, and cross-cutting runtime/API targets. The user also supplied a large hardware/software inventory covering PlayStation, Xbox, Nintendo, PC handhelds, Android devices and ROMs, Apple devices and OSes, Web technologies and browsers, AR/VR hardware and runtimes, and rendering/runtime APIs.

The single strongest carry-forward fact is the user’s latest platform priority statement: “Android, iOS, and Web will be our primary focus after PC, and should be primary top tier support likewise.” This means the future project state should treat PC as the first primary platform family and Android, iOS, and Web as primary top-tier support after PC. A future aggregator must not demote Android/iOS/Web into late ports or minor compatibility targets. PC means at least Windows/macOS/Linux planning, though exact minimum OS versions and CPU architectures were not established. Android, iOS, and Web must shape architecture from the beginning: lifecycle, input, UI, memory, storage, rendering, asset loading, browser/mobile constraints, QA, and release planning all need to reflect Tier-0 status.

The working model generated by the assistant after the user’s priority statement was: Tier-0 = PC + Android + iOS/iPadOS + Web; Tier-1 = current commercial closed consoles such as PlayStation/Xbox/Nintendo where official access exists; Tier-2 = legacy consoles, obsolete handhelds, retro OSes, bare-metal/firmware, and research/constrained builds. This model is useful but should be labeled as assistant synthesis, not a user-authored spec. The user wants broad coverage, but the chat did not establish literal full-parity support for every listed device.

Several concepts need careful treatment in aggregation. “Domino” was introduced by the assistant as a possible reduced engine/core name distinct from the full Dominium product, but the user never confirmed the name. Preserve it only as an unconfirmed placeholder. The assistant also proposed platform contracts, runtime categories, OpenXR/WebXR for XR, and Vulkan/Metal/WebGPU-first rendering ideas. These are planning suggestions or inferences, not final user decisions. The target inventory is a conversation artifact, not verified current external truth. Platform-holder rules, SDKs, store policies, browser features, hardware status, legal conditions, and product availability must be verified from current official sources before implementation or formal spec hardening.

The future spec should use a support taxonomy rather than the phrase “support everything.” Recommended categories include full product parity, full product with platform-specific constraints, reduced product, engine/core-only demo, emulator-hosted compatibility, remote-play only, community/research, and unsupported. This taxonomy is essential because the chat covers modern PCs, mobile OSes, browsers, closed consoles, retro consoles, handheld PCs, embedded/automotive Android, watch/TV/spatial Apple devices, and XR headsets.

The best next task after aggregation is to create a formal Platform Support Matrix. It should include platform family, specific device/OS/runtime targets, tier, support scope, minimum OS/API, CPU/ABI, graphics backend, input model, audio, storage/save model, networking, distribution channel, QA obligation, certification/legal constraints, dependencies, risks, verification status, and next action. Start with Tier-0 PC, Android, iOS/iPadOS, and Web. Then classify PC handhelds under PC, consoles as gated secondary targets, XR as undecided/future unless elevated, and legacy systems as research/constrained unless the user later prioritizes them.
