The most important rejected idea was trying to build the entire architecture at once. It was not rejected because it was wrong; it was rejected because it was too broad for the first implementation pass. The assistant recommended focusing on a Milestone-0 vertical slice first. A huge up-front dsys/dgfx abstraction was also deprioritized. The assistant recommended growing those APIs through real use cases. This avoids architecture becoming abstract but untested.

For startup behavior, environment/config mode overrides were deferred. The assistant recommended not adding `DOMINIUM_MODE` or config-based mode preferences in v1 because they complicate precedence and reproducibility. This is not a permanent rejection. It can be reconsidered after the config/instance system is stable.

The idea of falling back from GUI/TUI to another mode on any error was rejected in the recommendation. Only structural unsupported errors should trigger fallback. If the GUI launches successfully but encounters a real content/config error, hiding that by dropping to TUI or CLI would make debugging harder.

Including headless as part of AUTO startup fallback was also rejected. Headless is a game/server mode, not a user interface fallback. It should be explicit or triggered by server flags.

Duplicated product-specific CLI/TUI/GUI pipelines were rejected because they conflict with the user’s shared-common-components goal.

OS-native product drawing was rejected by the master architecture because all rendering must go through dgfx.

C++11+ and floating-point deterministic sim were rejected by the user’s constraints.

## 7. Important Reasoning and Rationale

The visible reasoning throughout the chat was guided by a few recurring tradeoffs.

The first tradeoff was **scope versus progress**. The architecture is intentionally ambitious, but trying to implement all of it immediately would create high failure risk. The recommendation to build Milestone 0 first was a way to preserve ambition while creating an executable foundation.

The second tradeoff was **portability versus convenience**. C89 and C++98 are less convenient than modern languages, but they support the project’s broad platform ambitions. This affects everything from numeric typedefs to startup code and CMake.

The third tradeoff was **determinism versus platform-native behavior**. Direct OS APIs, floating-point simulation, wall-clock timing, and renderer-dependent logic can all introduce variation. The project avoids this by routing through abstractions and isolating deterministic simulation from UI/render/platform details.

The fourth tradeoff was **generic architecture versus concrete vertical slices**. dsys, dgfx, UI, input, packs, and startup systems need to be general, but they should be proven by simple working paths. This is why the prompts often start with stubs or one platform/backend.

The fifth tradeoff was **user experience versus scriptability**. The startup policy balances both. Terminal invocation should be scriptable CLI. Desktop launch should be GUI-friendly. Explicit flags preserve control.

The sixth tradeoff was **continuity versus overcompression**. The later report package preserved a lot of detail because losing a prompt, caveat, or unresolved issue would cause future rework. The current report translates that structured package into prose so it is easier to understand.
