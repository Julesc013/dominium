# Human-Readable Chat Report — Dominium/Domino Engine Refactor Planning

## 1. One-Page Orientation

This chat was a long-form architecture and planning session for the **Dominium/Domino deterministic engine**. Its central purpose was not to write code, but to define how a very large, deterministic simulation engine should be structured so that it can support complex world simulation, agents, infrastructure, future DLCs, modding, replay, multiplayer lockstep, and large-scale performance without collapsing into hardcoded special cases.

**FACT:** The user began by establishing very strict architectural constraints. The **Domino Engine** must be ISO C89, deterministic, portable, fixed-point where determinism matters, and free of OS, platform, UI, rendering, or gameplay-semantic dependencies. The **Dominium** layer is the C++98 UI/tools layer and must only visualize, debug, or emit deterministic commands. The engine must preserve strict layering across systems such as BUILD, TRANS, STRUCT, DECOR, SIM, RES, ENV, JOB, and AGENT. It must use TLV-versioned content formats, stable IDs, deterministic tick scheduling, replay, and input-driven lockstep multiplayer.

The discussion started with advanced physical simulation systems: heat, power, fluids, atmospheres, vehicles, structural loads, destruction, and a mod-extensible physics framework. The important outcome was not a final implementation for each solver, but a general design rule: every heavy system should be content-driven, deterministic, fixed-point, bounded, and integrated through generic primitives such as graphs, fields, ports, events, messages, actions, deltas, and replay hashes.

The focus then shifted to agents and life systems. The user wanted the engine to support everything from unintelligent wildlife and pests to commandable pets, intelligent workers, robots, players, companions, plants, trees, crops, weeds, swarms, herds, flocks, and predator-prey dynamics. The solution that emerged was a generic **actor substrate**: actors are not hardcoded as “human,” “animal,” “robot,” or “plant.” They are entities composed from components. Their behavior flows through a deterministic pipeline: **sensors produce observations; minds or controllers produce intents; actions validate intents and produce deltas; deltas mutate state in a sorted commit phase.** This allows arbitrary stimuli and arbitrary actions without the core engine knowing what those things “mean.”

Performance became a major theme. The chat established that the engine cannot simulate every blade of grass, pest, animal, vehicle, building surface, pipe, or social interaction at full fidelity all the time. Instead, the engine should use a representation ladder: **R0 full**, **R1 lite**, **R2 aggregate**, and **R3 dormant**. Dumb animals and plants can be represented as fields or population aggregates until they become important. Buildings and infrastructure can be represented as parametric authoring state plus compiled caches. Networks can collapse to aggregate envelopes when far away. Work must be budgeted, deferred deterministically, and accumulated so that performance optimizations change latency or fidelity, not authoritative outcomes.

Later, the user introduced two future DLC directions: **Interstellar** (`data/space/`) and **Wargames** (`data/war/`). Rather than add “space” or “war” systems to the core, the architecture generalized their requirements. Space becomes a use case for **domains, frame graphs, and propagators**. Fog of war and information dissemination become use cases for **observer contexts, belief stores, visibility/occlusion, and communications graphs**. Combat becomes a use case for generic interactions, actions, effects, damage, and deltas rather than a hardcoded combat subsystem.

A major correction came when the user pasted a sister-chat prompt requiring arbitrary placement, no grid lock, full 3D curved geometry, TRANS corridors, STRUCT buildings, DECOR signage/markings, corridor co-location, and real-world infrastructure scale. This forced the plan to explicitly support **fixed-point pose + parametric anchors** as the source of truth, not baked meshes or grid cells. TRANS became the canonical corridor/alignment system with cross-section slots. STRUCT became the system for arbitrary footprints, volumes, enclosures, carriers, surfaces, and sockets. DECOR became a host-agnostic rulepack and override system for signs, markings, decals, props, and surface details.

The final major output was a **14-prompt Codex implementation roadmap**, followed by a documentation validation prompt. These prompts are the practical bridge from architecture to implementation. They cover the deterministic engine spine, packets, scheduler, fields/events/messages, LOD, pose/anchors, TRANS, STRUCT, DECOR, agents, graphs, domains, knowledge/comms, replay, regression tests, and documentation consistency.

The chat then produced a maximum-fidelity context transfer packet and later a downloadable report package. Those are artifacts of preservation, but the substance of the chat is the architecture itself: a deterministic, modular, data-driven engine plan that can scale from plants and wildlife to cities, power grids, spacecraft, fog of war, and future DLCs without violating strict determinism.

The most important thing to remember is this: **the project direction is to build a small number of generic deterministic primitives, then express all gameplay, simulation, mod, DLC, and content semantics through data-driven composition on top of those primitives.**

## 2. The Story of the Conversation

### The chat began with strict engineering constraints

The user opened by defining the assistant’s role as a senior simulation systems architect for the Dominium/Domino deterministic engine. This was not a casual brainstorming session. The user specified a technical tone and a set of hard constraints that governed the entire conversation.

**FACT:** The Domino engine must be ISO C89. The Dominium UI/tools layer must be portable C++98. Engine code must not use OS APIs. Determinism matters, so floats are disallowed in deterministic paths. The engine must avoid platform-dependent behavior and hardcoded game semantics. Everything should be content-driven.

The user also listed existing conceptual systems: dsys for platform abstraction, dgfx for renderer abstraction, deterministic world models using Q16.16 / Q24.8 coordinates, TLV content formats, deterministic scheduler and replay, lockstep multiplayer, and existing domain names such as RES, ENV, BUILD, TRANS, STRUCT, JOB, and AGENT.

The first request was to architect advanced simulation systems: heat, power, fluids, atmosphere, vehicles, structural loads, destruction, and a mod-extensible physics framework. The initial plan established a reusable simulation stack: deterministic scheduler, world storage, TLV registries, fixed-point math, physics families, graph and field layers, and cross-system interaction through ports, fields, events, and replay hashes.

### The discussion shifted from physics to actors and life

The user then redirected the chat toward “agents/actors/characters/players/npcs/humans/robots/companions/pets/pests/wildlife/flora/fauna/etc.” This changed the center of gravity from physical simulation to living and artificial actors.

The answer reframed actors as compositional entities rather than hardcoded classes. A human, robot, dog, deer, crop, tree, pest swarm, or machine controller would be distinguished by components and prototypes, not by engine branches. Components could include body, motion, needs, health, senses, mind, inventory, role, reproduction, social relationships, or group membership.

The user then asked how the engine could support unintelligent wildlife, commandable pets, highly intelligent workers and robots, plants, trees, grass, weeds, crops, swarms, herds, flocks, predator-prey behavior, and intelligent behavior. The answer introduced an “intelligence spectrum”:

* wildlife can use simple reactive minds;
* pets can use commandable minds with bond/training/ownership constraints;
* workers and robots can use job or planning systems;
* plants can use growth/resource components rather than minds;
* groups can use group controllers plus local steering;
* predator-prey can emerge from species interaction matrices, needs, sensors, and fields.

This was an important development because it prevented the engine from needing separate hardcoded systems for humans, animals, robots, plants, pests, and groups. Everything became a variant of the same substrate.

### Arbitrary behavior led to the typed IO/action pipeline

The user then asked a more general question: how to support arbitrary actors with arbitrary behaviors, arbitrary stimuli, and arbitrary actions. The answer generalized the agent system into a strict pipeline:

1. sensors convert world state into observations;
2. minds or controllers convert observations and internal state into intents;
3. actions validate and execute intents by producing deltas;
4. deltas are applied later in a deterministic commit phase.

This became one of the central architectural ideas of the chat. It makes behavior extensible without giving minds direct write access to the world. A mod can add new sensors, observations, intents, capabilities, or actions, but state mutation still passes through a deterministic pipeline.

The answer also introduced a deterministic bytecode VM as the preferred way to support modded behavior. A question was raised about whether native-code plugins should also be allowed, but the user did not answer. That remains unresolved.

### Performance concerns pushed the architecture toward LOD, fields, and aggregates

The user then asked how to optimize systems for dumb agents, especially large numbers of plants and animals. This led to the representation ladder idea:

* R0/Tier 0: full entity simulation;
* R1/Tier 1: lite entity simulation;
* R2/Tier 2: aggregate field or population simulation;
* R3/Tier 3: dormant persisted state.

For plants, the plan was to default to vegetation fields rather than individual entities. Grass, weeds, and early crop stages should be density/biomass/moisture/nutrient fields. Trees or large interactive specimens can become entities. For wildlife and pests, far-away or unimportant populations can be represented as counts, densities, age bins, energy bins, and movement fluxes. Individuals are promoted only near players, owned structures, hazards, interactions, or other deterministic interest volumes.

This performance discussion then expanded to the entire engine. The user asked whether the same methods could apply everywhere. The answer said yes: representation ladders, typed IO, deterministic budgets, and data-defined wiring should be core engine services, not agent-specific tricks.

### Heavy systems, especially buildings, became a major focus

The user asked what else could be improved and whether these techniques could optimize custom buildings. The answer identified buildings as one of the heaviest subsystems because they combine geometry, enclosures, ports, surfaces, pathing, heat, atmosphere, fluids, power, structure, and player editing.

The building plan introduced separate representations for buildings:

* R0 full building with detailed occupancy, enclosures, ports, surfaces, supports;
* R1 lite building with coarse zones and simplified thermal/load models;
* R2 aggregate building with scalar envelopes such as sealedness, thermal mass, power demand, storage totals, structural integrity;
* R3 dormant persisted state.

The answer also introduced incremental compilation from edit buffers into derived artifacts: occupancy, ports, connectivity, enclosures, surfaces, supports, nav hints, and boundary lists. This later became a key part of the TRANS/STRUCT/DECOR prompt plan.

### The plan was generalized across systems and made more modular

The user then asked which other systems could benefit from the same approach and how deterministic multiplayer would be maintained. The answer extended the methodology to pathfinding, economy/logistics, fluids/gases, power grids, heat/fire, structural integrity, terrain, rendering, and audio. It also gave a deterministic multiplayer checklist: interest volumes must be deterministic; promotion/demotion must use stable ordering; budgets must only affect latency, not outcomes; all queues must be deterministic; solvers must use fixed iterations; and hashes must localize desyncs.

The user then asked whether there was value in making systems even more modular, and which systems should be combined or split. The answer proposed a small set of reusable engine kernels: scheduler, registries, spatial index, graphs, fields, event/message buses, representation ladder, deterministic VM, probes, traces, hashing, replay. It recommended combining all network-like systems under one graph toolkit, all continuous-ish phenomena under fields, all decision systems under the same sensor/intent/action pipeline, and all editing under edit-buffer plus compiler pipelines. It recommended splitting STRUCT into occupancy, ports, connectivity, enclosures, supports, surfaces, and nav hints.

### Future DLC requirements were generalized into core primitives

The user introduced future **Interstellar** and **Wargames** DLCs. This could have led to hardcoded space and combat systems, but the answer deliberately generalized the requirements.

For Interstellar-like needs, the plan introduced generic **domains**, **coordinate frame graphs**, and **propagators**. These can represent different spatial models and time-evolving states without the engine knowing whether something is “space,” “surface,” or “orbit.”

For Wargames-like needs, the plan introduced generic **observer contexts**, **belief stores**, **visibility/occlusion**, **communications graphs**, and generic interactions. Fog of war became query filtering over a belief state, not a renderer trick. Comms became deterministic packet routing with integer tick latency and congestion queues.

The user explicitly asked to make this more extensible and coherent as core engine architecture, not space- or war-specific. That confirmed the direction: space and war were to be content/DLC applications of generic primitives.

## 3. Main Topics We Discussed

### Topic 1 — The deterministic engine foundation

The foundation of the chat was the user’s hard constraint set for Domino and Dominium. Domino is the deterministic engine core. Dominium is the UI/tools layer. This distinction shaped every answer.

The core idea is that deterministic simulation cannot tolerate platform-dependent behavior, nondeterministic iteration, floating-point inconsistencies, OS dependencies, or UI-driven state mutation. The engine must be portable, reproducible, and replayable. Therefore the architecture repeatedly favored fixed-size integer IDs, fixed-point math, TLV schemas, sorted iteration, deterministic queues, and replay hashes.

The conclusion was that the engine needs a small set of generic deterministic primitives. These primitives should not know gameplay semantics. They should know how to schedule, store, serialize, hash, route, query, and commit state changes. Content and mods define what those things mean.

What remains uncertain is the actual repository state. The chat designed architecture and prompts, but did not inspect source code. Therefore the plan needs repo verification before implementation.

### Topic 2 — Advanced simulation systems

The first substantive design topic was advanced simulation: heat, power, fluids, atmosphere, vehicles, structures, destruction, and mod-extensible physics.

The discussion established that these systems should be integrated through common mechanisms rather than isolated custom solvers. Heat uses fields and ports. Power uses deterministic graphs. Fluids use both terrain-scale hydrology and pipe/duct networks. Atmospheres use enclosed-space gas mixtures and pressure/temperature/humidity states. Vehicles use deterministic kinematics and constraints. Structures use support/load graphs and damage accumulation. All should be content-driven and fixed-point.

The conclusion was not a final physics implementation. It was a framework direction: each physics family should be a deterministic family plugged into the scheduler, using content-defined TLV prototypes, fixed iteration solvers, stable vtables, and replay hooks.

The unresolved work is to generate detailed subsystem implementation prompts for heat, power, fluids, atmospheres, vehicles, and structural loads after the core engine spine is implemented.

### Topic 3 — Agents, actors, life, and behavior

The user wanted the engine to support a wide range of living and artificial actors. The important design move was to avoid hardcoded categories. A tree, deer, worker, robot, pet, pest swarm, and player-controlled human are all entities or aggregate representations with different components and controllers.

The behavior substrate became: sensors generate observations; minds/controllers generate intents; actions validate intents and produce deltas. This allows arbitrary behavior and arbitrary stimuli without allowing arbitrary mutation. It also supports different intelligence levels. Wildlife can be reactive. Pets can be commandable. Workers and robots can be job-driven. Plants may not need minds at all; they can use growth/resource systems.

The remaining uncertainty is plugin policy. A deterministic VM was proposed, but whether native-code plugins are allowed remains unresolved.

### Topic 4 — Plants, wildlife, populations, and performance

The user explicitly asked how to support large numbers of plants and animals. The conclusion was that full entities should be the exception, not the default, for dumb or numerous life.

Plants can use vegetation fields: biomass, density, moisture stress, nutrient stress, species mix, regrowth timers. Wildlife and pests can use population fields or aggregate cell counts. Individual entities are promoted when they become important: near players, structures, hazards, jobs, or interaction zones.

The rationale was performance and determinism. Field sampling is cheap and deterministic. Aggregate rates avoid O(N²) interactions. Promotion/demotion allows local detail without simulating everything everywhere.

What remains uncertain is the exact thresholds and content policies for when things promote or demote.

### Topic 5 — Engine-wide modularity and optimization

The user asked whether the same methods could apply across the whole engine. The answer was yes. This became a central theme.

The architecture standardized several cross-cutting primitives:

* typed packets;
* fields, events, and messages;
* sorted delta commit;
* deterministic scheduler phases;
* budgets and carryover queues;
* representation ladder;
* graph toolkit;
* deterministic VM;
* probes, hashes, replay;
* dirty incremental rebuilds;
* source-of-truth versus derived cache separation.

This matters because without common primitives, every subsystem would reinvent scheduling, queues, graphs, IDs, serialization, and determinism. That would create bugs and inconsistency.

The conclusion was to combine shared mechanisms but split domain-specific artifacts. For example, power, fluids, comms, supports, enclosures, and routing all use graphs, but each defines its own node/edge semantics through data.

### Topic 6 — Buildings and infrastructure at scale

Buildings became a major example of heavy subsystem optimization. The user wanted custom buildings and infrastructure that could scale to real-world complexity. The answer reframed buildings as parametric authoring state plus deterministic compiled artifacts.

Instead of storing final geometry as truth, buildings store footprints, volumes, enclosure definitions, surfaces, sockets, carrier intents, and edit records. Derived compilers produce occupancy, voids, enclosure graphs, surface graphs, support graphs, and carrier artifacts. These are caches, not authority.

This matters because buildings connect many systems: heat, atmosphere, power, fluids, agents, decor, pathing, structural loads, destruction. Without derived artifacts and incremental dirty rebuilds, edits would require expensive full recomputation.

The unresolved issue is how much of STRUCT compilation should be implemented first. The prompt plan is broad; a real implementation may need a smaller first milestone.

### Topic 7 — Future Interstellar and Wargames DLCs

The user introduced future DLC directions. Instead of designing “space code” and “war code,” the plan generalized their needs.

For Interstellar, the core primitives are domains, frames, and propagators. A domain can represent a surface, orbital space, station interior, or other spatial context without hardcoding those names. A frame graph handles coordinate transforms. A propagator handles time-evolving states such as trajectories or moving fronts.

For Wargames, fog and information become observer contexts, belief stores, visibility/occlusion, and communications graphs. Knowledge is simulation state, not UI state. This is essential because fog of war affects what players and AI can know and act on; it must therefore be deterministic and replayable.

The conclusion was that DLCs should be content packs and optional plugin families built on generic primitives. They should not add semantic assumptions to the core.

### Topic 8 — Arbitrary placement, TRANS, STRUCT, and DECOR

The sister-chat prompt added a major spatial authoring requirement: no grid lock. All placement must support arbitrary position, rotation, inclination, and curved geometry in 3D. Engine logic cannot depend on a global grid. Grids may exist only as optional UI snapping aids.

This led to the pose-and-anchor model. Placement commands are quantized and stored as parametric anchors, not baked world-space mesh coordinates. A corridor anchor uses alignment station and local offsets. A structure surface anchor uses surface parameter coordinates. A socket anchor references a stable socket.

TRANS became the canonical system for corridors and alignments: splines or polylines, z-profiles, roll, cross-section slots, attachments over station intervals, and junctions. Co-location is handled through slots. A road, rail, utility duct, electrification line, and pipeline can share a single corridor model rather than fighting as independent splines.

STRUCT handles arbitrary buildings and infrastructure. DECOR handles signs, markings, decals, props, and surface details across all host types through rulepacks and overrides.

This topic is one of the most important contributions of the chat.

### Topic 9 — The Codex prompt roadmap

Once the architecture stabilized, the user asked for a plan of prompts to give Codex. The assistant first produced an 8-prompt plan, but that was superseded after the sister-chat requirements. The final version became a 14-prompt roadmap.

This roadmap is important because it translates architecture into implementation phases. It starts with invariants and scaffolding, then builds packet IO, scheduler, fields/messages/events, LOD, placement, TRANS, STRUCT, DECOR, agents, graphs, domains, knowledge/comms, and finally determinism hardening.

Each prompt included constraints, file paths, module goals, tests, documentation updates, and acceptance criteria. The prompts were designed to prevent Codex from overreaching into gameplay semantics too early.

### Topic 10 — Documentation validation and preservation

After the prompt plan, the user wanted the docs to be kept valid and consistent. A documentation validation prompt was generated. It instructed Codex to reconcile terminology, subsystem boundaries, determinism guarantees, no-baked-geometry rules, arbitrary placement correctness, and source-of-truth versus derived-cache tables.

Finally, the user asked for context transfer, then report packaging, then an in-chat reader, and now this human-readable report. These later steps were about preserving the chat’s output for future aggregation and spec-book construction.

## 5. Decisions Made and Why

A short map of the most important decisions:

| Decision area    | Decision                                                                | Label                    |

| Mutation         | Use sorted delta commit as the sole mutation point.                     | INFERENCE                |

### The foundational language and determinism decisions

**FACT:** The user explicitly set Domino as ISO C89 and Dominium as C++98. These are final unless the user later changes them. The reason is portability and deterministic control. C89 also restricts language features, which reduces accidental platform-specific behavior but increases the need for careful architecture.

**FACT:** Fixed-point math is required where determinism matters. This decision affects geometry, physics, movement, scheduling, replay, networking, and procedural generation. It was made because floating-point behavior can differ across platforms, compilers, and CPU modes.

**FACT:** No OS, platform, UI, or rendering APIs are allowed in the engine. This protects the deterministic core from external nondeterminism and preserves strict layering.

### The semantic-free engine decision

**FACT:** The user stated that no hardcoded game semantics are allowed and everything must be data-driven.

This decision drives nearly everything else. The engine cannot have special branches for humans, animals, robots, roads, rails, combat, space, crops, or factories. Instead, it has registries, TLV prototypes, type IDs, capabilities, actions, graphs, fields, and data-defined models.

This decision could be revisited only if the user chooses to sacrifice extensibility for simplicity in a specific subsystem, but that would contradict the chat’s core direction.

### The delta commit decision

**INFERENCE:** The assistant proposed that all state mutation should happen through sorted delta commit, and the user continued building on the plan. This should be treated as accepted working architecture, but it is not an explicit user sentence.

The rationale is strong. If subsystems mutate state directly while running, outcomes can depend on update order, memory layout, container order, or platform differences. A sorted delta commit creates one canonical mutation point.

This affects agents, actions, physics, construction, decor promotion, knowledge updates, and network effects.

### The behavior pipeline decision

**INFERENCE:** The user asked for arbitrary actors, arbitrary stimuli, and arbitrary actions. The assistant proposed the sensor/observation/mind/intent/action/delta pipeline. The user accepted the direction by continuing the conversation.

This decision makes behavior extensible while keeping mutation controlled. It allows simple wildlife, commandable pets, job-driven workers, robot controllers, group controllers, and machine controllers to share one model.

The alternative would be bespoke AI systems per actor type. That was not carried forward because it would violate modularity and hardcode semantics.

### The representation ladder decision

**INFERENCE:** The assistant proposed R0/R1/R2/R3 and the user repeatedly asked how to apply it more broadly. It became a central working decision.

The reason is scale. A game/simulation with large vegetation, wildlife, buildings, networks, atmospheres, and infrastructure cannot afford full-detail simulation everywhere. The representation ladder lets the engine preserve authoritative state while reducing fidelity and cadence where possible.

This decision depends on good promotion/demotion rules and accumulators. If implemented poorly, LOD can change simulation outcomes and break determinism.

### The pose and anchor decision

**FACT:** The sister-chat prompt explicitly required arbitrary placement, no grid lock, quantized command inputs, fixed-point math, and parametric anchoring. The assistant integrated this into the plan.

This is one of the strongest decisions in the chat. It means the source of truth for placement is not a grid cell or baked mesh. It is a quantized parametric anchor and local pose. This supports arbitrary rotation, inclination, curved corridors, and rebuildable geometry.

This affects BUILD, TRANS, STRUCT, DECOR, visibility, pathing, and every placement/edit command.

### The TRANS/STRUCT/DECOR decisions

**FACT:** The sister prompt explicitly required TRANS corridors, STRUCT arbitrary buildings, and DECOR signage/markings/surface detail. The assistant turned that into Prompts 7–9.

TRANS becomes the canonical corridor alignment system. STRUCT becomes the building/infrastructure/enclosure/carrier/surface system. DECOR becomes a host-agnostic rulepack/override system.

The key rationale is that real-world infrastructure cannot be represented cleanly as grid-locked objects or independent splines. Corridors need co-location. Buildings need arbitrary oriented volumes. Decor needs automatic generation plus manual user control.

### The space/war generalization decision

**INFERENCE:** The user asked to prepare for Interstellar and Wargames, then explicitly asked to make it core and not space/war-specific. The assistant generalized the systems.

The decision was to represent space-like needs through domains, frames, and propagators, and war-like information needs through observers, beliefs, visibility, and comms. This avoids hardcoded DLC semantics.

This decision would be revisited only if future DLC requirements prove the generic primitives insufficient.

### The Codex prompt roadmap decision

**FACT:** The user explicitly requested a master plan of prompts and then requested each prompt. This is a concrete artifact and process decision.

The prompts are implementation scaffolding, not code. They should be adapted to the actual repository before execution.

## 6. Ideas Considered but Rejected, Superseded, or Deprioritised

### Global grid-locked placement

This was rejected because the sister-chat requirements explicitly required arbitrary position, rotation, inclination, and curved 3D geometry. Grids may exist as UI snapping aids or spatial indices, but not as authoritative placement logic.

This rejection is final unless the user changes a hard requirement.

### Baked geometry as source of truth

This was rejected because it breaks deterministic rebuilds, replay, arbitrary editing, and parametric placement. Baked geometry can exist as a derived cache, but it cannot be the authoritative state.

This is final under the current plan.

### Stacked independent splines for co-located corridors

This was rejected because the user specifically required corridor overlap and co-location through cross-section slots. Independent splines would fight for space and make upgrades, utilities, electrification, road/rail sharing, and junctions harder.

The accepted alternative is one corridor alignment with slot-based occupants and attachments.

### Per-entity simulation for all grass, weeds, pests, and wildlife

This was rejected as the default because it does not scale. The accepted model uses vegetation fields, population aggregates, and deterministic promotion.

It can become relevant again for important individual plants or animals, but only after promotion.

### Global intelligence for every herd/flock/swarm member

This was rejected as unnecessary and too expensive. Group controllers and local bounded steering are preferred.

Full intelligence remains possible for important agents, leaders, companions, or player-visible individuals.

### Direct mutation from minds or controllers

This was rejected in the architecture because it would break deterministic ordering. Minds and controllers emit intents; actions emit deltas; the commit phase mutates state.

### Renderer-based fog of war

## 7. Important Reasoning and Rationale

The visible reasoning throughout the chat was dominated by four concerns: determinism, extensibility, scale, and source-of-truth discipline.

Determinism required stable IDs, fixed-point math, fixed iteration counts, canonical ordering, sorted deltas, replay hashes, deterministic PRNG streams, and avoidance of platform-dependent behavior. Every time a system could become nondeterministic, the plan pushed it behind a canonical API or ordered queue.

Extensibility required the engine not to know gameplay semantics. This is why the plan favors type IDs, registries, TLV schemas, component composition, capabilities, actions, fields, graph nodes/edges, and data-defined prototypes. The engine should provide the machinery; content defines the meaning.

Scale required the representation ladder and field/aggregate models. The chat repeatedly recognized that simulating everything in full detail is impossible. Plants, animals, buildings, networks, fog, comms, visibility, pathing, and fluids all need LOD, dirty rebuilds, caches, or aggregates.

Source-of-truth discipline became especially important after the arbitrary placement requirements. The plan distinguishes authoring state from derived caches. Authoring state is stable, parametric, replayable, and versioned. Compiled artifacts are rebuildable and disposable. This protects the engine from geometry drift, save corruption, and nondeterministic rebuilds.

The user seemed to care most about preserving rigorous architecture rather than rushing into code. The repeated requests asked how to make systems more modular, more extensible, more optimized, and more deterministic. The final prompt-generation phase shows the user wanted this thinking converted into concrete implementation instructions.

## 16. Final Plain-English Summary

This chat was a deep architecture and planning session for the Dominium/Domino deterministic engine. The user began by setting strict technical constraints: the Domino engine core must be ISO C89, deterministic, fixed-point where determinism matters, free of OS/platform/UI/rendering APIs, and free of hardcoded gameplay semantics. Dominium is the portable C++98 UI/tools layer and must only emit deterministic commands. The engine must use TLV-versioned data, stable IDs, deterministic scheduling, replay, and input-driven lockstep multiplayer.

The conversation first addressed advanced simulation systems: heat, power, fluids, atmosphere, vehicles, structures, destruction, and mod-extensible physics. The important outcome was a general architecture rather than specific solvers. These systems should be content-driven, fixed-point, graph/field-based, versioned, bounded, and replayable. They should interact through ports, fields, events, messages, actions, and deltas rather than direct hardcoded calls.

The focus then moved to agents and life. The user wanted the engine to handle humans, NPCs, robots, companions, pets, wildlife, pests, plants, trees, grass, crops, swarms, herds, flocks, and predator-prey behavior. The solution was a generic actor substrate: actors are component compositions, not hardcoded classes. Behavior flows through sensors, observations, minds or controllers, intents, actions, and deltas. This allows simple wildlife, commandable pets, job-driven workers, robots, plant growth systems, and group controllers to share one extensible model.

Performance became central. The chat established that full simulation everywhere is impossible. The engine should use a representation ladder: R0 full, R1 lite, R2 aggregate, and R3 dormant. Dumb animals, pests, grass, weeds, and crops can be represented by fields and aggregate populations until they become important. Buildings, networks, physics, pathfinding, visibility, and decor can also use lower-detail representations when far away or inactive. Work must be budgeted and deferred deterministically, using accumulators so that performance optimization affects timing or fidelity rather than authoritative outcomes.

The user then asked whether these same techniques could optimize heavy systems such as custom buildings. The plan separated authoring state from compiled artifacts. Buildings and infrastructure should be stored as parametric definitions and edit records, while derived caches provide occupancy, enclosures, surfaces, supports, ports, nav hints, and boundary lists. This lets the engine rebuild only dirty regions under deterministic budgets instead of recomputing everything.

The conversation later introduced future Interstellar and Wargames DLC needs. Instead of adding hardcoded space or war systems, the architecture generalized them. Space-like needs become domains, coordinate frame graphs, and propagators. Fog of war and information dissemination become observer contexts, belief stores, visibility/occlusion, and communications graphs. Knowledge is simulation state and must participate in replay and hashing. Combat-like interactions should eventually be expressed as generic actions and deltas, not special combat code in the core.

A major refinement came from a sister-chat prompt. It required arbitrary placement, arbitrary rotation and inclination, curved 3D geometry, no engine grid lock, corridor co-location, real-world infrastructure scale, signage, markings, and surface detail. This forced the plan to explicitly adopt fixed-point pose plus parametric anchors as the placement source of truth. Grids are only UI snapping aids or spatial indices. Baked geometry is never authoritative. TRANS became the canonical corridor/alignment system with cross-section slots and junctions. STRUCT became the system for arbitrary buildings, volumes, enclosures, carriers, surfaces, and sockets. DECOR became a host-agnostic system for rule-generated and manually overridden signs, decals, markings, and props.

The final practical product of the chat was a 14-prompt Codex implementation roadmap. The prompts cover, in order: engine invariants/specs/scaffolding; packet ABI/TLV registries; scheduler/budgets/delta commit; fields/events/messages; representation ladder; pose/anchors; TRANS; STRUCT; DECOR; agents/actions/groups; graph toolkit; domains/frames/propagators; observer/knowledge/visibility/comms; and determinism hardening/replay/regression. A separate documentation validation prompt was also created to keep all specs consistent and formal.

Several issues remain unresolved. The actual repository was not inspected, so paths and build assumptions require verification. Exact fixed-point Q formats are not settled. The project still needs a decision on whether deterministic mods should be VM-only or allow native-code plugins. The detailed physics subsystem prompts still need to be generated after the core spine. The minimum viable implementation scope for large systems such as STRUCT, TRANS, and DECOR may need to be narrowed when implementation starts.

The most important thing to preserve is the architectural direction: build a small set of deterministic, generic, data-driven engine primitives, and express all simulation, actor, infrastructure, DLC, and mod semantics through those primitives. Do not reintroduce grid-locked placement, baked geometry as truth, hardcoded humans/animals/robots/space/war logic, direct state mutation, unordered iteration, floating deterministic solvers, or UI-driven simulation state.

The best next action is to inspect the repository and then adapt or run Codex Prompt 1. From there, proceed through the prompt plan carefully, verifying each foundation before building the next layer.
