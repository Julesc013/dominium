# Human-Readable Chat Report — Dominium Launcher and Setup Architecture

## 1. One-Page Orientation

This chat was about designing the **Dominium launcher, setup system, runtime integration model, and surrounding user-facing infrastructure**. It began from a broad architectural philosophy for the Dominium project: keep the simulation core deterministic, boring, portable, and C89; put explicit APIs between layers; use versioned formats; make mods data-first; and prevent tools, launcher, and runtime code from drifting into separate incompatible implementations. From that starting point, the conversation progressively narrowed into a concrete system design for how users install, launch, configure, mod, supervise, and manage Dominium.

The central problem was not “how do we open a game window?” It was how to build a long-lived ecosystem around the game without corrupting the deterministic engine or creating hidden coupling. The launcher needed to be powerful enough to manage multiple instances, installs, mods, settings, logs, profiles, news, changelogs, and future ecosystem features, while the game itself still had to run perfectly well without it. The setup system likewise needed to support normal users, portable users, system administrators, CI, and custom deployments without forcing every code path into a single fragile installer.

A major early conclusion was that Dominium should use **one codebase but multiple binaries**. The assistant proposed, and the user continued from, the idea that setup, launcher, and game should be separate executables rather than one monolithic executable. The visible rationale was practical: setup may need elevated privileges, the launcher is a long-running management application, and the game/runtime must remain performance- and determinism-sensitive. Separate binaries preserve privilege boundaries and crash isolation while still sharing libraries, manifests, and conventions.

A second major conclusion was that the launcher must be **optional**. The user explicitly stated that the game should run perfectly fine with no launcher, with direct executable running, and with launcher contracts disabled if unused. That shaped the runtime contract: runtime binaries may accept launcher session and instance IDs, but must not require them. The launcher can supervise processes, aggregate logs, and pass metadata, but it must not become a gameplay dependency.

The user then defined the instance display model: **NONE, CLI, TUI, GUI**. This became the abstraction for how any runtime instance presents itself. A server might run with no display; a command-line client or tool might use CLI; a text UI could use TUI; a graphical client could use GUI. This abstraction matters because Dominium is intended to span modern and retro environments, not just one desktop platform.

The launcher’s user-facing purpose eventually crystallized into five required tabs: **News**, **Changes**, **Mods**, **Instances**, and **Settings**. The user made clear these should be fully interactive, not decorative. News should browse and open official updates; Changes should show releases, changelogs, betas, and alphas; Mods should be a full mod manager; Instances should be a full instance and installation manager; Settings should be powerful but intuitive. These tabs are the heart of the launcher’s user experience.

The largest change of direction came near the end. Earlier in the conversation, implementation prompts assumed a more conventional C++98 launcher structure with CLI/TUI/GUI frontends, possibly using ncurses, SDL, or similar systems. The user then introduced a new, stricter direction: the launcher should use the Dominium/“Domino” system stack, specifically **dsys** for platform abstraction and **dgfx** for rendering. This superseded the older frontend plan for the launcher. The latest active direction is a **C89 launcher using dsys for windowing, filesystem, input, timers, and processes, and dgfx for all rendering through IR/canvas/multi-view GUI**, with retro and modern backend support.

The chat also produced multiple Codex work-order prompts and finally a downloadable report package. Those artifacts are useful, but the substance is the architecture: keep engine deterministic, keep setup/launcher/runtime boundaries explicit, make the launcher optional and modular, use a strong process/instance model, and now implement the launcher over dsys/dgfx rather than ad hoc platform UI stacks.

The most important unresolved issue is the conflict between earlier JSON-based manifest/config ideas and the user’s broader TLV/versioned-file philosophy, reinforced by the later `.dmeta` direction. Future work must resolve whether launcher/setup persistent files use JSON, TLV `.dmeta`, or a transitional arrangement. The second major unresolved issue is that no repository files were inspected in this chat, so actual dsys/dgfx APIs, build layout, runtime binary names, and existing file-format libraries remain unverified.

## 6. Ideas Considered but Rejected, Superseded, or Deprioritised

### One monolithic executable

This was considered when the user asked whether setup, launcher, and game could be one. The assistant rejected it as the default because it mixes privileges and responsibilities. Setup may require admin/system changes; launcher should be a normal-user supervisor; the game should be independent and runtime-focused.

The rejection is strong, but not absolute. A constrained retro build might someday use a combined executable if process separation is impossible.

### Independent console windows by default

When discussing CLI and headless operation, the assistant argued against launching independent OS console windows by default. The reason was that they break centralized log aggregation and do not generalize to Linux/mac text-only environments, SSH, DOS, or CP/M.

This remains rejected as the default, though an explicit external-console/debug option could still be useful.

### Launcher required for gameplay

The user explicitly rejected this by requiring direct executable running and no-launcher operation. This is final unless the user reverses it.

### Launcher linking engine internals

This was rejected because it violates the deterministic engine boundary. The launcher should read manifests/metadata and communicate with runtimes through explicit contracts, not inspect simulation internals.

### Raw OS drawing or direct platform UI

This became superseded by the latest dsys/dgfx direction. Earlier ideas involving direct GUI/TUI libraries are no longer the active launcher path. The launcher should render through dgfx and receive platform/input/process services through dsys.

### Direct ncurses/SDL/ImGui frontend plan

Earlier prompts described C++98 CLI/TUI/GUI frontends and conditional ncurses/SDL/ImGui. That plan is now historical. It may still contain useful concepts such as view-models and tab workflows, but the implementation substrate is superseded by dsys/dgfx.

### Unversioned or silently changed formats

Rejected from the beginning by the user’s philosophy. All persistent formats should be versioned and migrated explicitly.

### Tools or launcher hand-rolling parsers

Rejected because the user wants one source of truth for formats. Tools and launcher should reuse shared format/IO libraries.

### Full graphical feature parity on CP/M

This was not explicitly rejected by the user, but it is a necessary degradation inference. The user said CP/M support “where possible,” and the assistant identified CP/M as effectively text-only. The launcher can support CLI/TUI there, not full graphical feature parity.

### Deleting user data by default

Rejected in setup design because it is dangerous. Uninstall should preserve user data unless explicitly told to remove it.

## 7. Important Reasoning and Rationale

The visible rationale throughout the conversation was about **long-term survivability**. The project is not just a one-off game launcher; it is intended to support a broad platform matrix, mods, tools, plugins, multiple instances, multiple install types, and future ecosystem services.

Several tradeoffs shaped the design.

First, the engine must stay simple and deterministic even if everything around it becomes complex. That is why launcher, setup, tools, and runtime frontends are kept out of the engine. The user’s repeated emphasis on a “boring C89 core” drove this boundary.

Second, the launcher must be powerful but optional. This balances user convenience with portability and resilience. A launcher-dependent game would be fragile for servers, scripts, debugging, retro platforms, and users who want direct executable control.

Third, separate binaries were preferred over a monolith because they match the real differences in responsibility. Setup mutates installations and sometimes needs elevated rights. The launcher supervises and manages. The game runs. Those are different failure domains.

Fourth, the display model needed to be abstract. `NONE`, `CLI`, `TUI`, and `GUI` allow one architecture to cover headless servers, command-line tools, text UIs, graphical clients, and retro environments.

Fifth, modularity was extended beyond gameplay mods because the launcher itself is expected to grow. A hardcoded launcher would eventually become brittle. A module/plugin architecture lets future features appear as tabs, commands, hooks, sidecars, or services.

Sixth, the dsys/dgfx shift reflects a deeper project decision: the launcher should not be tied to contemporary desktop UI frameworks. It should use the same abstract platform/rendering stack as the rest of Dominium’s cross-platform ambitions. This makes the launcher more consistent with retro support and internal rendering architecture, but it also increases the burden of verifying actual dsys/dgfx APIs before implementation.

The main assumption throughout is that the underlying project either already has, or will have, the shared libraries and platform abstractions described. That assumption is unverified in this chat.

## 16. Final Plain-English Summary

This chat was a long architecture and implementation-planning session for the Dominium launcher, setup system, and runtime integration model. It began with a strong project philosophy: Dominium should have one deterministic, boring C89 engine core; explicit APIs between layers; versioned file formats; data-first content and mods; tools and launchers that reuse shared libraries; and migration rather than silent breakage. That philosophy shaped every later decision.

The conversation first clarified the boundaries between the engine, runtime, launcher, setup, tools, content, mods, and SDK. The engine remains protected: deterministic, C89, no platform or renderer code, no launcher dependency. Runtime binaries sit above the engine and must be able to run directly. The launcher sits outside the game and manages processes, installs, mods, profiles, logs, and settings. Setup is a separate install/repair/uninstall authority. This separation matters because Dominium is intended to be long-lived, cross-platform, moddable, and resistant to hidden coupling.

One of the most important user-stated requirements was that the launcher must be optional. The game must run perfectly well without it. The launcher may supervise and pass metadata, but game binaries must not require launcher IDs, launcher IPC, or launcher configuration. This led to the standalone-versus-supervised runtime model. The user also defined instance display modes: `NONE`, `CLI`, `TUI`, and `GUI`. These modes became the basic abstraction for headless servers, command-line tools, text UIs, and graphical clients.

The setup system became its own major topic. The user wanted portable non-installation, per-user installation, and all-user/system installation across Windows, Linux, and macOS. The assistant proposed a setup tool, `dom_setup`, responsible for install, repair, uninstall, list, and info operations. Launcher and game should not duplicate this logic. Setup should support sane defaults for normal users and full customization for scripts, CI, power users, and unusual install layouts. It should preserve user data by default during uninstall.

The launcher grew into an all-in-one Dominium control center. The user wanted it to manage not just game launching but mods, accounts, tools, installations, servers, wiki/forum access, direct messages, playtime, and future ecosystem features. Later, the user fixed the initial visible launcher tabs as **News**, **Changes**, **Mods**, **Instances**, and **Settings**. These tabs must be real workflows. News browses and opens official updates. Changes browses releases, changelogs, betas, and alphas. Mods is a full mod manager. Instances manages running processes and installations. Settings exposes powerful but intuitive configuration.

Modularity was repeatedly emphasized. The architecture extended modularity to launcher modules, setup plugins, and individual instance sidecars or observers. The visible rationale was that the launcher should not become a rigid monolith. Built-in tabs and future plugins should use compatible registration concepts, and plugins should be ABI-versioned and optional.

The biggest change came late in the chat. Earlier implementation prompts assumed a C++98 launcher with separate CLI/TUI/GUI frontends, possibly using ncurses, SDL, or similar libraries. The user then introduced a new explicit direction: the launcher must be built using **dsys** and **dgfx**. In that latest direction, the launcher is strict C89. dsys provides all platform services: windowing, filesystem, input, timers, and processes. dgfx provides all rendering through IR, canvas, multi-view 2D/3D GUI, vector and raster primitives, and backend abstraction. Retro and modern backends are assumed to exist. This supersedes the earlier direct frontend implementation plan for the launcher.

Several artifacts were produced: architecture specs, API sketches, Codex prompts, a context transfer packet, and a final report package. These are useful, but they are not proof that repository code exists. No repo files were inspected in this chat. Future implementation must verify actual dsys/dgfx APIs, build layout, runtime binaries, file-format libraries, and setup code.

The most important unresolved issue is the conflict between earlier JSON-based examples and the user’s TLV/versioned-file philosophy, reinforced by the later `.dmeta` direction. A future assistant must not silently choose one. It should ask the user or inspect the repo. The likely direction, based on the user’s philosophy, is versioned TLV/dmeta, but that remains unconfirmed.

The best next action is to continue from the latest dsys/dgfx direction and generate the first implementation-ready Codex work order: core launcher config, install discovery, profiles, and mods data layer in C89 using dsys filesystem and versioned metadata. Before coding, verify the repo and resolve JSON versus TLV/dmeta.
