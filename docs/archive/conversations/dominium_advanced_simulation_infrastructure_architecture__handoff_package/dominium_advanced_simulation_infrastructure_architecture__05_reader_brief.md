# Reader Brief — Dominium Advanced Simulation and Infrastructure Architecture

## What This Chat Was About

[FACT] This chat developed architecture for the Dominium/Domino deterministic engine, beginning with advanced simulation systems and expanding into gameplay layering, infrastructure construction, arbitrary placement, corridors/splines, buildings/structures, signage, markings, decor, and implementation/refactor prompts. The central constraint is that Domino Engine must remain ISO C89, deterministic, fixed-point, portable, and free of hardcoded gameplay semantics, while Dominium UI/tools use a portable C++98 subset and emits deterministic commands.

[FACT] The largest outcome was a unified BUILD / TRANS / STRUCT / DECOR / SIM model. BUILD owns construction intent, phases, costs, and jobs. TRANS owns linear/networked topology such as corridors, junctions, slots, attachments, z-profiles, microsegments, and lane/track/utility graphs. STRUCT owns volumetric/load-bearing/enclosed structures, buildings, interiors, surface/anchor graphs, and generated carriers such as bridges/tunnels. DECOR owns visual signs, markings, decals, props, generated baselines, manual overrides, semantic keys, and tiles. SIM owns deterministic physics domains. RES/content owns semantics.

[FACT] The key late requirement is no grid lock: the engine should support arbitrary position, rotation, inclination, and full 3D curves. Grids are optional Dominium UI snapping aids only. Determinism should come from fixed-point math, quantized commands, stable IDs, ordered iteration, fixed iteration counts, and parametric host-relative anchoring.

## Most Important Things to Know

- Domino Engine = ISO C89; Dominium UI/tools = portable C++98.
- No OS APIs in engine.
- No floats where determinism matters; use fixed-point Q formats.
- No hardcoded gameplay semantics; content defines meaning.
- UI emits deterministic commands; it does not mutate engine state directly.
- Overlapping infrastructure should use corridor bundles with slots, not competing splines.
- Runtime performance uses compiled microsegments, chunk spatial indices, dirty flags, and hot/cold data split.
- Complex infrastructure should be built through corridors, junctions, layers, modules, and facilities.
- Signage/markings are DECOR: generated by rules and manually overridden with pin/suppress semantics.
- Host Surfaces + Anchor Frames generalize placement to buildings, interiors, terrain, props, and corridors.
- DECOR is visual-only unless promoted to a STRUCT/TRANS device.
- Engine is grid-agnostic; grids are UI snapping only.
- No repo files were inspected or modified in the original chat.
- Proposed paths and command names are conceptual until verified.
