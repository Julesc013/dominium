# FormalizationState Baseline

## 1) State Machine
- Canonical lifecycle: `raw -> inferred -> formal -> networked`.
- `raw`:
  - physical inputs only (`raw_sources`), no formal artifacts attached.
- `inferred`:
  - derived candidate set exists (`formalization_inference_candidates`), deterministic and budgeted.
  - no network graph creation and no commitment creation.
- `formal`:
  - accepted candidate with `formal_artifact_ref`.
  - commitment generated by process (`commit.formalization.constraint` or `commit.formalization.construction`).
- `networked`:
  - deterministic `network_graph_ref` attached with canonical node/edge stubs.
- Reversion:
  - `process.formalization_revert` clears formal/network refs and keeps `raw_sources`.
  - optional graph removal controlled by process input.

## 2) Inference Engine Behavior
- Module: `src/infrastructure/formalization/inference_engine.py`.
- Deterministic candidate generation from raw semantic sources:
  - candidate ordering is stable.
  - fingerprints are canonical.
  - no wall-clock usage.
- Budget behavior:
  - bounded candidate selection via cost units.
  - deterministic degrade reason: `degrade.formalization.inference_budget`.
- Inference engine remains derived-only:
  - no guide geometry creation.
  - no network mutation.
  - no commitment mutation.

## 3) Acceptance and Promotion Semantics
- `process.formalization_accept_candidate`:
  - requires explicit confirmation under `policy.diegetic_strict` / `policy.assisted`.
  - enforces optional spec-compliance gate (`refusal.formalization.spec_noncompliant`).
  - emits deterministic formalization event and commitment.
- `process.formalization_promote_networked`:
  - policy-gated and deterministic.
  - attaches deterministic graph node/edge set.
  - emits formalization event.
- `process.formalization_revert`:
  - policy-governed rollback to `raw`.
  - emits formalization event.

## 4) UX and Inspection Integration
- Control actions:
  - `action.formalize.infer`
  - `action.formalize.accept`
  - `action.formalize.promote_network`
  - `action.formalize.revert`
- Inspection integration:
  - target kind: `formalization`.
  - sections:
    - `section.formalization_summary`
    - `section.formalization_candidates`
    - `section.spec_compliance_summary`
    - `section.commitments_summary`
    - `section.events_summary`
- Render/overlay integration:
  - inferred/formal/networked overlays routed by formalization target payload.
  - inferred candidate overlays are epistemic-gated.

## 5) Policy and Epistemic Differences
- Diegetic policy:
  - no hidden candidate rows in overlays.
  - coarse summaries only in formalization candidate section.
  - explicit confirmation required for acceptance where policy enforces it.
- Planner/assisted policy:
  - candidate suggestions with explicit confirm path.
  - formalization transitions remain process-driven and logged.
- Admin/meta policy:
  - instant formalization/network promotion paths allowed by policy.
  - still fully decision-logged and event-sourced.

## 6) MOB-1 Extension Points
- `formal_artifact_ref` is the handoff anchor for future GuideGeometry realization.
- `network_graph_ref` provides deterministic network integration point for traffic/signal/schedule layers.
- `spec_id` bridge enables MOB safety/compliance checks without hardcoded rail/road assumptions.
